# Замечания по платформе OneBase

Список того, чего реально не хватает в платформе/DSL, выявленного при доработке демо-конфигурации `trade` до полноценной учётной системы (поступление, реализация, списание, оприходование, перемещение, ПКО, РКО).

Все пункты — про возможности и API платформы. Где можно было решить правкой конфигурации (модель данных, измерения, ресурсы) — решено правкой конфигурации.

## 40. `setPeriodFromFields` хранит `period = time.Now()`, а не дату документа → `.Остатки(&Момент)` всегда пустая ✅ закрыто 2026-05-21

`setPeriodFromFields` (handlers.go) искал date-поле по `fields[f.Name]`, но `Object.Set` приводил ключи к lower-case (`"дата"`), а `f.Name` хранил PascalCase (`"Дата"`) — промах. Аналогично `МоментВремени()` не находил дату → `Period` = zero. Теперь: регистронезависимый поиск через `strings.ToLower(f.Name)` + `AsTime()` (распознаёт `time.Time`, `*time.Time`, строку RFC3339).

---

## 39. Переменная, впервые присвоенная внутри блока — локальна к блоку ✅ закрыто 2026-05-21

**Все** блочные конструкции создают дочерний scope: `Если/Иначе`, `ИначеЕсли`, `Для Каждого`, `Для ... По`, `Пока`, `Попытка/Исключение`. Правило `env.set()`: если переменная уже есть в родительском scope — обновляет родителя; если нет — создаёт локально в child. Первое присвоение определяет scope.

```
// Сначала внутри блока → остаётся локальной, снаружи nil:
Если Условие Тогда
  Результат = 42;    // локально к Если
КонецЕсли;
Сообщить(Результат); // <nil>!

// Предобъявление → обновляет родителя:
Результат = 0;
Если Условие Тогда
  Результат = 42;    // обновляет родителя
КонецЕсли;
Сообщить(Результат); // 42 ✓
```

Это работает так в Go (лексическое scope), но в 1С/OneScript scope = функция. Разработчику на OneBase нужно **всегда предобъявлять** переменные перед блоками. Тест `scope_test.go` покрывает только случай с предобъявлением, маскируя проблему.

Симптом: движения регистров с nil-суммами (записываются как 0). Например, `НеттоСтроки = Сумма - СуммаНДС` внутри `Если/Иначе` → nil → ДвП.Сумма = nil → 0 в БД. Обход: предобъявлять перед блоком.

## 38. `МоментВремени()` не разбирал строковую дату — нулевой момент при перепроведении из UI ✅ закрыто 2026-05-21

После П.37 склад в ФИФО стал матчиться по имени, но перепроведение из UI всё равно давало «доступно 0».

Причина — в `internal/runtime/object.go`, `CallMethod("моментвремени")`: момент брался только если поле даты — `time.Time` (`v.(time.Time)`). При проведении из обработки `this.Дата` действительно `time.Time`, но при загрузке документа из формы/БД дата приходит **строкой** (`"2026-02-21T00:00"` и т.п.). Type-assertion не проходила → `Period` оставался нулевым (`0001-01-01`) → moment-фильтр `period < @` отсекал ВСЕ движения → ФИФО «доступно 0», хотя партии есть.

Показательно: `setPeriodFromFields` (период движений) строковые даты разбирал четырьмя layout'ами, а `МоментВремени()` — нет. Рассогласование между двумя путями получения даты документа.

Фикс: `МоментВремени()` теперь разбирает строковую дату теми же layout'ами (`RFC3339`, `2006-01-02T15:04:05`, `2006-01-02T15:04`, `2006-01-02`) плюс формат SQLite `2006-01-02 15:04:05`. Связано с П.1.

## 37. Ссылки в полях ШАПКИ не обогащались при проведении — `Строка(this.Склад)` давал UUID ✅ закрыто 2026-05-21

При перепроведении документа из UI ФИФО падал «Недостаточно остатков, доступно 0», хотя партии есть. В тексте ошибки склад выводился UUID-ом (`Строка(this.Склад)` = `6e064d9f-...`), а номенклатура — именем (`Строка(Ном)` = «Кресло…»).

Причина — в `internal/ui/handlers.go`: на пути проведения вызывался `enrichTPRowsWithRefs` только для строк ТАБЛИЧНЫХ ЧАСТЕЙ (UUID → `*interpreter.Ref{UUID,Name}`), а поля ШАПКИ оставались сырыми UUID из формы. Поэтому в `ОбработкаПроведения`:
- `Строка.Номенклатура` — `*Ref` → `Строка()` = имя;
- `this.Склад` — строка-UUID → `Строка()` = UUID.

Измерение `ПартииТоваров.Склад` — `type: string` (workaround П.17). Из обработок/сидов движения пишутся `Дв.Склад = this.Склад`, где `this.Склад` — `*Ref`, и в string-колонку попадает ИМЯ. А фильтр перепроведения `ГДЕ Склад = Строка(this.Склад)` давал UUID → 0 совпадений (рассинхрон: партии по имени, фильтр по UUID). То же затрагивало `ДенежныеСредства.Касса`, `Взаиморасчеты.Контрагент`. Связано с П.28.

Фикс: добавлен `enrichHeaderRefs` (симметрично `enrichTPRowsWithRefs`), вызывается в `runOnPostCtx` — единой точке для всех путей проведения (UI и `docWriter` обработок). Ссылочные поля шапки обогащаются до `*Ref{UUID,Name}`; ref-параметры и reference-измерения остаются по UUID (`unwrapArrayParams` приводит `*Ref` → UUID). Поля, уже являющиеся `*Ref` (проведение из обработки), пропускаются — двойной обработки нет. Миграция данных не нужна: партии записаны по имени, и фильтр теперь тоже по имени.

## 36. `.Остатки(&МоментВремени)` ломал SQL под SQLite (рассинхрон плейсхолдеров/аргументов) ✅ закрыто 2026-05-21

При первом же реальном проведении с моментом (П.1) запрос падал:

```
Ошибка выполнения SQL: run query: missing argument with index 8
... WHERE (period < ? OR (period = ? AND recorder != ?)) ...
... WHERE номенклатура IN(?, ?, ?) AND склад = ? AND организация = ? ...
```

Причина — в `internal/query/query.go`, `momentTimeCondition`: SQL содержит `period` дважды (`period < ? OR period = ?`), один и тот же `periodPH` подставлялся в строку два раза, а аргумент `period` добавлялся в `tr.args` один раз. У SQLite плейсхолдеры анонимные (`SQLiteDialect.Placeholder` возвращает `"?"`), поэтому каждый `?` — отдельный позиционный параметр: 3 плейсхолдера момента против 2 аргументов → все последующие аргументы съезжали на один (8 плейсхолдеров, 7 аргументов). Для Postgres (`$n`) бага не было — `$1` переиспользуется.

Тест `moment_test.go` не ловил: проверял только текст SQL и число аргументов (= 2, т.е. фиксировал само ошибочное поведение), но не выполнял запрос.

Фикс: `period` добавляется вторым аргументом под второй плейсхолдер (`periodPH2`); для нумерованных диалектов дубль безвреден. Тест обновлён на 3 аргумента.

## 35. Нет арифметики дат по дням и конструктора `Дата(Год, Месяц, День)`

Сдвинуть дату на N дней нечем. По коду интерпретатора (`internal/dsl/interpreter`):

- `ТекущаяДата() - Число` → **nil**: бинарный `-` зовёт `toFloat`, а `toFloat` не знает `time.Time` (`interpreter.go`), ветка MINUS без числовых операндов фоллбэка не имеет и возвращает nil.
- `ТекущаяДата() + Число` → **строка**: ветка PLUS при нечисловых операндах делает `fmt.Sprintf("%v", l) + fmt.Sprintf("%v", r)` — то есть конкатенацию, а не дату.
- Конструктора `Дата(2026, 5, 11)` нет — в `builtins`/`builtins_init` зарегистрирован только разбор типа `"дата"` (для приведения), но не функция-конструктор.
- Из сдвигов есть только `ДобавитьМесяц(дата, n)` (поддерживает отрицательные). `ДобавитьДень`/`ДобавитьГод` отсутствуют; `НачалоДня`/`КонецДня`/`НачалоМесяца` и т.п. только усекают, не сдвигают.

Симптом на практике: в `ЗаполнитьТестовуюБазу` нужно было датировать поступления раньше реализаций (чтобы перепроведение/отчёты шли в верном хронологическом порядке, см. П.1). Разнести по дням невозможно — пришлось разносить **помесячно** через `ДобавитьМесяц(Сегодня, -3 / -2 / -1)`. Для демо терпимо, но любой сценарий «вчера/позавчера», «+3 рабочих дня», «срок оплаты = дата + 10 дней» нереализуем.

Нужно любое из:
- конструктор `Дата(Год, Месяц, День[, Час, Минута, Секунда])`;
- арифметика `Дата ± Число` в секундах (как в 1С/OneScript) — для этого `toFloat`/ветки `+`/`-` должны уметь `time.Time`;
- функции `ДобавитьДень(дата, n)` / `ДобавитьГод(дата, n)`.

## 34. Нет функции `ЗначениеРеквизитаОбъекта()` — невозможно получить реквизит по ссылке ✅ закрыто 2026-05-21

Добавлена глобальная `ЗначениеРеквизитаОбъекта(Ссылка, ИмяРеквизита)` — выполняет запрос к БД и возвращает значение реквизита по ссылке. Вся конфигурация переведена с dot-access (`Номенклатура.СтавкаНДС`) на `ЗначениеРеквизитаОбъекта(Номенклатура, "СтавкаНДС")`.

Исходный текст:

`Ref` хранит только UUID и Name. `Ref.Get(field)` обрабатывает только «наименование», «ссылка», «уникальныйидентификатор» — все остальные поля возвращают nil. Получить реквизит объекта через точку (`Номенклатура.СтавкаНДС`) **невозможно**.

В 1С для этого не используют две точки, а вызывают `ЗначениеРеквизитаОбъекта(Ссылка, "Реквизит")` — функция выполняет запрос к БД и возвращает значение. В OneBase этой функции не было.

Ломает типовые паттерны ПриЗаписи:
```
Строка.СтавкаНДС = Строка.Номенклатура.СтавкаНДС;  // nil
СтавкаНДС.Процент  // nil
```

Каскад: СуммаНДС = nil → ИтогНДС = nil → this.Сумма = nil → движения с nil-суммами → SUM(NULL) = NULL.

## 33. ПриЗаписи не вызывается из обработок (`docWriter`) ✅ закрыто 2026-05-20

`docWriter.write()` (`internal/ui/dsl_documents.go`) теперь вызывает хук ПриЗаписи и нумератор — аналогично UI-хендлеру. Нумератор также запускается при сохранении документа через `docWriter`, поэтому документы созданные из обработок получают Номер автоматически.

Исходный текст:

`docWriter.write()` просто делал `Upsert` + `saveTablePartsDirect` — без вызова хука ПриЗаписи и без запуска нумератора. Из обработки (`*.proc.os`) вызов `Док.Записать()` или `Док.Провести()` миновал ПриЗаписи документа, и все расчёты (СуммаНДС, тоталы) оставались невычисленными. Документы создавались без Номер (NULL в колонке `номер`).

Следствие: `Строка.СуммаНДС = nil`, а интерпретатор при `50000 - nil` возвращает nil (не 0!) — nil записывался в БД, `SUM(nil)` = NULL, все итоги обнулялись. Очистка через `НайтиПоНомеру` не находила документы (Номер = NULL).

## 31. `Документы.X.НайтиПоНомеру()` не реализован ✅ закрыто 2026-05-20

Реализовано в `docProxy.CallMethod`: `НайтиПоНомеру` использует `FindCatalogByField` для поиска по полю «Номер», возвращает `Ref` с привязанным `Manager`.

## 32. `Ref` не реализует `MethodCallable` — вызов любого метода на ссылке молча возвращает nil ✅ закрыто 2026-05-20

`Ref` теперь реализует `CallMethod` (MethodCallable): поддерживает `Удалить()`, `ПолучитьОбъект()`, `Пустая()`, `УникальныйИдентификатор()`, а также `Get()` для доступа к `Наименование`/`UUID`. `ПолучитьОбъект()` возвращает саму ссылку (упрощение vs 1С — нет отдельной загружаемой копии). `Об.ПолучитьОбъект().Удалить()` ≡ `Об.Удалить()`.

## 1. `МоментВремени` / `Граница()` у виртуальных таблиц ✅ закрыто 2026-05-21

Платформа поддерживает `.Остатки(&МоментВремени)`: если значение параметра реализует `PointInTime()` (его возвращает `ЭтотОбъект.МоментВремени()` / `this.МоментВремени()`), компилятор строит условие `period < @ ИЛИ (period = @ И recorder != @doc)` — корректные остатки на момент документа с исключением его собственных прежних движений (`internal/query/query.go` → `momentTimeCondition`, `internal/runtime/object.go` → `MomentTime`).

Конфигурация переведена: проведения `РеализацияТоваров`, `СписаниеТоваров`, `ПеремещениеТоваров` запрашивают `ПартииТоваров.Остатки(&МоментВремени)` с `УстановитьПараметр("МоментВремени", this.МоментВремени())`. Теперь перепроведение задним числом не зависит от порядка/дат соседних документов.

Исходный текст:

`Регистр.Остатки()` без параметра возвращает текущее состояние. При перепроведении документа задним числом это «видит будущее» — все движения позже этого документа.

Нужен `.Остатки(&МоментВремени)` с граничной точкой = момент проведения, и предусмотренное API `МоментВремени` у документа в его проведении (для самого себя — `МоментВремени()` или аналог).

## 2. Управляемые блокировки

Сейчас два параллельных проведения реализаций одной и той же номенклатуры оба пройдут up-front проверку остатков и оба запишут движения — в итоге минусовой остаток.

Нужна либо явная `Блокировка` с гранулярностью (Номенклатура, Склад) на чтение `.Остатки()`, либо явный сериализуемый уровень изоляции для процедуры проведения.

## 3. Универсальный предикат «пустая ссылка»

В демо встречается `Поле = ""` — это сравнение ссылки со строкой; оно либо всегда `Ложь`, либо вводит в заблуждение в зависимости от приведений.

Нужен встроенный `ПустаяСсылка(...)` или `ЭтоПустаяСсылка` у любой ссылочной величины, без обхода через `ЗначениеЗаполнено(...) = Ложь`.

## 4. Атрибуты регистров в виртуальных таблицах

`Взаиморасчёты.ТипКонтрагента` объявлен как `attributes`, но в `.Остатки()` отфильтровать по нему нельзя — приходится агрегировать движения вручную.

Варианты решения:
- разрешить условия на атрибуты в виртуальных таблицах;
- либо изначально запретить атрибуты у регистров остатков (только измерения и ресурсы) — это сделает поведение предсказуемым.

## 5. Вызов общих модулей из `*.posting.os` и `*.os`

`утилиты.module.os` существует, но нет внятного способа сослаться на его функции из проведения. Из-за этого ФИФО-логика продублирована между Реализацией, Списанием и Перемещением.

Нужен синтаксис вроде `Утилиты.ФункцияФИФО(...)` или автоматическая экспортируемость в общую область видимости с явной декларацией модуля.

## 6. Нумераторы без области действия

`numerator.period: year` есть, но нет `scope: организация` (или `scope: касса` для денежных).

В мультифирменной базе сквозная нумерация ПКО недопустима — у каждой организации (и часто у каждой кассы) свой счётчик.

## 7. Графы журнала

`field: Контрагент / fallback: [Поставщик, Покупатель]` — на 7 документах терпимо, на 30 это превратится в неуправляемый список разных полей.

Нужен «реквизит журнала», который у каждого включённого документа маппится на своё поле — концепция `ОбщийРеквизит` из 1С.

## 8. «Сумма прописью»

Стандартная для ПКО/РКО функция отсутствует — пришлось писать заглушку. Нужно встроенное `ЧислоПрописью(сумма, валюта)` с правильным склонением рублей и копеек.

## 9. Округление в ФИФО (распределение суммы)

`Окр(КолВоИзПартии * ЦенаПартии, 2)` не гарантирует, что после полного списания партии её `СуммаОстаток` обнулится — копейка-другая «прилипает». В отчётах это видно как остатки партий с нулевым количеством и ненулевой суммой.

Нужна встроенная функция распределения суммы пропорционально количеству с компенсацией остатка в последней доле — либо хранение сумм в копейках/целых.

## 10. Сосуществование YAML- и `.os`-печатных форм

В демо одна и та же `Накладная` лежит и в `printforms/накладная.yaml`, и в `printforms/РеализацияТоваров/накладная.layout.yaml + .os`. Документация не объясняет, какой приоритет и можно ли смешивать.

Нужна одна явная иерархия — например, `.os` форма всегда перебивает YAML-форму с тем же именем, и сообщение в логе при коллизии.

## 11. Демо-режим конкурирует с прод-расписанием

`demo.reset_schedule` в `app.yaml` без явного режима «production / demo» может неожиданно стереть данные на проде, если конфигурацию случайно подняли с включённым демо-блоком.

Нужен явный режим запуска с отдельным набором заданий, баннером и защитой от случайного включения в продакшене (например, отказ запускаться, если `demo.enabled = true` и переменная окружения `ONEBASE_ENV != "demo"`).

## 12. Значения параметров процедур/функций по умолчанию

`Функция РассчитатьНДС(Сумма, Ставка = 20)` парсер не принимает (`expected IDENT, got "="`). Чтобы дать функции «разумное» значение по умолчанию, приходится либо плодить однострочные обёртки (`РассчитатьНДС20` / `РассчитатьНДСПоСтавке`), либо передавать `Неопределено` и проверять внутри.

Нужны параметры по умолчанию в синтаксисе языка — самая базовая вещь, без неё API общих модулей разрастается обёртками.

## 13. В `*.proc.os` фактически разрешена только одна процедура

Попытка завести в `.proc.os` пару процедур (`Выполнить` и вспомогательная `СоздатьТипЦен`) приводит к `unknown function` даже при правильном порядке объявления и переименованной helper-функции. Стенд считает любую вспомогательную процедуру в том же файле невидимой — приходится инлайнить дублирующий код.

Общие модули (`*.module.os`) есть, но из `.proc.os` они тоже не вызываются (см. п.5). В итоге для обработки с любым повторяющимся блоком кода единственный путь — копипаста внутри `Выполнить()`.

Нужно либо разрешить несколько процедур в одном `.proc.os` с обычной видимостью, либо открыть вызов общих модулей из обработок.

## 14. Предопределённые элементы не могут ссылаться друг на друга

В `predefined` элементы создаются по очереди, но внешние ключи проверяются сразу, а не в конце транзакции. Поэтому `Розничная.БазовыйТип: Закупочная` падает с `FOREIGN KEY constraint failed`, хотя обе записи определены в одном `predefined`-блоке.

Решения:
- проверять FK на предопределённые в конце миграции (deferred constraints);
- либо явно гарантировать порядок создания и не падать, если ссылка указывает на ещё-не-вставленную запись из того же блока.

Сейчас приходится либо ограничивать `predefined` записями без cross-ref, либо вручную создавать зависимые элементы (что я и сделал — оставил `Закупочная`, остальные настраиваются обработкой/руками).

## 15. События формы для табличных частей

Авто-подстановка цены при добавлении строки в ТЧ — типовой UX. Сейчас обходимся `ПриЗаписи`, что даёт «подставится только когда сохранишь». Нужен явный набор событий формы: `ПриДобавленииСтроки<ТЧ>`, `ПриИзменении<Поле>`, `ПриАктивизации<ТЧ>` — с документированными сигнатурами и без необходимости отдельной «формы документа» в DSL.

## 16. Нет горячей перезагрузки `*.os` модулей

После редактирования `*.proc.os` / `*.posting.os` / `*.os` платформа продолжает выполнять закэшированный байткод предыдущей версии. Сообщения об ошибках содержат текст, которого уже нет в файле на диске. Нужен либо ручной рестарт процесса, либо команда «Reload configuration», либо watcher файловой системы с автоперекомпиляцией.

Это сильно тормозит разработку — после каждой правки модуля надо вспомнить о перезапуске. Особенно болезненно при отладке обработок, где успех/ошибка зависят от только что внесённого изменения.

## 17. Ref-типы в измерениях регистров не сериализуются в SQL

Объявить измерение регистра как `type: reference:Номенклатура` и затем в проведении присвоить ему `Строка.Номенклатура` (Ref на каталог) — падает с

```
write movement остаткитоваров row 1: sql: converting argument $7 type: unsupported type interpreter.Ref, a struct
```

Платформа не знает, как привести `interpreter.Ref` к скаляру (ID или строка) при INSERT в таблицу регистра. Приходится держать измерения регистров как `type: string` — тогда Ref проходит через свою `String()` и сериализуется по имени (с потерей референциальной целостности).

Странно, что часть демо-конфигурации (например, `ПартииТоваров.Номенклатура: reference:Номенклатура`) работает — значит, либо проблема в комбинации измерений, либо в позиции колонки. Нужен или нормальный сериализатор Ref для всех ref-типов в регистрах, или явная документация «какие типы где допустимы».

## 18. Виртуальное поле `Ссылка` в запросах не работает

`ВЫБРАТЬ Ссылка ИЗ Справочник.ТипЦен` падает на `no such column: ссылка`. При этом в собственном демо платформы (`processors/выгрузка_в_excel`, файл `src/выгрузкавexcel.proc.os`) используется выражение `н.Ссылка = ц.Номенклатура` — значит, либо демо изначально нерабочее, либо колонку переименовали без обратной совместимости.

Нужно либо стабилизировать виртуальную колонку «вся запись как ссылка» (`Ссылка`/`Reference`), либо чётко сказать, что вместо неё использовать (`id`? сам ключ?) и поправить демо.

## 19. Запросы — это «русский синтаксис поверх голого SQLite»

Две связанные проблемы:

**(а) Служебные имена полей.** В запросах фигурирует `period` и `вид_движения` — латинско-snake_case на фоне русско-PascalCase остальных полей. Видно, что под капотом колонки регистра имеют служебные имена, но они утекают в DSL запросов и в условия отчётов.

**(б) Функции не транслируются.** `НачалоДня(period)` падает с `SQL logic error: no such function: началодня` — DSL не превращает русские функции в SQL-эквиваленты. Пришлось писать `date(period)` — SQLite-функцию напрямую, в нижнем регистре. То есть конфигуратор должен знать SQLite-синтаксис, что нивелирует смысл «своего» языка запросов.

Решения:
- унифицировать имена полей с остальной моделью (`Период`, `ВидДвижения`);
- транслировать стандартный набор: `НачалоДня`, `КонецДня`, `НачалоМесяца`, `Год`, `Месяц`, `День`, `Формат` → соответствующие SQL-функции;
- либо хотя бы исчерпывающе задокументировать «что не транслируется и какие SQL-функции разрешены».

## 20. Доступ к предопределённым через `Справочники.X.ИмяПредопределённой`

`Справочники.ТипЦен.Закупочная` возвращает nil, хотя элемент создан декларацией `predefined` в YAML и виден в списке справочника. Стандартный 1С-способ адресации предопределённой записи не работает — приходится искать по `Наименование`.

Нужно либо стабилизировать прямой доступ, либо чётко сказать «адресуйте через `НайтиПоНаименованию`» (см. п.21) и поправить ожидания в документации.

## 21. `НайтиПоНаименованию` возвращает nil для существующих записей

`Справочники.ТипЦен.НайтиПоНаименованию("Закупочная")` возвращает nil, при том что запись «Закупочная» физически есть в справочнике (создана как предопределённая или ручной обработкой). То есть оба способа достать ссылку на существующую запись по имени — не работают.

В сумме с п.18 (виртуальная `Ссылка` в запросе тоже не работает) получается, что в коде нельзя получить ссылку на существующую запись справочника **никаким** способом. Это блокирует все сценарии, где ссылочное поле должно указывать на другую запись того же или соседнего справочника.

## 30. Нет типа параметра «флажок/булево» для обработок

Параметры обработки (`processors/*.yaml`, блок `params`) поддерживают `string` / `number` / `reference:*` / `date`, но булева типа (флажок-чекбокс) нет. Чтобы сделать опцию «вкл/выкл» (например, «Очистить документы перед загрузкой» в `ЗаполнитьТестовуюБазу`), приходится заводить строковый параметр и сравнивать с текстом «да»/«нет» — неудобно для пользователя и подвержено опечаткам.

Нужен `type: bool` для параметра обработки (рендерится чекбоксом) с поддержкой `default`. Связанное: `type: bool` для полей справочников/документов в конфигурации тоже не используется и не подтверждён — стоит проверить и задокументировать.

## 29. Нет цикла `Пока … Цикл`

Поддерживается только `Для Каждого X Из <коллекция> Цикл … КонецЦикла`. Конструкция `Пока <условие> Цикл … КонецЦикла` валит парсер (`unexpected "Цикл" in expression`) и ломает загрузку ВСЕЙ конфигурации.

Из-за этого нельзя применять классический перебор выборки `Пока Выборка.Следующий() Цикл …`; перебор документов (например, для удаления) пришлось делать через `Для Каждого Док Из Документы.X.Выбрать() Цикл …`. Нужен оператор `Пока` — либо явно задокументировать, что его нет, и какой API перебора считается штатным.

## 28. String-измерение регистра нельзя фильтровать ref-параметром

Измерение `ПартииТоваров.Склад` объявлено как `type: string` (workaround П.17). Проведение пишет `Дв.Склад = this.Склад` (ссылка) — в поле сохраняется ИМЯ (`Строка(ссылка)`, напр. «Главный склад «Где-то там»»). Но запрос `ГДЕ Склад = &Склад` с ref-параметром (`УстановитьПараметр("Склад", this.Склад)`) даёт 0 строк — платформа не приводит ссылку к строке в условии. Для настоящих ref-измерений (`Номенклатура`, `Организация`) тот же ref-параметр работает.

Симптом: ФИФО «Недостаточно остатков, доступно 0», хотя партии есть (по `Номенклатура` фильтр находит 107 шт, добавление `И Склад = &Склад` обнуляет).

Обход (применён): передавать `Строка(this.Склад)`. Правильный фикс — перевести string-измерения в `reference:*` (П.17 закрыт). Доп.: `ссылкаНаСклад.Наименование` = `<nil>`, имя даёт только `Строка(ссылка)`.

## 27. Движения проведённых документов не видны внутри той же обработки ❌ снято 2026-05-20 (ложная тревога)

Подтверждено: поступление→реализация в ОДНОМ запуске обработки работают. Прежние «0» были из-за бага фильтра `Склад = &Склад` ref-параметром (см. П.28), а не из-за видимости движений. Пункт недействителен.

Исходный текст:

После закрытия П.26 документы из обработки создаются и проводятся, движения сохраняются — но становятся видны только ПОСЛЕ завершения обработки. Внутри одного запуска `Выполнить()` запрос остатков сразу после `Док.Провести()` возвращает 0.

Диагностика (`ЗаполнитьТестовуюБазу`): создали и провели 2 поступления (Табурет 100 шт на «Главный склад»), тут же
`ВЫБРАТЬ КоличествоОстаток ИЗ РегистрНакопления.ПартииТоваров.Остатки() ГДЕ Номенклатура = &Н И Склад = &С` → **0** (и с фильтром по организации, и без). После завершения обработки те же остатки видны (повторный запуск их находит).

Следствие: «поступление → реализация» нельзя провести за один запуск обработки — реализация по ФИФО всегда видит 0 и падает «Недостаточно остатков». Пришлось делать сид двухэтапным (запуск 1 — поступления, запуск 2 — продажи).

Решения (любое из):
- флашить/коммитить движения сразу после `Док.Провести()`, чтобы последующие запросы в той же обработке их видели;
- либо документировать, что внутри обработки результаты проведения недоступны до её завершения (тогда seed/миграции с зависимыми проведениями надо разбивать на шаги).

Связано с П.1 (МоментВремени) — общая тема видимости движений во времени/транзакции.

## 26. Создание/проведение документов из обработки — no-op ✅ закрыто 2026-05-20

После починки П.25 запись справочников из обработки (`Справочники.X.Создать()/.Записать()`) работает. Но создание и проведение ДОКУМЕНТОВ из обработки по-прежнему ничего не сохраняет:

```
Док = Документы.ПоступлениеТоваров.Создать();
Док.Дата = ТекущаяДата(); Док.Поставщик = ...; ...
Стр = Док.Товары.Добавить(); Стр.Номенклатура = ...; Стр.Количество = 100; Стр.Цена = 500;
Док.Записать();
Док.Провести();
```

Симптом (диагностика в `ЗаполнитьТестовуюБазу`): после создания+проведения 2 поступлений и 2 реализаций —
- `ВЫБРАТЬ Номенклатура ИЗ РегистрНакопления.ОстаткиТоваров.Остатки() ГДЕ КоличествоОстаток > 0` → **0 строк**;
- `ВЫБРАТЬ Номер ИЗ Документ.ПоступлениеТоваров` → **0 строк**;
- исключения не возникает, обработка доходит до конца.

То есть `Документы.X.Создать()/.Записать()` (и `.Провести()`) из `*.proc.os` — no-op, ровно как было со справочниками до П.25. Запись справочников из той же обработки при этом сохраняется, а проведение документов, созданных вручную через UI, работает (движения пишутся).

Нужно: чтобы `Документы.X.Записать()` и `Док.Провести()`, вызванные из обработки, сохраняли документ и его движения — так же, как это уже работает для справочников.

Примечание: синтаксис запроса к документам `ИЗ Документ.<Имя>` рабочий (вернул 0, а не ошибку).

## 25. Транзакции/запись из обработок не работают ✅ закрыто 2026-05-20

В обработке (`*.proc.os`) вызов `НачатьТранзакцию()` падает с `unknown function "НачатьТранзакцию"`. То же ожидается для `ЗафиксироватьТранзакцию()` / `ОтменитьТранзакцию()`.

При этом:
- функции задокументированы в QUICKSTART как штатный DSL (раздел «DSL — встроенные объекты и функции»);
- их использует родной демо-код платформы `processors/импорт_номенклатуры` (`src/импортноменклатуры.proc.os`) — значит этот пример из коробки тоже не работает.

Похоже на рассинхронизацию рантайма с докой/демо. Рядом тот же симптом: импорт зовёт `ReadJSON`/`WriteJSON`, а дока — `ПрочитатьJSON`/`ЗаписатьJSON`.

Связанная (более серьёзная) проблема — ПОДТВЕРЖДЕНА: запись справочников из обработки через `Справочники.X.Создать()/.Записать()` не сохраняется ВООБЩЕ. Диагностика: в одной процедуре `Выполнить()` создаём и пишем 6 контрагентов, тут же запросом `ВЫБРАТЬ Наименование ИЗ Контрагент` — результат **0 строк**. То есть данные не видны даже внутри той же сессии обработки (не «нет коммита», а запись — no-op). При этом предопределённые элементы в базе есть, а проведение документов (движения регистров) работает.

Локализация: запись того же справочника **руками через веб-интерфейс работает** — элемент сохраняется и остаётся после перезагрузки. То есть само хранилище и запись исправны; сломан конкретно путь `Справочники.X.Создать()/.Записать()` из DSL обработки (`*.proc.os`).

Итог: наполнить базу справочниками/документами из обработки сейчас нельзя ничем — ни с транзакцией (функции нет), ни без неё (запись no-op). Это блокер для любых seed/миграционных обработок. Чинить надо именно DSL-метод `Записать()` объекта справочника, вызванный из обработки (через UI он работает).

Нужно: либо реализовать `НачатьТранзакцию()/ЗафиксироватьТранзакцию()/ОтменитьТранзакцию()` (и привести имена `*JSON` к доке), либо задокументировать, что обработки авто-коммитятся, и починить персист `Справочники.X.Записать()` из обработки.

## 24. Нет API «текущий пользователь» для персональных настроек ✅ закрыто 2026-05-20

Классический сценарий «основные значения для подстановки в документы» (склад, организация, тип цен по умолчанию) хранится per-user. Для этого из кода проведения/`ПриЗаписи` нужно знать, кто сейчас работает.

Платформа добавила `ИмяПользователя()` (строка-логин) и `ТекущийПользователь().Имя`. Справочник `НастройкиПользователя` теперь персональный: хелпер `НастройкиПоУмолчанию()` ищет запись с наименованием = `ИмяПользователя()`, при отсутствии — запись «Общие».

## 23. `Движения.X.Добавить()` для периодического регистра сведений не сохраняется ✅ закрыто 2026-05-20

В проведении документа конструкция

```
ДвЦ = Движения.ЦеныНоменклатуры.Добавить();
ДвЦ.Период       = this.Дата;
ДвЦ.Номенклатура = Строка.Номенклатура;
ДвЦ.ТипЦен       = ТипЗакупочной;
ДвЦ.Цена         = Строка.Цена;
```

не приводит ни к ошибке, ни к появлению строки в регистре `ЦеныНоменклатуры`. Та же конструкция для регистров накопления (`ОстаткиТоваров`, `ПартииТоваров`, `Взаиморасчеты`) работает.

Воспроизведено и на отдельном документе `УстановкаЦен`, который пишет ТОЛЬКО в `ЦеныНоменклатуры` — после проведения отчёт по регистру пуст.

Диагностика: `Для Каждого Дв Из Движения.ЦеныНоменклатуры Цикл … КонецЦикла` сразу после `.Добавить()` даёт 0 итераций — то есть коллекция движений в проведении документа для периодического регистра сведений вообще не материализуется.

Решения (любое из):
- поддержать запись в `ЦеныНоменклатуры` через `Движения.X.Добавить()` так же, как для регистров накопления;
- либо явно задокументировать другой API — `РегистрыСведений.X.СоздатьМенеджерЗаписи()` / `РегистрыСведений.X.Записать(<НаборЗаписей>)` — и поправить демо (там используется именно `Движения.X`);
- либо ввести явный режим регистра «подчинение регистратору» в YAML и принимать `Движения.X` только когда документ объявлен регистратором.

Сейчас единственный наблюдаемый способ положить запись в `ЦеныНоменклатуры` — миграция/seed; из проведения это невозможно.

## 22. Присвоение внутри `Если...Иначе` не выходит за границы блока ✅ закрыто 2026-05-21 (тот же фикс, что и П.39)

Дублирует П.39. Блочные конструкции больше не создают дочерних scope — переменные, впервые присвоенные внутри блока, видны снаружи.

Исходный текст:

В коде вида

```
Действие = "?";
Если ЗначениеЗаполнено(Эл) Тогда
  Действие = "Обновлён";
Иначе
  Действие = "Создан";
КонецЕсли;
Сообщить(Действие);
```

`Сообщить` выводил `<nil>` — переменная была локальна к Если/Иначе-блокам. В 1С/OneScript scope = функция, и это типичный паттерн.

```yaml
catalogs/типцен.yaml:
  fields:
    - Наименование
    - ВидРасчета: enum (Хранимый / Наценка / Скидка)
    - БазовыйТип: reference:ТипЦен  # self-reference
    - Процент: number
    - Округление: number
  predefined:
    - Закупочная: { ВидРасчета: Хранимый }
    - Розничная: { ВидРасчета: Наценка, БазовыйТип: Закупочная, Процент: 30 }
    - Оптовая:   { ВидРасчета: Наценка, БазовыйТип: Закупочная, Процент: 10 }
```

`ПолучитьЦену(Номенклатура, ТипЦен, Дата)` рекурсивно поднимается по `БазовыйТип` и применяет процент.

**Это не взлетело из-за совокупности уже описанных проблем.** Чтобы расширенная схема заработала, в платформе нужно починить (минимально):

1. **П.14** — предопределённые с cross-ref внутри одного справочника (deferred FK / порядок вставки).
2. **П.20** — стабильный доступ к предопределённой через `Справочники.X.ИмяПредопределённой`.
3. **П.21** — рабочий `НайтиПоНаименованию` (или другой способ получить ссылку по имени из кода).
4. **Миграция схемы** — при изменении типа поля в YAML (например, `reference:X` → `string`) платформа обязана перестроить таблицу справочника. Сейчас этого не происходит: остаётся старая колонка, новые присваивания тихо отбрасываются. Если бы миграция работала, было бы проще итеративно подбирать рабочую схему.

Дополнительно полезные, но не критичные для именно этого кейса:
- П.5 — вызов общих модулей из `*.proc.os` (чтобы `ПолучитьЦену` лежала в `утилиты.module.os`, а не дублировалась).
- П.13 — несколько процедур в `*.proc.os` (вынести `СоздатьЕслиНет(Имя, ...)` вспомогательной).
- П.22 — нормальный scope переменных в `Если/Иначе` ✅ закрыто вместе с П.39.

С этими четырьмя пунктами расширенная схема становится тривиальной — без обработок-настройщиков, без констант с процентами, без копипасты в проведении Поступления. Только справочник и одна функция `ПолучитьЦену` в утилитах.

## 41. Журнал изменений (`_audit`) всегда пуст

Таблица `_audit` существует в БД, но платформа не пишет в неё записи ни при каких операциях (создание/изменение/удаление справочников и документов, проведение). Запрос `SELECT COUNT(*) FROM _audit` → 0.

Диагностировано через dbquery (C:\Project\onebase-1\tools\dbquery) на `TradeDemo.db`. Нужен аудит действий пользователей для любой учётной системы.

## 42. В просмотре регистров отображаются UUID вместо представлений

Через «Все функции → Регистры накопления → ОстаткиТоваров» в колонках `Номенклатура`, `Организация` (reference-измерения) видны UUID, а не наименования. При этом `resolveRegisterRows` в handlers.go существует и разрешает UUID → имя для части сценариев, но для прямого просмотра регистра не вызывается.

Для string-измерений (`Склад`) отображается имя корректно.

## 43. Виджеты дашборда не показывают данные

Несколько проблем:

**(а) Часовой пояс.** Запросы виджетов с фильтром по дате (`WHERE period >= date('now', '-30 days')`) работают в UTC, а данные записаны в MSK (+3). Виджет «Продажи за 30 дней» показывает 0.

**(б) `scope: current_user`.** Виджет «Мои последние документы» фильтрует по текущему пользователю, но таблица `_users` пуста — nobody logged in → пустой результат.

**(в) Русские SQL-ключевые слова.** Виджеты с `КОЛИЧЕСТВО(*)`, `СУММА()`, `СГРУППИРОВАТЬ ПО` — неизвестно, транслирует ли DSL их в корректный SQL (см. П.19).

## 44. ПриЗаписи получает UUID-строки вместо Ref — ЗначениеРеквизитаОбъекта падает

`runOnPostCtx` (проведение) вызывает `enrichHeaderRefs` — заменяет UUID-строки в ссылочных полях шапки на `*Ref{UUID,Name}`. Но `runOnWriteCtx` (ПриЗаписи) этого не делает. Результат: при сохранении документа из UI (кнопка «Записать») все ссылочные поля шапки — сырые UUID-строки.

Любой документ, чья ПриЗаписи вызывает `ЗначениеРеквизитаОбъекта(this.<ссылочноеПоле>, ...)`, падает:
- ПКО: `ЗначениеРеквизитаОбъекта(this.Касса, "Вид")`
- РКО: `ЗначениеРеквизитаОбъекта(this.Касса, "Вид")`
- Поступление на Р/С: `ЗначениеРеквизитаОбъекта(this.БанковскийСчёт, "Вид")`
- Списание с Р/С: `ЗначениеРеквизитаОбъекта(this.БанковскийСчёт, "Вид")`
- Установка цен: `ЗначениеРеквизитаОбъекта(this.ТипЦен, "ВидРасчета")`
- Реализация (ПриЗаписи): `ЗначениеРеквизитаОбъекта(this.Покупатель, "ТипЦенПродажи")`
- Счёт: `ЗначениеРеквизитаОбъекта(this.Покупатель, "ТипЦенПродажи")`

Фикс: добавить `enrichHeaderRefs` в `runOnWriteCtx` симметрично `runOnPostCtx`. Одна строка в handlers.go. Связано с П.37.
