# Конвертер форм OneBase ↔ 1С — руководство

Двусторонний конвертер между управляемыми формами OneBase (`.form.yaml` +
`.form.os` + `_resources/`) и формами Enterprise-системы (`Form.xml` +
`Module.bsl` + `Items/*`).

- **Импорт**: `onebase forms convert-from-1c`
- **Экспорт**: `onebase forms convert-to-1c`
- **Валидация**: `onebase forms validate`

Также в UI конфигуратора (узел **◇ Управляемые формы**) есть форма импорта
с выбором файлов и кнопка экспорта в редакторе каждой формы.

См. также: [`docs/forms.md`](forms.md) — спецификация формата `.form.yaml`.

## Импорт из 1С

```powershell
onebase forms convert-from-1c `
  --src "C:\Projects\АА5БП3\УТ11УТ11\ПереносДанныхУТ11УТ11_52\Forms\Форма" `
  --entity ПереносДанныхУТ11 `
  --form-name Форма `
  --dst C:\Projects\my-project
```

| Флаг          | Описание                                                            |
|---------------|---------------------------------------------------------------------|
| `--src`       | Путь к каталогу `Forms/<FormName>` или прямо к `Ext/` внутри.       |
| `--entity`    | Имя сущности OneBase, к которой привязывается форма.                |
| `--form-name` | Имя формы. По умолчанию — имя каталога `Forms/<X>`.                 |
| `--form-kind` | `object` (по умолчанию `custom`).                                   |
| `--dst`       | Корень проекта OneBase. Внутри создаётся `forms/<entity>/`.         |

Реальный пример: форма УТ11 «ПереносДанныхУТ11УТ11_52» (5434 строки XML +
Module.bsl 39 540 строк + 9 ресурсов) импортируется за ~1с с 0 errors.
Получаются:

```
<project>/forms/переносданныхут11/
  форма.form.yaml           # 243 KB
  форма.form.os             # 3 MB (212 BSL-процедур)
  форма/_resources/         # 9 PNG/GIF
```

## Экспорт в 1С

```powershell
onebase forms convert-to-1c `
  --src "C:\Projects\my-project\forms\переносданныхут11\форма.form.yaml" `
  --dst "C:\Projects\ut11-export\Forms\Форма\Ext"
```

| Флаг    | Описание                                                                       |
|---------|--------------------------------------------------------------------------------|
| `--src` | Путь к `.form.yaml` в проекте OneBase.                                         |
| `--dst` | Целевой каталог `Forms/<X>/Ext` в выгрузке 1С (создаётся автоматически).       |

Конвертер автоматически:
1. Ищет соседний `.form.os` рядом с `.form.yaml` — если есть, пишет
   `<dst>/Form/Module.bsl` с восстановлением `&НаСервере` / `&НаКлиенте`
   из аннотаций `// @directive=...`.
2. Ищет соседний каталог `<base-name>/_resources/<X>/...` — если есть,
   копирует в `<dst>/Form/Items/<X>/`.
3. Пишет `<dst>/Form.xml` v2.20 с namespace-ами `logform`, `v8`, `cfg`,
   `xr`, `xs`. Заголовки оборачиваются в `<v8:item><v8:lang>ru</v8:lang>...`.

## Валидация

```powershell
onebase forms validate --src .\forms\контрагент\объекта.form.yaml
```

Exit-код 1 если есть error-предупреждения, иначе 0.

Что проверяется:
- Схема `onebase.form/v1`.
- Каждый элемент имеет `kind` (error W010 при отсутствии).
- `data_path` обязателен для `ПолеВвода` / `Флажок` / `Переключатель` /
  `ПолеДаты` / `ПолеСписка` (error W012).
- Каждый реквизит имеет `name` и `type` (error W022).
- Уникальность имён элементов (info W050 при дубликате).
- Неизвестные `kind` — warn W010 (не блокирует, но обращает внимание).

## Предупреждения (W001 – W050)

Каждый warning имеет `code`, `severity` (`info` / `warn` / `error`),
имя элемента, поле, номер строки и текст с подсказкой.

| Код   | Категория            | Пример                                  | Действие                              |
|-------|----------------------|-----------------------------------------|---------------------------------------|
| W001  | I/O                  | Файл не найден                          | Проверить путь                        |
| W002  | Парсер XML           | Неверный XML                            | Открыть Form.xml вручную              |
| W003  | Парсер YAML          | YAML не парсится                        | Запустить `validate`                  |
| W004  | Несовпадение версий  | `<Form version="X">` ≠ 2.20             | Информационно                         |
| W010  | Неизвестный элемент  | `<MagicWidget>` в Form.xml              | Узел сохранён в UnknownXML            |
| W011  | Неизвестное свойство | `<Behavior>` без аналога                | Сохранено в props                     |
| W012  | data_path отсутствует | `ПолеВвода` без `data_path`            | Добавить путь данных                  |
| W013  | Ресурс не найден      | Picture.png отсутствует в Items/       | Проверить путь                        |
| W020  | Composite type      | `<Type>` имеет 2+ `<v8:Type>`           | Свёрнут к первому                     |
| W021  | AnyRef              | `cfg:AnyRef`                            | Информационно: проверить вручную      |
| W022  | Неизвестный тип     | `cfg:NewType.X`                         | Type оставлен без нормализации        |
| W030  | Неотмаппленое событие | `OnDragStart`                         | Сохранено в `props.events_unmapped`   |
| W031  | Обработчик без процедуры | Имя процедуры не найдено в `.form.os` | Добавить процедуру                  |
| W040  | BSL → DSL           | `Новый Запрос`, `ОбщегоНазначения.X`    | Заменить на DSL-аналог                |
| W041  | DSL → BSL           | `Транзакция.Начать()`, `JSON.Decode`    | Заменить на BSL-аналог                |
| W042  | Директива не указана | `// @directive=...` отсутствует        | Поставлен `&НаСервере` по умолчанию   |
| W050  | Manual review       | Дубликат имён, пустые поля              | Просмотреть и подправить              |

Полный список — в `internal/onec_forms/diagnostics.go`.

## Round-trip

Конвертер хранит достаточно метаданных, чтобы цикл `Form.xml → .form.yaml
→ Form.xml'` оставлял семантическое содержимое неизменным:

- Каждому элементу/реквизиту/команде проставляется `original_id` —
  оригинальный `id` из 1С. При экспорте используется он, иначе генерируется
  числовой id ≥ 1000 (вне типичного диапазона 1С).
- Экзотические XML-узлы (ДКС, Conditional Appearance, прочие расширения)
  сохраняются в `UnknownXML` элемента или в `oneC_meta.unknown_xml`
  как base64 — выводятся обратно при экспорте.
- Локализованные заголовки (`<Title>/<v8:item>`) сохраняются как
  `title: {lang: text}` и восстанавливаются дословно.
- Аннотации `// @directive=...` над процедурами в `.form.os` восстанавливают
  `&НаСервере` / `&НаКлиенте` при экспорте.

Конкретные различия в нашем выходе vs оригинале (нормальное поведение):

- XML-форматирование (отступы, порядок атрибутов).
- Не выгружаются «декоративные» namespace-ы которые мы не используем в выводе
  (`dcscor`, `dcssch`, `lf`, `style` и т. п.).
- `<ExtendedTooltip>` с пустыми `id`-ссылками не воспроизводятся.

Регрессии этих свойств покрыты `TestExport_FullRoundTrip` в
`internal/onec_forms/export_test.go`.

## Module.bsl: что переносится автоматически

При импорте — лексер `internal/onec_forms/module_bsl.go`:

- Режет файл на верхнеуровневые директивы и процедуры/функции через
  state-машину; AST не строится.
- Поддерживает директивы `&НаСервере`, `&НаКлиенте`, `&НаСервереБезКонтекста`,
  `&НаКлиентеНаСервереБезКонтекста`. Сохраняются как `// @directive=...`
  над процедурой в `.form.os`.
- Тело процедуры копируется как есть. Параметры (включая `Знач X : Число`)
  и `Экспорт` — сохраняются дословно.
- Комментарии перед процедурой прилипают к ней.

При экспорте — `module_bsl_writer.go`:

- Из `// @directive=...` восстанавливается строка директивы.
- Если аннотации нет — ставится `&НаСервере` + W042 (так безопасно по
  умолчанию: серверный код работает в 1С шире, чем клиентский).
- Тело копируется дословно.

## Что НЕ переносится автоматически

- **ДКС (Data Composition System)** — `<UserSettings>` / `<Filter>` сохраняются
  в `oneC_meta.unknown_xml` и восстанавливаются при обратном экспорте, но
  средствами OneBase не редактируются.
- **Conditional Appearance** — аналогично, через `UnknownXML`.
- **Расширения форм** — отдельный модуль, не покрывается.
- **Семантика BSL** — `Запрос.Выполнить()`, `ОбщегоНазначения.*`,
  `Метаданные.*`, `Сообщить()`, `Состояние()`, клиентские диалоги
  (`ПоказатьВопрос`, `ОписаниеОповещения`), `НСтр(...)`, `СтрШаблон(...)` —
  все эти конструкции BSL не имеют прямого аналога в DSL OneBase. Сканер
  при импорте эмитит W040 с подсказкой. Тела процедур копируются как есть,
  но при первом запуске потребуют ручной адаптации.
- **Стандартные иконки** (`<xr:Ref>StdPicture.X</xr:Ref>`) переносятся как
  `picture: stdpic:X` без копирования файлов. Использование на стороне OneBase
  — TBD (рендерер сейчас не имеет встроенной библиотеки иконок).

## Юридическая чистота

Конвертер работает только с публичными текстовыми форматами выгрузки
конфигурации:

- **Form.xml** и **Module.bsl** — XML/BSL-текст, открытый по спецификации
  Enterprise Development Tools (EDT). Те же файлы используются сторонними
  инструментами (OScript, EDT-плагины, ConfStorage) уже много лет.
- **Парсер XML** написан с нуля на `encoding/xml`. XSD-схемы 1С в репозиторий
  не включаются и не упоминаются.
- **Лексер Module.bsl** — минимальный токенизатор без построения
  синтаксического дерева. Тела процедур копируются как цитаты для
  интероперабельности, без реверс-инжиниринга исполнения.
- **Все маппинги** (типы реквизитов, события, элементы форм) реализованы
  как собственные таблицы соответствий по общедоступной документации.
- **Имена ссылочных типов** (`CatalogRef.X`, `DocumentRef.X`,
  `EnumRef.X` и т. д.) — общеупотребительные термины русскоязычного
  бизнес-софта, не охраняемые товарные знаки.
- **OneBase не аффилирована** с правообладателем формата. См. главный
  дисклеймер в `README.md`.

В UI/CLI/маркетинге используется нейтральная формулировка «совместимость с
XML управляемой формы Enterprise-системы», а не «формат 1С».

## Минимальный пример end-to-end

```powershell
# 1. Импорт.
onebase forms convert-from-1c `
  --src C:\1c-config\Forms\МояФорма `
  --entity ДокументX `
  --form-name Объекта `
  --dst C:\my-project

# 2. Проверка.
onebase forms validate `
  --src C:\my-project\forms\документx\объекта.form.yaml

# 3. (Опционально) — отредактировать в конфигураторе через UI.
#    onebase запускает GUI, открыть базу → ◇ Управляемые формы.

# 4. Экспорт обратно.
onebase forms convert-to-1c `
  --src C:\my-project\forms\документx\объекта.form.yaml `
  --dst C:\1c-config-edited\Forms\МояФорма\Ext
```
