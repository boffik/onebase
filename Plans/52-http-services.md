# Этап 52 — HTTP-сервисы (публикация собственных REST-эндпоинтов на DSL)

**Статус:** ✅ Реализовано — маршрутизация, объекты Запрос/Ответ, аутентификация
none/basic/session, авторизация по ролям, CORS, авто-JSON, дискавери, OpenAPI 3.0,
встроенная интерактивная документация `/hs/docs` (RapiDoc, офлайн)

## Контекст

В платформе уже есть три «исходящих/служебных» механизма интеграции:

| Что | Направление | Кто пишет обработчик |
|---|---|---|
| HTTP-клиент (план 19) | onebase → внешний мир | DSL-скрипт инициирует запрос |
| REST API v1 (`/catalogs`, `/documents`) | внешний мир → onebase | платформа (автогенерация CRUD) |
| Webhooks (план 29) | onebase → внешний мир | конфиг описывает событие → URL |

Чего **не было** — возможности конфигурации **опубликовать собственные входящие
HTTP-эндпоинты с произвольной логикой на встроенном языке**. Это ключевой
механизм 1С — объект **«HTTPСервис»**: разработчик задаёт корневой URL, шаблоны
ресурсов и методы, а тело обработчика пишет на встроенном языке, получая объект
запроса и возвращая объект ответа. Без него невозможны: приём вебхуков от
платёжных систем/мессенджеров с собственной валидацией, мобильные backend-API со
своей бизнес-логикой, обмен с произвольной схемой URL, интеграции, не ложащиеся в
generic-CRUD.

Этот этап закрывает пробел: **HTTP-сервисы onebase** — аналог 1С «HTTPСервис»,
с рядом улучшений с учётом специфики платформы (см. «Чем лучше 1С»).

> **Про «Web-сервисы» (SOAP/WSDL).** В 1С отдельно существуют SOAP «Web-сервисы».
> Сегодня SOAP — нишевый legacy-формат; современные интеграции строятся на
> REST/JSON. Поэтому здесь реализованы именно HTTP-сервисы (REST/JSON) как
> актуальная и более удобная замена. Полноценная SOAP-публикация (генерация WSDL,
> XDTO-типы, SOAP-конверт) — большой и узкоприменимый объём; вынесена в отдельный
> опциональный план (см. «Дальнейшее развитие»).

## Синтаксис / UX

### 1. Метаданные сервиса — `services/<имя>.yaml`

```yaml
name: API
title: Публичное API магазина
root_url: api            # сервис монтируется на /hs/api/…
auth: none               # none | basic | session  (по умолчанию none)
roles: [Менеджер]        # (необяз.) разрешённые роли; пусто — без ограничения
cors:                    # (необяз.) политика CORS для браузерных клиентов
  origins: ["*"]
  headers: ["Content-Type"]
  credentials: false
  max_age: 600
templates:
  - template: /health
    methods:
      GET: Здоровье
  - template: /номенклатура
    methods:
      GET:  СписокНоменклатуры
      POST: СоздатьНоменклатуру
  - template: /номенклатура/{артикул}   # {param} — сегмент-параметр пути
    methods:
      GET: НоменклатураПоАртикулу
  - template: /файлы/{*путь}            # {*param} — жадный хвост (все сегменты)
    methods:
      GET: Файл
```

Префикс `/hs/` повторяет соглашение 1С для http-сервисов. Полный адрес ресурса:
`/hs/<root_url><template>`, например `GET /hs/api/номенклатура/АРТ-001`.

### 2. Обработчики — `src/<имя>.service.os`

Имя файла (без учёта регистра) должно совпадать с `name` сервиса. Каждый метод
шаблона указывает на функцию-обработчик, принимающую `Запрос` и возвращающую
ответ:

```bsl
Функция СписокНоменклатуры(Запрос) Экспорт
    З = Новый Запрос;
    З.Текст = "ВЫБРАТЬ Наименование, Артикул ИЗ Справочник.Номенклатура";
    Результат = З.Выполнить();

    Список = Новый Массив;
    Для Каждого Стр Из Результат Цикл
        Поз = Новый Структура;
        Поз.Вставить("наименование", Стр.Наименование);
        Поз.Вставить("артикул", Стр.Артикул);
        Список.Добавить(Поз);
    КонецЦикла;

    Возврат Список;       // «голый» Массив → JSON-ответ 200 автоматически
КонецФункции

Функция НоменклатураПоАртикулу(Запрос) Экспорт
    Артикул = Запрос.ПараметрыURL.Получить("артикул");
    // …поиск…
    Возврат ОтветJSON(404, Новый Структура("error", "не найдено"));
КонецФункции
```

В обработчике доступен **полный набор DSL-переменных**, как в обработках:
`Запрос`, `Справочники`/`Документы` (создание/запись/проведение), `Перечисления`,
`Константы`, `ПредопределённыеЗначения`, HTTP-клиент, Email, транзакции,
`ТекущийПользователь()`/`ИмяПользователя()`.

## Объекты DSL

### HTTPСервисЗапрос (параметр обработчика)

| Член | Тип | Описание |
|---|---|---|
| `.Метод` | строка | HTTP-метод (GET/POST/…) |
| `.БазовыйURL` | строка | `/<root_url>` |
| `.АдресРесурса` / `.Путь` | строка | полный путь запроса |
| `.ПараметрыURL` | Соответствие | параметры из шаблона (`{id}` → значение) |
| `.ПараметрыЗапроса` | Соответствие | query-string параметры |
| `.Заголовки` | Соответствие | заголовки запроса |
| `.ПолучитьТелоКакСтроку()` | строка | тело как текст |
| `.ПолучитьЗаголовок(имя)` | строка | один заголовок |
| `.ПараметрURL(имя)` / `.ПараметрЗапроса(имя)` | строка | удобный доступ к одному параметру |
| `.ТелоJSON()` | Структура/Массив | **разбор тела как JSON одним вызовом** (сверх 1С) |

### HTTPСервисОтвет

- `Новый HTTPСервисОтвет(КодСостояния = 200 [, ТипСодержимого])`
- `.КодСостояния` — чтение/запись
- `.Заголовки` — живая Соответствие (можно `.Вставить(...)`)
- `.УстановитьТелоИзСтроки(текст)`
- `.УстановитьТелоИзДвоичныхДанных(данные)`
- `.УстановитьТелоJSON(значение)` — сериализует и ставит `Content-Type: application/json` (сверх 1С)
- `.УстановитьЗаголовок(имя, значение)`

### Короткие функции-конструкторы (сверх 1С)

- `ОтветJSON(КодСостояния, Значение)` → готовый ответ с JSON-телом
- `ОтветТекст(КодСостояния, Текст [, ТипСодержимого])` → ответ с текстом

### Авто-сериализация «голого» возврата (сверх 1С)

Если обработчик вернул не `HTTPСервисОтвет`, а обычное значение:
- строка → `200 text/plain`
- Структура/Соответствие/Массив/число/булево → `200 application/json`
- `Неопределено` (нет `Возврат`) → `204 No Content`

## Аутентификация (по сервису)

Поле `auth` в YAML, проверяется до запуска обработчика:

| Режим | Поведение |
|---|---|
| `none` (по умолчанию) | без аутентификации — для публичных API и приёма вебхуков |
| `basic` | HTTP Basic против пользователей базы (`_users`); `401 + WWW-Authenticate` при провале |
| `session` | cookie `onebase_session` или `?_tk=` — как у веб-интерфейса |

При `basic`/`session` пользователь кладётся в контекст: `ТекущийПользователь()`,
`ИмяПользователя()` и **журнал аудита** видят вызывающего, запись данных идёт от
его имени. Сервисы монтируются **вне** session-middleware веб-интерфейса, поэтому
публичные эндпоинты работают без cookie.

### Авторизация по ролям

Необязательный список `roles:` в YAML. Если непуст — вызов разрешён только
аутентифицированному пользователю, у которого есть одна из перечисленных ролей
(администратор проходит всегда). Анонимный или не обладающий ролью пользователь
получает `403`. Подразумевает `auth: basic|session`.

## CORS

Необязательный блок `cors:` (origins/headers/credentials/max_age). При его наличии:
- на все ответы сервиса добавляется `Access-Control-Allow-Origin` (с `credentials`
  эхо-ится конкретный источник, иначе — `*` либо совпавший origin);
- запрос `OPTIONS` (preflight) обрабатывается автоматически: `204` +
  `Access-Control-Allow-Methods` (методы шаблона), `-Allow-Headers`, `-Max-Age`.

## Дискавери и OpenAPI

- `GET /hs` (или `/hs/`) — машиночитаемый JSON-список сервисов (имя, заголовок,
  `root_url`, `auth`, `roles`, шаблоны, методы). Заменяет `?wsdl` из мира SOAP.
- `GET /hs/openapi.json` — спецификация **OpenAPI 3.0** из шаблонов сервисов
  (paths, path-параметры, requestBody для POST/PUT/PATCH, `securitySchemes` для
  basic/session). Без внешних зависимостей; импортируется в Swagger UI/Postman.
- `GET /hs/docs` — **интерактивная документация** (RapiDoc) поверх `openapi.json`
  с «Try it out». RapiDoc вендорится в бинарь (как Monaco/ECharts) — работает
  офлайн, без CDN. Доступ только администратору (на свежей базе без пользователей —
  открыт, как и весь UI).

## Чем лучше 1С

1. **`.ТелоJSON()` / `.УстановитьТелоJSON()` / `ОтветJSON()`** — работа с JSON в
   одну строку вместо ручных `ЧтениеJSON`/`ЗаписьJSON`-потоков.
2. **Авто-сериализация «голого» возврата** — простейший JSON-эндпоинт умещается в
   `Возврат Структура;`.
3. **Жадный параметр `{*путь}`** — проксирующие/файловые сервисы из коробки.
4. **`auth` декларативно на уровне сервиса** (none/basic/session) — не нужно
   возиться с публикацией и веб-сервером, как в 1С.
5. **`GET /hs` дискавери** — список API без внешней документации.
6. **Hot-reload (`--watch`)** — новые сервисы подхватываются без перезапуска.

## Дальнейшее развитие (необязательно)

- **Конфигуратор**: визуальный редактор сервисов и шаблонов.
- **Rate limiting** на уровне сервиса (лимит частоты/размера).
- **Версионирование API** (`/hs/api/v2/…`) — соглашением через `root_url`.

Реализовано в этом этапе: авторизация по ролям, CORS, OpenAPI 3.0, `/hs/docs`.

### SOAP/WSDL («Web-сервисы» в терминологии 1С) — сознательно НЕ делаем

Решение: **не реализуем** до появления конкретного потребителя, умеющего только
SOAP. Обоснование:
- SOAP — устаревающий формат; новые интеграции строятся на REST/JSON, которые уже
  закрыты HTTP-сервисами. SOAP остаётся в legacy (госсектор, банки, старые ESB).
- Полная реализация — недели работы: WSDL, XSD/XDTO-типы, SOAP-конверт, Faults,
  маппинг сложных типов, иногда WS-Security/WS-Addressing.
- Серверная SOAP-экосистема в Go слабая (в отличие от Java JAX-WS / .NET WCF).
- Дублирует уже готовое: бизнес-логика та же (DSL-обработчик), меняется лишь
  формат обмена на более тяжёлый.

Если потребность появится — **дешёвый путь** вместо полного XDTO: тонкий
SOAP-адаптер поверх HTTP-сервисов (приём конверта → вызов DSL-обработчика →
обёртка ответа) + **статический WSDL** для фиксированного контракта (операция =
обработчик, тело как строка/структура). Покрывает простые обмены малой долей
усилий; полный маппинг произвольных типов — отдельный большой план.
