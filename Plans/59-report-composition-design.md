# Компоновка отчётов (лайт-СКД) — дизайн

**Дата:** 2026-06-16 · **План:** [59-report-composition.md](59-report-composition.md) · **Ветка:** `feature/59-report-composition`

## Контекст

Сейчас отчёт (`internal/report/report.go`) плоский: `Query` + `Params` + опциональный
`ChartProc` → таблица. Нет группировок, итогов, drill-down, условного оформления,
декларативных графиков. Это самый частый запрос после базового учёта
(`АнализПроекта-2026-06-10.md` §4.4; `improvement-roadmap.md` §2.3). ECharts завендорён
(`internal/webassets/echarts`), дашборды/виджеты есть — есть на что опереться. Слоя
компоновки нет — это и есть пробел.

## Цель и объём v1

«Лайт-СКД» — полный набор за одну фичу:
- многоуровневые **группировки** + промежуточные/общие **итоги**;
- **drill-down**: сворачивание/разворачивание групп + расшифровка детальной строки →
  исходная запись;
- **условное оформление** (раскраска по правилам);
- декларативные **графики** (ECharts).

Решения, принятые при брейнсторме:
- структуру задаёт **разработчик** в конфигурации (не рантайм-настройки пользователя);
- сборка — **визуальный конструктор в конфигураторе** (генерирует YAML под капотом), с самого начала;
- вычисление — **пост-обработка плоских строк в Go** (диалект-независимо PG/SQLite,
  drill-down и оформление на одних данных, чистая функция → тестируемо);
- **обратная совместимость полная**: блок `composition` опционален, старые отчёты не меняются.

## Функционал для пользователя (рантайм)

```
┌─ Продажи по менеджерам ───────────────────────────────────────────┐
│ Период: [01.06.2026] — [16.06.2026]   [Сформировать]  [⭳Excel] [⭳PDF]│
├───────────────────────────────────────────────────────────────────┤
│   📊  Суммы по менеджерам                                          │
│       Иванов  ███████████████  1.25М                              │
│       Сидоров ▏ -45К                                              │
│       Кузнец. ████████  720К                                      │
├───────────────────────────────────────────────────────────────────┤
│  Менеджер / Клиент              Кол-во        Сумма, ₽            │
│  ▼ Иванов                          140      1 250 000            │
│     ▼ ООО «Ромашка»                 90        980 000            │
│          Реализация №12             50        600 000   →        │
│          Реализация №18             40        380 000   →        │
│     ▶ ИП Петров                     50        270 000            │
│     ··· Итого: Иванов ···          140      1 250 000            │
│  ▼ Сидоров                          30        -45 000            │ ← красным (<0)
│     ▶ ООО «Долги»                   30        -45 000            │
│     ··· Итого: Сидоров ···          30        -45 000            │
│  ▼ Кузнецов                         95        720 000            │
│     ▶ ООО «Альфа»                   95        720 000  ▓▓▓▓      │ ← зелёный фон (>100К)
│  ═══ ВСЕГО ═══                     265      1 925 000            │

```

- многоуровневые группировки с отступами;
- сворачивание/разворачивание групп (drill-down); по умолчанию уровни групп развёрнуты,
  детальные строки свёрнуты;
- промежуточные итоги по группам + общий итог внизу;
- условное оформление (на строку или на отдельную ячейку);
- график над таблицей (столбцы/линия/круг) из верхнего уровня группировки;
- расшифровка: клик по детальной строке (`→`) открывает исходный документ/элемент;
- экспорт в Excel/PDF (с группами и итогами);
- баннер при срабатывании потолка строк.

## Опыт разработчика (конфигуратор)

Визуальный конструктор в карточке отчёта; YAML пишется автоматически (по философии
проекта «YAML для бизнес-пользователя сложен»). Вкладки:

```
┌─ Конфигуратор · Отчёт: Продажи по менеджерам ──────────────────────┐
│ [Запрос] [Структура] [Оформление] [График] [Предпросмотр]          │
├────────────────────────────────────────────────────────────────────┤
│  СТРУКТУРА                                                          │
│  Группировки (перетаскивание = вложенность):                        │
│    ☰ Менеджер  [✕]      ☰ Клиент  [✕]      [ + поле ▾ ]              │
│  Показатели (итоги):                                                │
│    Сумма       агрегат:[Сумма▾]  подпись:[Сумма, ₽]  [✕]            │
│    Количество  агрегат:[Сумма▾]  подпись:[       ]   [✕]            │
│  ☑ Промежуточные итоги  ☑ Общий итог  ☑ Детальные строки            │
│  Сортировка: [Сумма▾] [убыв.▾]                                      │
│  ──────────────────────────────────────────────────────────────    │
│  ОФОРМЛЕНИЕ:  когда [Сумма][<▾][0] → текст[🟥];                     │
│              когда [Сумма][>▾][100000] → фон[🟩]                    │
│  ГРАФИК:  тип[Столбцы▾]  категория[Менеджер▾]  ряд[Сумма]           │
└────────────────────────────────────────────────────────────────────┘
```

- **Запрос** — переиспользует существующий конструктор запросов (`internal/ui/query_builder.go`);
  колонки результата питают остальные вкладки.
- **Структура** — группировки (перетаскивание = вложенность), показатели
  (поле + агрегат sum/count/avg/min/max + подпись), галочки итогов/деталей, сортировка.
- **Оформление** — список правил: условие (поле/оператор/значение) → стиль
  (цвет текста / фон / жирный), опциональное целевое поле.
- **График** — тип, категория (поле группировки), ряды (показатели).
- **Предпросмотр** — запуск с параметрами, готовый отчёт (как существующие вкладки YAML/preview).

Конструктор сериализует всё в блок `composition` YAML — git-friendly хранение сохраняется.

## Формат хранения (YAML)

```yaml
name: ПродажиПоМенеджерам
title: Продажи по менеджерам
params:
  - { name: НачалоПериода, type: date }
  - { name: КонецПериода, type: date }
query: |
  ВЫБРАТЬ Менеджер КАК Менеджер, Клиент КАК Клиент,
         Сумма КАК Сумма, Количество КАК Количество
  ИЗ ... ГДЕ Дата МЕЖДУ &НачалоПериода И &КонецПериода

composition:                       # ← НОВЫЙ опциональный блок (nil = старое поведение)
  groupings: [Менеджер, Клиент]    # порядок = вложенность уровней
  measures:
    - { field: Сумма,      agg: sum, title: "Сумма, ₽" }
    - { field: Количество, agg: sum }
  totals: { grand: true, subtotals: true }
  detail: true
  sort:  [ { field: Сумма, dir: desc } ]
  conditional:
    - { when: "Сумма < 0",      field: "",     style: { color: "#c00", bold: true } }
    - { when: "Сумма > 100000", field: Сумма,  style: { background: "#e6ffed" } }
  chart: { type: bar, category: Менеджер, series: [Сумма] }   # bar|line|pie
```

Структура `Report` дополняется (только добавление, ничего не ломается):

```go
type Report struct {
    // ...существующие поля (Name, Title, Params, Query, ChartProc, External)...
    Composition *Composition `yaml:"composition"` // nil = плоская таблица (старое поведение)
}
type Composition struct {
    Groupings   []string   `yaml:"groupings"`
    Measures    []Measure  `yaml:"measures"`   // field, agg(sum|count|avg|min|max), title
    Totals      Totals     `yaml:"totals"`     // grand, subtotals (bool)
    Detail      bool       `yaml:"detail"`
    Sort        []SortKey  `yaml:"sort"`       // field, dir(asc|desc)
    Conditional []CondRule `yaml:"conditional"`// when(DSL), field(опц., ""=вся строка), style
    Chart       *ChartSpec `yaml:"chart"`      // type, category, series
}
```

## Архитектура (кратко)

- `internal/report` — типы `Composition`/`Measure`/… + парсинг YAML.
- `internal/report/compose` — чистая `Compose(rows []Row, spec report.Composition, ev Evaluator)
  (*Result, error)`: разбивка по уровням группировок, агрегация показателей снизу вверх
  (**деньги через `shopspring/decimal`** — точность), сортировка на каждом уровне по
  доступным полям, условное оформление через интерфейс `Evaluator` (**первое сработавшее
  правило задаёт стиль**); потолок строк (`Capped`). Зависит только от `report` +
  `Evaluator` — без БД/HTTP. **Ядро тестов.**
- `internal/ui/handlers_reports.go` — ветка: `Composition==nil` → старый плоский путь;
  иначе `Compose` → `renderComposed`. Адаптер `interpEvaluator` на интерпретаторе
  (безопасен после плана 52; выражения `when` компилируются один раз на запуск отчёта).
- **Рендер** — единая `<table>`, плоский список `<tr>` c `data-level`/`data-group-path`,
  JS-сворачивание (CSS-класс, `aria-expanded`); ECharts из `Result`. Шаблон — отдельным
  юнитом, **не в god-файл `templates.go`** (в русле плана 55, фронт в `go:embed`).
- **Конструктор** — в `internal/launcher` (конфигуратор): вкладки Структура/Оформление/
  График/Предпросмотр; структурный тест рендера; визуальная приёмка вручную.
- **`onebase check`** — при наличии `composition` валидирует до рантайма: поля группировок/
  показателей/`chart.category`/`chart.series` есть в колонках запроса (запрос и так
  компилируется/исполняется), выражения `when` компилируются, агрегаты на числовых полях.
- **Экспорт** (план 21) — проходит по тому же `Result` (строки групп/подитогов/итога),
  стили (фон/жирность) по возможности маппятся в excelize.

## Совместимость и миграция

Миграции нет. `composition` опционален: отчёты без него — байт-в-байт как сейчас;
`ChartProc` сохраняется и работает; внешние (БД) отчёты не затронуты; `check` для старых
отчётов не меняется.

## Вне объёма v1

- рантайм-настройки пользователя в стиле СКД (менять группировки/отборы на форме отчёта);
- сохранённые пользователем варианты отчёта;
- произвольная расшифровка по любому измерению «на лету» (нужны рантайм-настройки);
- группировки/итоги средствами SQL (`ROLLUP`/`GROUPING SETS`) — диалект-расхождение PG/SQLite.

## Тестирование

- `internal/report/compose/compose_test.go` — **ядро**: плоские строки → дерево групп,
  многоуровневые подитоги, общий итог, сортировка на уровнях, условное оформление
  (фейковый `Evaluator`), потолок строк. Без БД/LLM.
- валидация `check`: неизвестное поле группировки/показателя/категории графика → ошибка
  до рантайма.
- обработчик/интеграция: отчёт с `composition` на SQLite → HTML содержит заголовки групп,
  подитоги, общий итог.
- сериализация данных графика для ECharts.
- структурный тест панели конструктора (вкладки/контролы рендерятся).

## Открытые умолчания (легко изменить)

- группы развёрнуты, детальные строки свёрнуты по умолчанию;
- график над таблицей;
- потолок строк ~50 000 (настраивается).
