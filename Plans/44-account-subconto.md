# Этап 44 — Субконто (аналитика на счетах бухгалтерского учёта)

## Контекст

Этап [11 — Регистр бухгалтерии](11-account-register.md) реализовал ядро (11.1):
план счетов, простые проводки `Дт-Кт`, остатки/обороты по счёту. В самом плане 11
было заложено продолжение **11.2 — субконто** как отдельный документ. Это он.

**Субконто** — аналитические разрезы на счёте (по аналогии с 1С): контрагент,
номенклатура, договор, подразделение и т.п. Без них учёт даёт только сальдо по
счёту целиком, но не отвечает на вопрос «сколько должен конкретный поставщик» или
«остаток какого товара лежит на 41». Это критично для практической бухгалтерии:
ОСВ по субконто, карточка счёта с аналитикой, взаиморасчёты по контрагентам.

### Что выявил аудит конфигурации `examples/accounting` (2026-06-03)

Конфигурация бухгалтерии была доработана в расчёте на субконто:
`accountregs/бухучёт.yaml` содержал блок `subconto:`, posting-модули писали
`Дв.Субконто1 = …` / `Субконто2`, отчёт `КарточкаСчёта` читал `Субконто1`.
**Платформа этого не поддерживает**, поэтому:

- блок `subconto:` молча игнорировался (`rawAccountReg` в
  `internal/metadata/accountreg.go` парсит только `resources`);
- колонки субконто не создавались в `акк_<имя>` (`migrateAccountReg`);
- присваивания `Дв.СубконтоN` в DSL уходили «в никуда» —
  `WriteAccountMovements` пишет только `ar.Resources`;
- отчёт по `Субконто1` в рантайме упал бы на `no such column`;
- **валидатор `onebase check` это не ловил** — запросы к регистру бухгалтерии не
  проверяются по именам колонок (см. этап [39](39-query-engine-correctness.md),
  отдельный техдолг).

Конфигурация временно упрощена (субконто убраны), чтобы пример был рабочим. Этот
план возвращает функциональность уже на уровне движка.

**Зависимости:** этап 11 (регистр бухгалтерии) ✅, этап 5 (виртуальные таблицы) ✅.

## YAML — объявление субконто на регистре

`accountregs/бухучёт.yaml`:

```yaml
name: БухУчёт
title: Бухгалтерский учёт
accounts: Основной
resources:
  - name: Сумма
    type: number
subconto:
  - name: Контрагент
    type: reference:Контрагент
  - name: Номенклатура
    type: reference:Номенклатура
  - name: Подразделение
    type: string
```

Поле субконто = `name` + `type` (как у обычного поля: `reference:<Справочник>`,
`string`, `enum:<…>`). Порядок объявления задаёт нумерацию `Субконто1`,
`Субконто2`, … для краткой записи в DSL.

### Опционально: привязка субконто к счёту (вторая итерация)

В 1С набор субконто задаётся видами субконто на счёте, а не на регистре.
Для MVP достаточно плоского списка субконто на регистре (как выше). Привязку
«какие субконто разрешены на счёте» можно добавить позже как валидацию:

```yaml
# accounts/основной.yaml (будущее расширение)
- code: "60"
  name: Расчёты с поставщиками
  kind: active_passive
  subconto: [Контрагент]        # допустимые разрезы на этом счёте
```

## Проводки в DSL

Два способа адресации (оба регистронезависимы):

```bsl
// По имени субконто
Дв.Субконто.Контрагент = this.Поставщик;
Дв.Субконто.Номенклатура = Стр.Номенклатура;

// По номеру (краткая запись, как в 1С)
Дв.Субконто1 = this.Поставщик;     // первый объявленный субконто
Дв.Субконто2 = Стр.Номенклатура;
```

Незаданные субконто пишутся как NULL. Присваивание несуществующего субконто
(`Субконто3` при двух объявленных, или неизвестное имя) — **ошибка проведения**,
а не молчаливая потеря данных (главный урок аудита).

## Хранилище

### Миграция таблицы регистра

`акк_<имя>` получает колонку на каждый субконто. Тип — как у ссылки/значения:

```sql
ALTER TABLE акк_бухучёт ADD COLUMN субконто1 UUID;   -- reference:Контрагент
ALTER TABLE акк_бухучёт ADD COLUMN субконто2 UUID;   -- reference:Номенклатура
ALTER TABLE акк_бухучёт ADD COLUMN субконто3 TEXT;   -- string
CREATE INDEX idx_бухучёт_сб1 ON акк_бухучёт (счётдт, субконто1, period);
```

Имя колонки: `субконто<N>` по порядковому номеру (стабильно при переименовании
поля). Альтернатива — `субконто_<имя>`; номерной вариант проще для краткой записи
`СубконтоN` и устойчив к переименованию. Решить при реализации, зафиксировать в
DEVELOPER.md.

### Запись движений

`WriteAccountMovements` (`internal/storage/accountreg.go`) сейчас итерирует
`ar.Resources`. Добавить аналогичную итерацию по `ar.Subconto`: собрать колонки
`субконто1..N`, взять значения из движения (`ciGet(row, "СубконтоN")` или по
имени), записать в INSERT. `GetAccountMovements` — добавить субконто в SELECT.

## Виртуальные таблицы (расширение этапа 5 / 11)

Главная ценность — разворот остатков/оборотов по субконто:

```
ВЫБРАТЬ Счёт, Субконто1 КАК Контрагент, СуммаОстатокКт
ИЗ РегистрБухгалтерии.БухУчёт.Остатки(&НаДату, Счёт = "60")

ВЫБРАТЬ Субконто2 КАК Номенклатура, СуммаОборотДт, СуммаОборотКт
ИЗ РегистрБухгалтерии.БухУчёт.Обороты(&Начало, &Конец, Счёт = "41")
```

`AccountBalances` / `AccountTurnovers` (`internal/storage/accountreg.go`) сейчас
группируют только по `a.code`. Нужен режим с дополнительной группировкой по
выбранным субконто-колонкам, когда они присутствуют в `ВЫБРАТЬ`/фильтре.
Это потребует, чтобы компилятор запросов (`internal/query/query.go`) умел
прокидывать список запрошенных субконто в построение виртуальной таблицы.

## Изменения в коде

- `internal/metadata/accountreg.go` — `AccountRegister.Subconto []Field`; парсинг
  `subconto:` в `rawAccountReg` (по аналогии с `resources`).
- `internal/storage/accountreg.go` — `migrateAccountReg` (колонки + индексы),
  `WriteAccountMovements` / `GetAccountMovements` (запись/чтение субконто),
  `AccountBalances` / `AccountTurnovers` (группировка по субконто).
- `internal/runtime/movements.go` (или место сборки движения бухрегистра) —
  поддержка `Дв.СубконтоN` и `Дв.Субконто.<Имя>`; **ошибка** при неизвестном
  субконто.
- `internal/query/query.go` — `СубконтоN` как колонка регистра бухгалтерии в
  обычных запросах и в виртуальных таблицах `.Остатки()/.Обороты()`; разрешение
  имени → `субконто<N>`.
- `internal/configcheck/check.go` — валидация: имена субконто в запросах/движениях
  существуют (закрывает пробел из аудита; перекликается с этапом 39).
- `internal/ui/handlers.go` — отображение субконто в таблице проводок на форме
  документа; колонки субконто в отчётах.
- `internal/launcher/configurator.go` — редактор субконто на форме регистра
  бухгалтерии.
