# План 39: Корректность движка запросов — валидация и семантика (п.45–49)

## Контекст

Аудит всех конфигураций в `OneBaseConfs` (`onebase check` + `describe` + прогон на данных) выявил
класс ошибок, которые **проходят `onebase check`, но ломаются или дают неверный результат в
рантайме**. `check` → `CheckQueries` (`internal/configcheck/check.go:167`) только **компилирует**
запрос через `query.Compile` (DSL → строка SQL), но **не исполняет** его против схемы и данных.

Три уровня ошибок:

| Уровень | Симптом | Кто ловит сейчас |
|---|---|---|
| 1. Не компилируется | синтаксис DSL, неизвестная функция | `check` ✅ |
| 2. Компилируется, но не исполняется | `no such table`, `ambiguous column`, `near "имею"` | никто (рантайм) → **п.45/46/47/48** |
| 3. Компилируется и исполняется без ошибки, но результат семантически неверен | тихо неверные/пустые строки | никто (только на данных) → **п.49** |

Все адреса в исходниках ниже проверены.

---

## п.45. `check` компилирует запросы, но не валидирует против SQL-движка

`CheckQueries` зовёт `query.Compile`, которая отдаёт строку SQL без `prepare`/`EXPLAIN`
(`check.go:167`). `no such table` (VT без скобок), `ambiguous column`, `near "имею"` проходят `check`.

**Предложение:** опционально прогонять `prepare`/`EXPLAIN` каждой скомпилированной строки против
схемы. Предпочтительно — **in-memory схема из метаданных** (работает в CI без живой БД); генерация
физических таблиц (включая бэкенды виртуальных) из метаданных — та же машинерия пригодится для
dry-run миграций. `prepare` (а не только `EXPLAIN`) обязателен — именно он отдаёт `ambiguous column`.
Ловит весь уровень 2 на этапе проверки.

- Трудоёмкость: средняя. Эффект: высокий.

## п.46. Неизвестное ключевое слово молча эмитится как идентификатор

`tr.emit(lower)` (`internal/query/query.go:1426/1452/1455`): `ИМЕЮ`, `ГРУППИРОВАТЬ` без `ПО`,
опечатки ключевых слов уходят в SQL как идентификаторы → синтаксическая ошибка SQLite в рантайме.

**Предложение:** если идентификатор не функция/поле/ссылка и стоит в позиции ключевого слова —
ошибка компиляции «неизвестное ключевое слово ‘ИМЕЮ’; возможно, ‘ИМЕЯ’». После п.45 — это улучшение
текста ошибки, не единственный способ поймать.

- Трудоёмкость: низкая. Эффект: средний.

## п.47. Суффикс виртуальной таблицы без `()` → имя физической таблицы

VT распознаётся только при `tr.peek(5).kind == tLParen` (`query.go:1256`). `РегистрНакопления.X.Остатки`
без скобок уходит в обычный источник → `рег_x.остатки` → `no such table`.

**Предложение:** если после `Тип.Сущность.` идёт известное имя VT (`Остатки`/`Обороты`/`СрезПоследних`…)
**без** `(` — понятная ошибка «виртуальная таблица ‘Остатки’ требует скобки: `.Остатки(...)`».

- Трудоёмкость: низкая. Эффект: средний.

## п.48. Авто-JOIN ссылок не квалифицирует колонки основной таблицы → `ambiguous column`

При запросе к `Документ/Справочник.X` транслятор добавляет `LEFT JOIN <каталог>` для каждого
reference-поля (`query.go:1326-1329`), но собственные неключевые колонки основной таблицы эмитит без
префикса (`tr.emit(col)` на `:1444`). Если присоединённый каталог имеет одноимённую колонку
(`статус`, `дата`, `наименование`) — SQLite падает `ambiguous column`.

**Предложение (точечно):** когда активны авто-JOIN'ы (`len(tr.refDims) > 0`), квалифицировать ссылки
на собственные поля основной таблицы префиксом `tr.mainTable` (отслеживается на `:355`, `:1313`).
Снимает класс без изменения DSL и без ручной квалификации в конфигурациях.

- Трудоёмкость: низкая. Эффект: высокий.

## п.49. Числовые поля на SQLite сравниваются/сортируются как строки

**Корень — в DDL, не в `query.go`.** `SQLiteDialect.TypeNumber` возвращает `"TEXT"`
(`internal/storage/dialect.go:159`) — `number`-поля хранятся как TEXT ради **точного decimal**
(shopspring/decimal, деньги). При сравнении `INTEGER < TEXT` SQLite коэрсит к строковому сравнению.
`PgDialect.TypeNumber` (`dialect.go:94`) даёт настоящий numeric → **баг только на SQLite-бэкенде**.

`check` молчит: `query.Compile` лишь компилирует; `colMap[lower]` на `query.go:1444` несёт имя колонки
**без типа**, поэтому type-aware CAST сейчас невозможен.

Проявления (все — валидный SQL, рантайм-ошибки нет, результат неверный):
- HAVING: `СУММА(...) < Н.МинимальныйОстаток` → `17 < '4'` строкой → `1` (ложное срабатывание).
  Реальный кейс: `widgets/товары_к_заказу.yaml` (PuT).
- колонка vs литерал: `ГДЕ Цена > 100` → TEXT-аффинность литерала → `"50" > "100"` строкой.
- сортировка: `УПОРЯДОЧИТЬ ПО Сумма` → лексикографически («10» раньше «9»). Частый случай топ-N.

**Не решение:** менять `TypeNumber` на `NUMERIC` — аффинность конвертирует «1234.56» → REAL (float)
при вставке → потеря точности денег (ради чего и выбран TEXT). Отклонено.

**Предложение (системно):** type-aware CAST в компиляторе, хранение не трогаем:
- завести `tr.colTypes[lower] = field.Type` рядом с `colMap` (метаданные доступны в `CompileOpts`);
- на эмите (`query.go:1444-1456`): если `colTypes[lower] == number` **и** секция ∈ {WHERE, HAVING,
  ORDER BY, JOIN ON} — эмитить `CAST(<col> AS NUMERIC)` (НЕ `INTEGER` — усекает дробь; в `ВЫБРАТЬ`
  не кастить, чтобы вывод оставался точным TEXT);
- агрегаты не трогать (`СУММА(...)` уже numeric);
- гейт по диалекту: на Postgres не нужно (numeric корректен); `CAST AS NUMERIC` на Pg безвреден.

Стопгап в конфигурациях до фикса: оборачивать сырую колонку `CAST(... AS NUMERIC)` в ГДЕ/ИМЕЯ/УПОРЯДОЧИТЬ.

- Трудоёмкость: низкая (один `colTypes` + гейт на эмите). Эффект: высокий (единственная защита от уровня 3).

## Дешёвый статический lint (до п.45) — `internal/configcheck`

Три проверки без БД, переносят логику внешнего `_analyze.py` в `onebase check`:
- `.Остатки`/`.Обороты`/… без `()` (п.47);
- `ИМЕЮ` и прочие неизвестные ключевые слова (п.46);
- `number`-колонка в сравнении/сортировке без `CAST` на SQLite (п.49) — предупреждение;
- (опц.) служебные `period`/`вид_движения` в запросах.
