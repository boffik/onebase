## Контекст и цель

Дальше по плану 57 ИИ будет генерировать DSL-код конфигурации. Исполнять
сгенерированный (или установленный из marketplace, план 60) код **вслепую опасно**:
он может выйти в сеть, тронуть файловую систему или зациклиться и повесить
однопоточную сессию платформы.

Сегодня средства ограничения **глобальные на весь процесс**: файловая песочница
`fileSandboxRoot` ставится один раз на старте (`SetFileSandbox`), лимит итераций
`maxWhileIter` и глубина рекурсии — константы. Нельзя сказать «вот этому **конкретному
запуску** — строго, а обычному коду конфигурации — как раньше».

**Цель этапа 0:** ввести **профиль ограничений на запуск** — обычные обработки
работают без изменений (нулевая регрессия), а недоверенный запуск ограничен:
без сети, без файлов, с жёстким лимитом времени и итераций.

Этап 0 не имеет пользовательского потребителя (UI/ИИ — этапы 1–2) и проверяется
только тестами. Это осознанно: это «движок поводка» под будущие этапы.

## Принятый подход (A — расширение паттерна guard)

В коде уже есть проверенный паттерн **`NetGuard func() error`** (план 62): перед
каждой сетевой операцией вызывается «вахтёр», который может вернуть ошибку. Guard
внедряется per-run через extraVars-фабрики (`NewHTTPFunctions(guard)`,
`NewEmailFunctions(sender, guard)`) и читается свежим в момент вызова.

Этап 0 повторяет этот приём для остальных осей ограничений, **не меняя сигнатуры
~200 встроенных функций**. Отвергнут подход B («весь профиль в `execCtx`, builtins
читают его оттуда») — builtins имеют фиксированную сигнатуру `(args, file, line)`
без доступа к `execCtx`, и протаскивание потребовало бы инвазивной правки
диспетчеризации ради косметической стройности.

Разделение ответственности:
- **Возможности** (сеть/почта/файлы) → per-run **guard'ы**, внедряются через extraVars-замыкания (как уже сделано для сети).
- **Ресурсы** (wall-clock, итерации) → в **`execCtx`**, проверяются в цикле операторов/`Пока`, где `execCtx` уже доступен.

## Компоненты

### 1. `SandboxProfile` (новый, пакет `interpreter`)

Структура-настройка на стороне сборки запуска. Решает, какие guard'ы внедрить и
какие ресурсные лимиты положить в `execCtx`.

```go
// SandboxProfile описывает, что разрешено одному запуску DSL. Нулевое значение
// (или nil) = «всё разрешено» = поведение по умолчанию (без регрессии).
type SandboxProfile struct {
    AllowNet     bool          // сеть: HTTP-клиент и email
    AllowFile    bool          // файловые builtins
    MaxWallClock time.Duration // 0 = без лимита времени
    MaxLoopIters int           // 0 = дефолт (maxWhileIter)
}

// RestrictedProfile — строгий профиль для недоверенного кода (ИИ/marketplace):
// сеть и файлы запрещены, жёсткий таймаут, ужатый лимит итераций.
func RestrictedProfile() SandboxProfile {
    return SandboxProfile{
        AllowNet: false, AllowFile: false,
        MaxWallClock: 10 * time.Second, MaxLoopIters: 1_000_000,
    }
}
```

Профиль `nil`/zero **не подставляется** обычными путями записи/проведения — они
продолжают внедрять сегодняшние guard'ы (читающие глобальный `net.enabled`) и не
ставят дедлайн. Гарантия нулевой регрессии.

### 2. `FileGuard` (новый, зеркало `NetGuard`)

```go
// FileGuard вызывается перед каждой файловой операцией. nil → без ограничений.
type FileGuard func() error
```

Файловые builtins (`КопироватьФайл`, `ПереместитьФайл`, `УдалитьФайлы`,
`СоздатьКаталог`, `НайтиФайлы`, объект `Файл`, `ЧтениеТекста`/`ЗаписьТекста`)
вызывают `checkFile(guard)` перед операцией — аналогично `checkNet`.

Внедряется per-run так же, как net/email: через `NewFileFunctions(guard)` в
extraVars, перекрывающие глобально зарегистрированные файловые builtins для этого
запуска. **Задача реализации (проверить первым делом):** подтвердить, что extraVars
перекрывают одноимённые записи глобальной карты `builtins` при разрешении имён;
если нет — это первый шаг реализации (resolution order), без него ось «файлы» не
заработает per-run. Глобальный `fileSandboxRoot` остаётся как был (ортогонален).

### 3. Ресурсные лимиты в `execCtx`

Добавить в `execCtx` (план 52):
- `deadline time.Time` (zero = без лимита) — проверяется в начале каждого оператора в `execBlock` (там же, где сейчас пишутся `curFile/curLine`);
- `maxLoopIters int` (0 = `maxWhileIter`) — подменяет константу в проверке циклов `Пока`/`Для`.

При срабатывании дедлайна или лимита — см. «Обработка ошибок».

### 4. Проводка профиля в запуск

Точка, где собираются extraVars и создаётся `execCtx` (`Interp.Run`/`RunWithResult`
и слой, готовящий extraVars в `ui`/`entityservice`), принимает `SandboxProfile`.
Профиль порождает: net-guard, file-guard (в extraVars) и deadline/maxLoopIters
(в `execCtx`). Сигнатуру `Run` расширяем аккуратно (новый необязательный параметр
или вариант `RunSandboxed`) — детали в плане реализации; обычные вызовы остаются
работать без профиля.

## Поток данных

```
Запуск → выбор профиля (по умолчанию = неограниченный)
       → профиль порождает: NetGuard, FileGuard (extraVars) + deadline/iters (execCtx)
       → builtins/цикл проверяют их в момент исполнения
       → запрет возможности → ловимая ошибка; превышение времени → жёсткий стоп
```

## Обработка ошибок

- **Запрет сети/файлов/почты** — обычная `userError` (паника, ловится
  `Попытка/Исключение`), человеческое сообщение: «Сеть запрещена в этом режиме»,
  «Файловые операции запрещены в этом режиме». Согласовано: код **может** это ловить.
- **Таймаут / лимит итераций** — **жёсткий, непойманный** стоп. Реализуется
  отдельным sentinel-типом паники, который `Попытка` **не перехватывает** (в отличие
  от `userError`): обработчик `Попытка` пропускает sentinel дальше по стеку, запуск
  реально завершается. Сообщение: «Превышено максимальное время выполнения» /
  «Превышен лимит итераций цикла». Это не даёт сгенерированному коду «проглотить»
  таймаут и продолжить крутиться.

## Тесты (`sandbox_profile_test.go`)

1. Строгий профиль: `HTTPСоединение`/`ОтправитьПисьмо` → ошибка «сеть запрещена»;
   `КопироватьФайл`/`ЗаписьТекста` → ошибка «файлы запрещены».
2. Wall-clock: бесконечный цикл под профилем с `MaxWallClock` → завершается
   таймаутом в пределах ожидаемого окна; `Попытка` вокруг цикла **не** перехватывает.
3. Лимит итераций: цикл свыше `MaxLoopIters` → жёсткий стоп.
4. Нулевая регрессия: профиль по умолчанию (nil) → сетевой/файловый вызов проходит
   через guard=nil как раньше; существующие тесты интерпретатора зелёные.
5. Ловимость запретов: `Попытка` **перехватывает** запрет сети/файлов и продолжает.

## Follow-up для этапа 2 (по итогам финального ревью)

Этап 0 реализован (`RunSandboxed`, `SandboxProfile.Vars()` с запретом сети/файлов/ИИ,
ресурсные лимиты, жёсткий таймаут). Перед первым реальным потребителем учесть:

- **Footgun «забыли `p.Vars()`».** Запреты возможностей действуют только если
  вызывающий передаёт `p.Vars()` в extraVars (ресурсные лимиты идут из `p`
  отдельно). Рассмотреть «единую точку входа», где `RunSandboxed` сам домерживает
  `p.Vars()` ПОСЛЕ extraVars вызывающего. Внимание к семантике: нулевой
  `SandboxProfile{}` имеет `AllowNet=false/AllowFile=false` (zero value) → авто-merge
  превратит «нулевой профиль» в «запретить возможности». Это безопаснее
  (deny-by-default для тех, кто осознанно зовёт `RunSandboxed`), но меняет текущую
  задокументированную семантику «нулевое = всё разрешено» — решить осознанно.
- **Граница ИИ-функций.** `Vars()` уже перекрывает `ЗапросИИ`/`РаспознатьДокумент`
  и пр. (`llm_builtins.go`). При добавлении новых внедряемых через `dslvars.Build()`
  builtin'ов, которые ходят в сеть/ФС, не забыть закрыть их в `Vars()` (или добавить
  e2e-тест границы, гоняющий вызов через резолвинг env→builtins).
- **Тайминг wall-clock-теста.** `TestSandbox_WallClockHardStop` полагается на то, что
  пустой `Пока Истина` не успеет добежать до `maxWhileIter` (50M) за 50 мс (каждая
  итерация делает `time.Now()`). Запас большой, но при желании — задать высокий
  `MaxLoopIters` или проверять, что счётчик не достиг потолка.
