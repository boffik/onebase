# Этап 42 — Точные десятичные числа (decimal) вместо float64

> **Статус: ✅ Реализовано.** Принятые решения: единое представление `decimal.Decimal`
> для всех чисел DSL; `DivisionPrecision = 16`; `Окр(x, n)` по умолчанию half-away-from-zero,
> режим `1` — банковское. Целочисленные счётчики (`Количество()`, индексы, счётчик `Для`,
> `_version`, `line_number`) остаются int64/float64 — `toDecimal`/`toFloat` принимают оба.
> Чтение из БД: PG `pgtype.Numeric` → `decimal.NewFromBigInt(Int, Exp)` (точно, без float);
> SQLite `TEXT` → `decimal` по типу поля (`normalizeFieldValue`). Запись — через `driver.Valuer`
> shopspring (строка) на обоих движках.
>
> **Нюанс отображения:** `decimal.String()` отбрасывает хвостовые нули, поэтому `100.00`
> выводится как `100`, а `0.10` — как `0.1`. Для фиксированных 2 знаков используйте
> `Формат(...)`/`ФорматнаяСтрока(...)`. Исходный баг (`0.1 * 1.05 → 0.10500000000000001`)
> закрыт: теперь ровно `0.105`. Golden-тесты — `internal/dsl/interpreter/decimal_test.go`,
> round-trip БД — `internal/storage/decimal_roundtrip_test.go`.

## Контекст

Числовой тип DSL сейчас реализован на `float64`. Любой числовой литерал парсится в `float64` (`interpreter.go:385`), вся арифметика (`+ - * /`) и сравнения идут через `toFloat()` → `float64` (`interpreter.go:467–567`, `807`). Для учётной системы это дисквалифицирующая ошибка: двоичная плавающая точка не представляет десятичные дроби точно, и погрешность накапливается.

Воспроизведение (из `examples/trade/src/пересчётЦен.proc.os:26`, `Цена * (1 + Процент/100)`):

```
13:19:00 Пересчёт цен на 5%:
13:19:00 test01: 0.1 → 0.10500000000000001
```

Симптомы в проде: расхождения в суммах проводок, итогах отчётов, сверках остатков; «копеечные» ошибки при суммировании тысяч строк.

**Что уже заложено в проекте (но не доведено):**

- Слой DDL уже выбрал точные типы: PostgreSQL → `NUMERIC`, SQLite → `TEXT` (а **не** `REAL`). См. `internal/storage/dialect.go:94` (`PgDialect.TypeNumber`) и `:159` (`SQLiteDialect.TypeNumber`), с комментариями-намерениями: *«TEXT для денежной точности: shopspring/decimal на стороне Go»* (`:158`, `:41–42`).
- Риск зафиксирован в `Plans/sqlite-support.md` (риск #3): *«Точность чисел… Для денежных сумм это потенциальная проблема. Решение: TEXT с конвертацией в Go через shopspring/decimal. Решить до Этапа 2»*.
- Уже есть локальные костыли против float-погрешности, подтверждающие диагноз: `amount_in_words.go:124` (`int64(math.Round(amount*100))`).

**Чего не хватает:** сама «граница Go» не реализована — `shopspring/decimal` нет в `go.mod`, нигде не импортируется. Данные в БД лежат точно, но **схлопываются в `float64` при чтении**: `storage/crud.go:186–194`, `normalizeValue` конвертирует `pgtype.Numeric` → `float64` через `Float64Value()`. Дальше интерпретатор считает в `float64`.

Этот план доводит до конца уже принятое в проекте направление, а не вводит новое.

## Решение: единый тип `Число` на `shopspring/decimal`

Единый числовой тип DSL поверх `github.com/shopspring/decimal` — фиксированная десятичная арифметика. Это ровно модель 1С: один тип «Число» с настраиваемой точностью, деньги = частный случай (точность 2). Отдельный «денежный» тип не вводим — порождает правила приведения число↔деньги и баги на их границах.

**Отвергнутые альтернативы:**
- *«Decimal только для денег»* — зоопарк типов, неявные приведения, не по-1С.
- *`big.Rat`* — идеальное деление, но ручное округление/форматирование на каждом шаге и неестественный маппинг на `NUMERIC`. Для учёта нужна фиксированная десятичная, а не рациональная.

Зависимость: `github.com/shopspring/decimal` (де-факто стандарт Go для денег; модуль проекта — `github.com/ivantit66/onebase`).

## Семантика (как для пользователя)

Внешне DSL не меняется — числа выглядят так же. Меняется точность вычислений:

```
test01: 0.1 → 0.105        // было 0.10500000000000001
0.1 + 0.2 = 0.3            // было 0.30000000000000004
```

**Деление и точность.** `shopspring/decimal` при делении требует заданной точности (`decimal.DivisionPrecision`, по умолчанию 16) — иначе `10/3` снова «поедет». Фиксируем глобальную точность деления (предлагается 16) и режим округления. В 1С это регулируется точностью результата — поведение повторяемо.

**Округление.** `Окр(Число, Точность, Режим)` с режимами (план B14 в `36-dsl-language-roadmap.md:209`):
- округление-к-большему (round-half-up) — по умолчанию, как `Окр` в 1С;
- банковское (round-half-even).

`shopspring` даёт `Round`, `RoundBank`, `RoundUp/Down` — прокидываем в builtin.

**Целые числа не меняются.** Счётчики, индексы, длина нумератора, `_version`, `line_number`, `Количество()` коллекций остаются целочисленными (int64). decimal — только для типа `number`/`Число`.

## Хранилище / SQL

Схему **менять не нужно** — типы уже правильные:
- PostgreSQL: `NUMERIC` — точное хранение.
- SQLite: `TEXT` — десятичное как строка.

Работа — только на границе Go (чтение/запись), не в DDL:

- **Чтение (PG):** `storage/crud.go:186–194` `normalizeValue` — вместо `pgtype.Numeric → float64` возвращать `decimal.Decimal` (через строковое/`Value()`-представление `pgtype.Numeric`, без промежуточного float).
- **Чтение (SQLite):** `TEXT` приходит как `string` → парсить в `decimal.NewFromString`, а не в `float64` через `toFloat`.
- **Запись (оба):** числовые параметры передавать как `decimal.Decimal` (реализует `driver.Valuer` → строка), а не `float64`. Проверить все места формирования `args` в `crud.go`, `inforeg.go`, `register.go`, `accountreg.go`, `journal.go`.
- **Агрегаты в SQL** (СУММА/СРЕДНЕЕ, виртуальные таблицы Остатки/Обороты) считаются на стороне БД корректно (PG numeric), но результат снова проходит через `normalizeValue` — после фикса п.1 он вернёт decimal автоматически.

## Изменения в коде

### `go.mod`
- Добавить `github.com/shopspring/decimal`.

### `internal/dsl/interpreter/interpreter.go` — ядро
- `evalExpr`/`*ast.NumberLit` (`:384–386`): `strconv.ParseFloat` → `decimal.NewFromString`.
- `evalBinary` (`:467–567`): арифметика `+ - * /` на `decimal` (`Add/Sub/Mul/Div`). Деление — с `DivisionPrecision`. Сохранить спец-семантику `Дата ± Число` (сейчас через `dateAddSeconds(float64)` — взять `Decimal.IntPart()`/`InexactFloat64()` для секунд).
- nil-толерантность (`:518–523`, `:540–564`): пустое + число → число и т.п. — перенести на decimal-ноль.
- `compare` / `equal` (используются на `:486–497`): сравнение чисел через `Decimal.Cmp`, с учётом смешанных int64/decimal/строка-число.
- `toFloat` / `toFloatOr0` (`:755`, `:807–841`): ввести `toDecimal(v any) (decimal.Decimal, bool)` как основной путь; `toFloat` оставить только для мест, которым реально нужен float (даты, тригонометрия, графики).

### `internal/dsl/interpreter/builtins.go`
- `Число()` (`:138`, `:149`): `ParseFloat(strings.ReplaceAll(s, ",", "."))` → `decimal.NewFromString`.
- `Окр`/`round` (`:249–259`): `math.Round(n*p)/p` → `decimal.Round`/`RoundBank` + параметр режима.
- Прочие числовые builtins (мин/макс/abs/целочисленные) — провести по `toDecimal`.

### `internal/dsl/interpreter/interpreter.go` + `builtins_ext.go`, `chart.go`, `debug_hooks.go`, `json_builtins.go`
- Везде `case float64:` (`builtins_ext.go:18`, `chart.go:229`, `debug_hooks.go:125/155`, `json_builtins.go:35`) добавить `case decimal.Decimal:`.
- `Строка(decimal)` → `Decimal.String()` (без хвоста `.000…`); формат с разделителями — в форматтерах.
- JSON-сериализация decimal — как число без потерь (`MarshalJSON` у shopspring это умеет); десериализация числа из JSON — в decimal.

### `internal/storage/crud.go`
- `normalizeValue` (`:179–199`): ветка `pgtype.Numeric` → `decimal.Decimal`; добавить ветку `string` (SQLite TEXT числового поля) — но только для числовых колонок, поэтому конвертацию лучше делать с учётом `metadata.Field.Type` (см. ниже).
- Места формирования `args` для записи — передавать decimal как `driver.Valuer`.

### `internal/storage/*.go` (inforeg, register, accountreg, journal, numerator)
- Точки `Scan(&f float64)` и `var total float64` (есть в тестах и рантайме) — заменить на decimal там, где значение представляет сумму/количество, а не служебный счётчик.

### Маппинг типов (опционально, но желательно)
- `normalizeValue` для SQLite не знает тип колонки. Чтобы не парсить случайные строки в decimal, прокинуть в чтение `entity.Fields[i].Type` и конвертировать в decimal только `FieldTypeNumber`. Иначе — простая эвристика «парсится как decimal → decimal» рискует превратить строковые поля в числа.

### `internal/widget/runner.go` (отчёты/дашборды/KPI)
- `toFloat`, `formatMoney`, `formatInt`, `isInteger` (`:421–514`): принимать decimal; денежное форматирование считать на decimal, не на `(f-whole)*100`.

### `internal/dsl/interpreter/amount_in_words.go`
- `int64(math.Round(amount*100))` (`:124`) → точный `Decimal.Mul(100).IntPart()` (костыль убирается, диагноз исчезает).

## Решения, которые надо зафиксировать до старта

1. **`DivisionPrecision`** — глобальная точность деления. Предложение: 16 (дефолт shopspring). В 1С регулируется точностью результата; 16 покрывает типовые расчёты.
2. **Режим `Окр` по умолчанию** — round-half-up (как `Окр` в 1С), банковское — явным параметром.
3. **Представление в интерпретаторе** — хранить число всегда как `decimal.Decimal`, либо int64 для целых + decimal для дробных. Предложение: единый `decimal.Decimal` для простоты (меньше веток приведения), целые — отдельно только для служебных счётчиков. Решить в начале реализации.
4. **SQLite-чтение** — конвертировать в decimal по `metadata.Field.Type`, а не эвристикой «строка похожа на число».

## Что НЕ трогаем

- `_version` (оптимистичные блокировки, `optimistic_lock.go`), `line_number`, длина нумератора (`Numerator.Length`), `count(*)` — целые.
- `Количество()` коллекций, индексы массивов, границы `Для i = … По …` — int64.
- Даты/время — отдельный тип, decimal не затрагивает (кроме сдвига `Дата ± Число` секунд).
