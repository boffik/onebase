# План 82 - Полнотекстовый / глобальный поиск

Дата проектирования: 2026-07-08.
Статус: ⬜ Проект (пробел из инвентаря `35-1c-feature-inventory.md`).

## Контекст

Поиск сейчас **пооъектный**: список справочника/документа фильтруется по
подстроке в колонках (план 23, `internal/storage/crud.go`). Нет сценария «найти
везде» — единой строки поиска, которая по одному запросу проходит по всем
справочникам и документам базы и возвращает совпадения с переходом к объекту.

Для платформы с большим числом справочников/документов это заметный UX-пробел:
пользователь помнит «где-то был контрагент Ромашка / счёт на 15000 / телефон
+7999…», но не помнит, в каком объекте искать. В 1С это «Полнотекстовый поиск
данных».

## Цель

Глобальный поиск по данным базы:

- одна строка поиска в шапке UI (и REST-эндпоинт) → ранжированные результаты по
  всем разрешённым пользователю объектам;
- переход из результата к карточке объекта;
- учёт прав: **объектный RBAC + строковые политики (план 79)** — в выдачу не
  попадает то, что пользователь не имеет права видеть;
- одинаково на SQLite (dev) и PostgreSQL (prod).

## Хранилище / SQL

Два движка за общим интерфейсом `FullTextIndex` (по диалекту):

- **PostgreSQL** — `tsvector` + GIN-индекс. Материализованная колонка
  `search_tsv` на индексируемых таблицах или отдельная таблица
  `_fts(owner_kind, owner_name, owner_id, tsv)`; запрос — `websearch_to_tsquery`
  с `ts_rank`;
- **SQLite** — виртуальная таблица `FTS5` (`_fts` с `content=`), `MATCH` +
  `bm25()` для ранжирования.

Наполнение:

- какие поля индексировать — декларация в метаданных объекта:
  `fulltext: [Наименование, Комментарий, ...]` (по умолчанию — строковые
  реквизиты представления);
- инкрементально при записи/проведении (`entityservice`) и полный ребилд
  командой `onebase reindex`.

## Синтаксис / DSL (опционально)

Доступ из DSL для сценариев автоматизации:

```
Найдено = ПолнотекстовыйПоиск("Ромашка договор", 20);
Для Каждого Стр Из Найдено Цикл
    Сообщить(Стр.Объект + ": " + Стр.Представление);
КонецЦикла;
```

## Изменения в коде

- `internal/metadata/` — блок `fulltext` (список полей) у `Entity`;
- `internal/storage/fts_pg.go` / `fts_sqlite.go` — интерфейс `FullTextIndex`
  (Index/Delete/Search), DDL, инкрементальное обновление;
- `internal/entityservice/` — хук индексации в транзакции записи;
- `internal/ui/` — строка поиска в шапке, страница результатов (учёт
  `queryRowFilters` / `rowAllowsSelected` из плана 79);
- `internal/api/` — `GET /api/v2/search?q=...` с теми же правами;
- `internal/dsl/interpreter/` — builtin `ПолнотекстовыйПоиск` (за общими
  правами пользователя сессии);
- `internal/cli/` — команда `reindex`.

## Тесты

- индексация/поиск на обоих диалектах: релевантность, морфология (для PG —
  русская конфигурация `russian`), пагинация;
- права: результаты не содержат объектов без object-read и строк вне row-policy;
- инкремент: запись/удаление объекта обновляет индекс; `reindex` восстанавливает
  из данных.

## Verification

1. Завести данные в нескольких справочниках/документах.
2. Ввести фрагмент в глобальный поиск → результаты из разных объектов,
   переход открывает карточку.
3. Под ограниченной ролью (план 79) — чужие строки в выдаче отсутствуют.

## Эстимейт

6–8 дней: интерфейс+PG (2), SQLite FTS5 (1–2), индексация в entityservice (1),
UI+REST (2), DSL+CLI+права-тесты (1–2). Риск — согласованность индекса при
откате транзакций (индексировать в той же транзакции) и производительность
ребилда на больших базах (батчами).
