# План 38: Инструменты для AI-разработки в OneBase — углубление и интеграция

## Контекст

В платформе уже есть базовый набор инструментов, позволяющий ИИ-ассистенту создавать, проверять, понимать и модифицировать конфигурацию OneBase целиком через CLI (headless, текст/JSON):

| Команда | Файл | Что делает |
|---|---|---|
| `onebase init [--template]` | `internal/cli/init.go` | Scaffold проекта: структура директорий, стартовая конфигурация, `AGENTS.md`. |
| `onebase check [--json]` | `internal/cli/check.go` | Валидация: синтаксис `.os`, YAML-схема, неизвестные функции, компиляция запросов. Exit code ≠ 0 → пригодно для CI/pre-commit. |
| `onebase describe` | `internal/cli/describe.go` | JSON-«рентген» конфигурации: метаданные, поля, регистры, перечисления, обработки (с параметрами), модули (имена процедур), список builtins. |
| `onebase ai-guide [--output]` | `internal/cli/aiguide.go` | Генерирует `AGENTS.md` из самой платформы: структура, рабочий цикл, builtins по группам, схема метаданных, модель безопасности. Тот же текст кладёт `init`. |
| `onebase procrun` | `internal/cli/procrun.go` | Офлайн-запуск обработки с печатью `Сообщить()`. |

**Итог текущего состояния:** ИИ может пройти полный цикл «создать → проверить → понять → изменить». Это сильная база. Дальнейшая ценность — в **глубине данных** (чтобы ИИ ошибался реже) и **прямой интеграции** (чтобы ИИ работал без копипасты команд).

## Цель

Сделать OneBase платформой, с которой ИИ-ассистент работает «из коробки» с минимумом ошибок и максимумом автономии: богатый машинный контракт о конфигурации и языке, детерминированные артефакты (без шумных диффов), быстрый sandbox для проверки гипотез и нативная интеграция через MCP.

## Выявленные пробелы (по чтению кода)

1. **`describe` поверхностен для части объектов.** `Reports`, `Subsystems`, `Journals`, `Widgets` сериализуются как простые строки имён (`describe.go:106-110`). Нет: сигнатур процедур (только имена, `procNames`), движений регистров документами, форм, ролей/RBAC, `home_page`, source-локаций (`file:line`). ИИ не видит data-flow и не может точно сослаться на исходник.
2. **Нет сигнатур builtins.** `ai-guide` перечисляет только имена и прямо пишет «сигнатуры смотрите в примерах» (`aiguide.go:100-101`); `KnownBuiltinNames()` возвращает `map[string]struct{}` без типов аргументов/возврата (`builtins.go:368`). Это главный источник ошибок ИИ — неверные аргументы.
3. **`check` без автофикса и без правил «типовых ошибок ИИ».** Есть только `--json` (`check.go:31`). Нет `--fix`, нет линт-правил (неизвестное поле, регистр без измерений, неиспользуемый параметр обработки).
4. **Нет канонизации формата.** Нет `onebase fmt`. Конфигуратор и ручная правка дают разные отступы/порядок ключей (реальный churn: переформатирование `касса.yaml` 2↔4 пробела). ИИ-диффы зашумлены, ревью затруднено.
5. **Нет быстрого sandbox для DSL/запросов.** Есть только `procrun` (обработки) и UI-консоли. ИИ не может headless-проверить выражение или запрос без обёртки в обработку.
6. **Нет машинной схемы YAML.** Нет JSON Schema по видам объектов — ИИ и редакторы не получают валидацию/автокомплит структуры.
7. **MCP только заявлен.** `describe.go:21` называет JSON «контрактом для ИИ и будущего MCP», но сервера нет — ИИ-ассистенты подключаются вручную через shell.

## Предложения (фазами, с приоритетом)

Оценка трудоёмкости: **S** ≈ 0.5–1 день, **M** ≈ 2–4 дня, **L** ≈ неделя+.

### Фаза A — Углубление существующих команд (быстрые победы, низкий риск)

**A1. Обогащение `describe` — `M`.** Расширить `describeOutput` (`describe.go:99-113`):
- Процедуры: параметры с типами и `Экспорт`-флагом (из `ast.Program`/`ast.Procedure`), а не только имена.
- Документы: список движений по регистрам (какой документ в какой регистр пишет) — статически из модулей проведения или из метаданных.
- Новые секции: `forms` (из `internal/metadata/form_module.go`), `roles` (RBAC: права на объекты/программы/формы), `homePage`, `basedOnGraph` (ввод на основании).
- `source` у объектов и процедур: `{file, line}` для точных ссылок.
- Поле `schemaVersion` в корне вывода — версионирование контракта.
- Файлы: `describe.go` (+ хелперы в `internal/project`).

**A2. Реестр сигнатур builtins — `M`.** Добавить метаданные функций: имя, аргументы (имя/тип/опциональность), тип возврата, одностроч­ное описание, группа. Источник — единая таблица рядом с реестром (`internal/dsl/interpreter/builtins*.go`).
- `ai-guide` печатает сигнатуры вместо отсылки к примерам (заменить блок `aiguide.go:94-101`).
- `describe` отдаёт `builtins` структурами, а не строками (`describe.go:206-209`).
- Тест: количество описанных сигнатур == `len(KnownBuiltinNames())` (защита от рассинхрона).
- Файлы: новый `internal/dsl/interpreter/builtin_signatures.go`, правки `describe.go`, `aiguide.go`.

**A3. Линт-правила и `check --fix` — `M`.** Добавить правила поверх `configcheck`: неизвестное поле в форме/запросе, регистр без измерений/ресурсов, неиспользуемый параметр обработки, ссылка на несуществующий объект/перечисление, дублирующиеся `order`/имена. Автофикс безопасных случаев (форматирование, очевидные переименования) под `--fix`.
- Файлы: `internal/configcheck/*`, флаг в `check.go`.

### Фаза B — Новые команды (средний риск)

**B1. `onebase fmt` — `M`.** Детерминированная канонизация YAML и `.os`: фиксированный порядок ключей, отступы, нормализация списков. Идемпотентность (повторный прогон — без изменений). Флаг `--check` для CI. **Конфигуратор должен писать тот же байтовый результат** (устранить churn 2↔4 пробела). Это устраняет источник шумных диффов, который мы наблюдали на практике.
- Файлы: новый `internal/cli/fmt.go`, пакет `internal/canonical/`, интеграция в `saveConfigFile` (`internal/launcher/widget_handlers.go:185`).

**B2. `onebase eval` / `onebase query` — `M`.** Headless-исполнение DSL-выражения/модуля и запроса с печатью результата. Read-only по умолчанию, `--commit` для мутаций (явно). Переиспользует интерпретатор и движок запросов из UI-консолей.
- Файлы: новый `internal/cli/eval.go` (+ переиспользование `internal/dsl/interpreter`, query-движка).

**B3. `onebase examples <kind>` — `S`.** Выдаёт канонический мини-пример per вид объекта (справочник, документ с проведением, регистр, отчёт с запросом, обработка, виджет, форма). `ai-guide` ссылается на примеры — пусть команда их и отдаёт. Источник — те же встроенные шаблоны, что у `init --template`.
- Файлы: новый `internal/cli/examples.go`, данные в `internal/project` (templates).

**B4. `onebase schema [--kind]` — `S`.** Экспорт JSON Schema по видам YAML-объектов (валидация/автокомплит для ИИ и редакторов; основа для `$schema` в файлах конфигурации).
- Файлы: новый `internal/cli/schema.go`, генерация из тегов структур `internal/metadata`.

### Фаза C — Интеграция и анализ (стратегическое)

**C1. MCP-сервер `onebase mcp` (stdio) — `L`.** Обёртка существующих возможностей как Model Context Protocol:
- **Tools:** `check`, `describe`, `eval`, `query`, `procrun`, `examples`, `fmt` (read-only по умолчанию; мутации — за явным флагом/подтверждением).
- **Resources:** `AGENTS.md`, JSON Schema (B4), снимок `describe`.
- Подключение Claude Code / IDE-ассистентов напрямую к проекту без shell-обёрток. `describe.go:21` уже декларирует это направление как контракт.
- Файлы: новый пакет `internal/mcp/`, команда `internal/cli/mcp.go`.

**C2. Анализ влияния `onebase impact --object X [--field Y]` — `M`.** По графу из обогащённого `describe` (A1): что сломается при изменении/удалении поля/объекта (формы, запросы, модули, движения регистров, ввод на основании). Снижает риск правок ИИ.
- Файлы: новый `internal/cli/impact.go`, граф в `internal/project`.

## Рекомендуемая последовательность

```
A2 (сигнатуры) → A1 (describe+) → B1 (fmt) → A3 (lint/--fix)
   → B2 (eval/query) → B4 (schema) → B3 (examples)
      → C1 (MCP, поверх готовых tools) → C2 (impact)
```

Обоснование: A2 и A1 дают ИИ точные данные о языке и базе (наибольший эффект на снижение ошибок при минимальном риске). B1 (`fmt`) убирает накопленный шум в диффах и делает артефакты детерминированными — критично перед расширением автоматических правок. MCP (C1) строится последним, переиспользуя уже готовые и протестированные команды как tools, а не реализуя логику заново.

## Критерии приёмки

- `describe` отдаёт сигнатуры процедур, движения регистров, формы, роли, `home_page`, `source`-локации; `schemaVersion` присутствует; golden-тест стабилен.
- `ai-guide`/`describe` печатают полные сигнатуры builtins; тест ловит рассинхрон с реестром.
- `onebase fmt` идемпотентен; конфигуратор пишет байт-в-байт тот же результат; `--check` зелёный на чистом дереве.
- `onebase eval "<выражение>"` и `onebase query "<запрос>"` печатают результат headless; мутации только под `--commit`.
- `onebase mcp` проходит handshake MCP и выполняет `check`/`describe`/`eval` из ИИ-клиента.

## Безопасность

Сохранить принцип из `ai-guide`: инструменты разработки (`eval`, `query`, `procrun`, MCP-tools) предназначены для **разработки**. Мутирующие операции — только явно (`--commit`/подтверждение в MCP). На опубликованной базе debug/eval-поверхности недоступны без внутреннего токена и сессии администратора. MCP-сервер по умолчанию работает в read-only режиме.

## Заметки

- Все новые команды — headless, детерминированный текст/JSON, ненулевой код возврата при ошибках (единый стиль с `check`/`describe`).
- Контракт версионируется (`schemaVersion`), чтобы ИИ-клиенты и MCP не ломались при эволюции вывода.
- Приоритет — снижение частоты ошибок ИИ (точные сигнатуры и схема) и устранение шума (детерминированный `fmt`), затем автономия (MCP).
