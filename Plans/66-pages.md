# План 66 — Страницы (произвольные представления на DSL)

Произвольная страница — самостоятельное представление, которое автор конфигурации
**программирует** на встроенном языке, когда готовых виджетов/форм не хватает.

- Метаданные: `pages/<имя>.yaml` (`name`, `title`/`titles`, `icon`, `roles`, `params`).
- Обработчик: `src/<имя>.page.os`, `Процедура ПриФормировании(Страница, Параметры) Экспорт`.
- Маршрут: `/ui/page/<имя>` (`internal/ui/handlers_page.go`).
- В меню: через `contents.pages: [...]` в `subsystems/<раздел>.yaml`.

## Страницы vs панели подсистем

Легко спутать: и то, и другое — «сводный экран». Разница — **декларативный
дашборд из готовых виджетов** против **произвольного экрана, собранного кодом**.

| | Панель подсистемы (`home_page`) | Страница (план 66) |
|---|---|---|
| Как описывается | Декларативно, YAML | Императивно, DSL-код |
| Из чего собрана | Виджеты `widgets/*.yaml` (отдельные переиспользуемые объекты) | Блоки построителя `Страница` (живут внутри процедуры) |
| Логика | один `query` на виджет, без кода | циклы, условия, несколько запросов, любые вычисления |
| Параметры | нет | да (`?param=…` → структура `Параметры`) |
| Где живёт | блок `home_page` внутри `subsystems/X.yaml` | `pages/X.yaml` + `src/X.page.os` |
| Открытие | автоматически при выборе подсистемы | по URL `/ui/page/X` (опц. в меню через `contents.pages`) |
| Сколько | одна на подсистему | сколько угодно, не привязаны к разделам |

Аналогия:

- **Панель подсистемы** — «собрать дашборд из готовых плиток»: быстро, но
  ограничено типами виджетов (`kpi`/`chart`/`actions`/`list`). Для обзорного
  экрана раздела из типовых KPI / графиков / списков.
- **Страница** — «написать свой мини-отчёт/форму кодом»: когда нужна нестандартная
  таблица со ссылками на объекты, своя логика выборки, параметры, сложная
  компоновка.

Тот же дашборд-движок рисует и глобальный рабочий стол (`config/home_page.yaml`),
и панели подсистем — это один механизм на разных уровнях. Страница стоит особняком:
отдельный объект навигации со своим адресом.

## Построитель «Страница» (блоки)

`internal/dsl/interpreter/page_builtins.go` — объект `Страница` наполняется блоками:

- `Заголовок(текст)`, `Абзац`/`Текст(текст)`, `Разделитель()`.
- `Показатель(подпись, значение, формат)` — формат `money`/`percent`/`number`.
- `Кнопка`/`Ссылка(текст, url)`.
- `Таблица(заголовок)` → `Колонки(...)`, `ДобавитьСтроку()` → `Установить(колонка, значение)`, `Ссылка(колонка, url)`.
- `Список(заголовок)` → `Пункт(текст, url)`.
- `График(заголовок, вид)` → `Категории(...)`, `Серия(имя, массив)` — рендер через ECharts, как виджеты.
- `ДобавитьСыройHTML(html)` — escape-hatch, санитизируется (`sanitizePageHTML`), «на ответственности автора».

`СсылкаНаОбъект(...)` (`internal/dslvars/dslvars.go`) — URL карточки
`/ui/<вид>/<сущность>/<id>`, две формы:

- `СсылкаНаОбъект(Ссылка)` — вид/сущность берутся из объекта-ссылки (например из `НайтиПоКоду`);
- `СсылкаНаОбъект("Сущность", Ид)` — для результатов запроса, где колонка `Ссылка`
  компилируется в `id` и приходит UUID-строкой без типа (давайте ей псевдоним: `ВЫБРАТЬ Ссылка КАК Ид …`).

## План доработок

### 1. Кнопки-действия (вызов процедуры обработчика) — ✅ СДЕЛАНО

Цель: кнопка на странице вызывает серверную процедуру (провести, экспортировать,
пересчитать…), а не только переходит по URL.

Опора — события управляемых форм (`internal/ui/handlers_managed_events.go`):
фронт шлёт POST, сервер резолвит процедуру и исполняет через `interp.Run`, в ответ
— JSON с `messages`/`values`/`error`. Ключевое: `Registry.GetPageProcedure(pageName,
procName)` уже резолвит **любую** Экспорт-процедуру из `src/<имя>.page.os` по имени —
инфраструктура резолва готова, как у `ПриФормировании`.

Подход:
- DSL: метод `Страница.КнопкаДействие(текст, "ИмяДействия")` → блок `button` с полем
  `Action` (имя процедуры) вместо `URL`.
- Обработчик — `Процедура ИмяДействия(Страница, Параметры) Экспорт` в том же `.page.os`.
- Роутер: `POST /ui/page/{name}/action/{action}` — `GetPageProcedure` → `interp.Call`
  с теми же dslVars (`Страница`, `Параметры`, `MovementsCollector`), сбор `Сообщить()`.
- Результат: проще всего — после действия заново выполнить `ПриФормировании` и
  перерисовать страницу + флеш-сообщения (full reload); позже — AJAX-обновление блоков.
- Файлы: `page_builtins.go` (метод + поле `Action`), `handlers_page.go` (маршрут),
  шаблон `page-custom` (кнопка как submit/AJAX), langref + DEVELOPER.md, тест.
- Объём: ~0.5–1 день.

### 2. Поддержка страниц в конфигураторе лаунчера

Цель: создавать/редактировать страницы в визуальном конфигураторе (сейчас — только в файлах).

Опора — существующие редакторы объектов конфигуратора (формы:
`internal/launcher/forms_handlers.go`, `forms_tmpl.go`; layout-редактор). Страницы — по аналогии.

Подход:
- Список: `page.LoadDir("pages")` (уже есть) → узел в дереве конфигуратора.
- Редактор метаданных: форма для `pages/<имя>.yaml` (name/title/titles/icon/roles/params).
- Редактор обработчика: текст-редактор `src/<имя>.page.os` (как для модулей) + кнопка
  «Проверить» (вызов `configcheck.CheckPages`).
- Создание: шаблон новой страницы (yaml + `.page.os` со скелетом `ПриФормировании`).
- Файлы: `launcher/configurator.go` (маршруты + дерево), новый `pages_handlers.go` +
  шаблон; добавить `page.SaveFile` (сейчас есть только загрузка).
- Объём: ~1–1.5 дня. Перед стартом — свериться, какие объекты конфигуратор уже CRUD-ит.

### 3. i18n подписей блоков

Цель: статические подписи блоков переводятся для нерусских локалей.

Опора — `i18n.Bundle.T(lang, key)` (русский-как-ключ), `s.tr(lang, key)` в `server.go`,
язык из `resolveLang` (пользователь → база → Accept-Language). Служебные строки роутера
уже идут через `s.tr`.

Подход:
- Авто-уровень: при рендере прогонять через `s.tr` **только авторские подписи** —
  `heading.Text`, `paragraph.Text`, `kpi.Label`, `table.Title`, `table.Columns`,
  `list.Title`, `button.Text`, `item.Text`. **Не трогать данные** (`cell.Text`, `kpi.Value`,
  строки таблиц) — это разделение уже отражено в полях `PageBlock`.
- Явный уровень (опц.): DSL-функция `НСтр("текст")` для подписей, собираемых динамически.
- Ключи: автор добавляет русские подписи в `internal/i18n/locales/*.json`; непереведённые
  ловит pre-commit `i18ncheck`.
- Файлы: `handlers_page.go`/шаблон `page-custom` (FuncMap `tr`), langref (если `НСтр`), тест.
- Объём: ~0.5 дня (авто-уровень).
