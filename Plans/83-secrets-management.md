# План 83 - Управление секретами (защита at-rest)

Дата проектирования: 2026-07-08.
Статус: 🟡 Первый шаг сделан (env-ссылки ИИ, PR по итогам ревью); остальное — проект.

## Контекст

Секреты базы (API-ключи ИИ, SMTP-пароль, токены вебхуков/интеграций) хранятся в
конфиге/`_settings` в открытом виде. Уже есть частичные меры:

- SMTP-пароль поддерживает `env:VAR` (`internal/mailer/mailer.go`);
- ключи ИИ поддерживают `${env:VAR}`; после ревью разыменование перенесено в
  `llm.Config.Resolve` — работает и для баз из веб-конфигуратора (`_settings`),
  а не только для `app.yaml`; в выдаче ключи маскируются (`Redacted`);
- сетевой предохранитель `net.enabled` (план 62) глушит исходящие вызовы.

Чего не хватает для «серьёзной» базы: **единого** механизма и защиты значения,
которое всё-таки хранится в базе (когда env-ссылку использовать неудобно —
например, десятки токенов интеграций). Сейчас такой секрет лежит plaintext в
`_settings`/дампе бэкапа/экспорте конфигурации.

## Цель

- единый способ ссылаться на секрет из окружения/секрет-стора **для всех**
  подсистем (ИИ, SMTP, вебхуки, HTTP-сервисы, оборудование);
- опциональное **шифрование at-rest** для значений, которые всё же хранятся в
  базе, на мастер-ключе, живущем вне базы;
- секреты не утекают в `describe`, экспорт конфигурации, бэкап-дампы и журналы.

## Дизайн

**1. Единый резолвер секретов.** Вынести `${env:VAR}` в общий
`internal/secrets` (сейчас логика дублируется в `project` и `llm`). Поддержать
источники: `env:NAME`, `file:/path` (для docker/k8s secrets). Все подсистемы
резолвят через него в момент использования, не при загрузке.

**2. Шифрование at-rest (опционально).** Значение вида `enc:<base64>` в
конфиге — AES-GCM, ключ из `ONEBASE_MASTER_KEY` (env) или файла. Резолвер
расшифровывает в момент использования. Команды:

```
onebase secret set smtp.password        # спросит значение, положит enc:... в _settings
onebase secret set llm.z_ai.api_key
onebase secret rotate --master-key ...   # перешифровать всё на новый ключ
```

Без мастер-ключа `enc:`-значения не расшифровываются → fail-closed (подсистема
отключена, а не падает).

**3. Гигиена вывода.** Единый список «секретных» путей конфигурации; `describe`,
экспорт (`.obz`/ZIP), бэкап-дамп и журналы прогоняются через редактор,
маскирующий/исключающий эти значения (расширить текущий `Redacted` до общего
механизма). Бэкап по умолчанию хранит `env:`/`enc:`-ссылку, не расшифрованное
значение.

## Изменения в коде

- `internal/secrets/` — резолвер (`env:`/`file:`/`enc:`), AES-GCM, загрузка
  мастер-ключа; `project`/`llm`/`mailer` переходят на него;
- `internal/cli/secret.go` — `secret set/rotate/list`;
- `internal/storage/settings.go` — хранение `enc:`-значений;
- `internal/backup/`, `internal/configdb/` (export) — прогон через редактор
  секретов;
- `onebase check` — lint `secret.*`: секрет в открытом виде там, где ждём ссылку.

## Тесты

- резолвер: `env:`/`file:`/`enc:` (успех, отсутствие переменной, битый ключ);
- round-trip шифрования; `rotate` перешифровывает без потери;
- экспорт/бэкап не содержат расшифрованных секретов (проверка по дампу);
- отсутствие мастер-ключа → подсистема disabled, не паника.

## Verification

1. `onebase secret set llm.z_ai.api_key` → в `_settings` лежит `enc:...`, ИИ
   работает при заданном `ONEBASE_MASTER_KEY`.
2. Снять бэкап → грепом убедиться, что ключа в дампе нет.
3. Убрать мастер-ключ → ИИ отключается с понятным сообщением, сервер жив.

## Эстимейт

4–6 дней: общий резолвер+рефактор подсистем (1–2), AES-GCM+CLI (1–2), гигиена
экспорта/бэкапа/журналов (1), тесты (1). Низкий риск (аддитивно поверх рабочих
env-ссылок), но затрагивает несколько подсистем — делать за одним рефактором
резолвера.
