# Этап 11 — Регистр бухгалтерии (план счетов + проводки)

## Контекст

Для финансовых приложений (учёт денежных средств, взаиморасчётов, основных средств) — основа учёта. Текущих регистров накопления (`Приход/Расход`) и сведений недостаточно для двусторонних проводок `Дт-Кт` с балансом по плану счетов.

⚠️ Это большая фича. Рекомендую разбить на 2 итерации:
- **11.1** — ядро: план счетов + простые проводки + остатки/обороты по счёту
- **11.2** — субконто (аналитика на счетах) — отдельный план в будущем

Этот документ — план только для **11.1**.

**Зависимости:** этап 5 (виртуальные таблицы регистров) — для отчётов по счетам.

## YAML — план счетов

`accounts/основной.yaml`:

```yaml
name: Основной
title: Основной план счетов
accounts:
  - code: "01"
    name: Основные средства
    kind: active
  - code: "10"
    name: Материалы
    kind: active
  - code: "41"
    name: Товары
    kind: active
  - code: "60"
    name: Расчёты с поставщиками
    kind: active_passive
  - code: "62"
    name: Расчёты с покупателями
    kind: active_passive
  - code: "90"
    name: Продажи
    kind: passive
  - code: "90.1"
    name: Выручка
    kind: passive
    parent: "90"
  - code: "90.2"
    name: Себестоимость продаж
    kind: active
    parent: "90"
```

Поля счёта:
- `code` — строковый код
- `name` — описание
- `kind: active | passive | active_passive`
- `parent` — для иерархии счетов (опц.)

## Хранилище

### План счетов

```sql
CREATE TABLE _accounts (
    plan      TEXT NOT NULL,
    code      TEXT NOT NULL,
    name      TEXT,
    kind      TEXT NOT NULL,    -- active|passive|active_passive
    parent    TEXT,
    PRIMARY KEY (plan, code)
);
```

При старте — sync из YAML в `_accounts` (upsert).

### Регистр

```sql
CREATE TABLE акк_<имя> (
    id          UUID PRIMARY KEY,
    period      TIMESTAMPTZ NOT NULL,
    регистратор UUID NOT NULL,    -- ссылка на документ-регистратор
    счёт_дт     TEXT NOT NULL,
    счёт_кт     TEXT NOT NULL,
    сумма       NUMERIC NOT NULL
);
CREATE INDEX idx_<имя>_dt ON акк_<имя> (счёт_дт, period);
CREATE INDEX idx_<имя>_kt ON акк_<имя> (счёт_кт, period);
CREATE INDEX idx_<имя>_рег ON акк_<имя> (регистратор);
```

При перепроведении документа — `DELETE WHERE регистратор = $id` и заново.

## Виртуальные таблицы (расширение этапа 5)

```
ВЫБРАТЬ Счёт, СуммаОстаток
ИЗ РегистрБухгалтерии.БухУчёт.Остатки(&НаДату, Счёт = "62")

ВЫБРАТЬ Счёт, СуммаОборот
ИЗ РегистрБухгалтерии.БухУчёт.Обороты(&Начало, &Конец)
```

### Особенность остатков по бухсчетам

Для активных счетов: `остаток = СуммаДт − СуммаКт`. Для пассивных: `остаток = СуммаКт − СуммаДт`. Для активно-пассивных — оба направления показываются отдельно (`СуммаОстатокДт`, `СуммаОстатокКт`).

SQL примерно такой:

```sql
SELECT счёт, 
       SUM(CASE WHEN счёт_дт = счёт THEN сумма ELSE 0 END)
       - SUM(CASE WHEN счёт_кт = счёт THEN сумма ELSE 0 END) AS остаток
FROM акк_<имя>, _accounts
WHERE _accounts.plan = '<план>'
  AND period <= $НаДату
  AND (счёт_дт = _accounts.code OR счёт_кт = _accounts.code)
GROUP BY счёт
```

(Реальный запрос сложнее: нужен учёт kind для смены знака.)
