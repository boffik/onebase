# Этап 39 — Виджеты главной страницы по подсистемам

## Контекст

Стартовая страница (`/ui`) рендерит единственный глобальный
`config/home_page.yaml` независимо от активной подсистемы — в любой подсистеме
показываются одни и те же виджеты (см. реализованный `Plans/31-home-page-widgets.md`).
В 1С командный интерфейс настраивается per-подсистема: у раздела «Закупки» —
свои отчёты и обработки на «рабочем столе», у «Продаж» — свои. Нужно то же:
кликнул «Закупки» → виджеты закупщика (залежалые товары, товары в пути,
заказы поставщикам); другой раздел → другой набор.

Инфраструктура «текущей подсистемы» уже есть и переиспользуется без доработок:
- подсистема живёт в URL как `?subsystem=Имя`;
- вкладки разделов ведут на `/ui/?subsystem={{.Name}}` (`templates.go:480`);
- ссылка «Главная» сохраняет текущий раздел (`templates.go:485`);
- навигация строится под раздел (`buildNavForSubsystem`, `server.go`).
Единственное место, которое игнорирует подсистему — `index()`
(`internal/ui/handlers.go:110`), всегда берущий `s.reg.HomePage()`.

**Решение (AskUserQuestion 2026-05-28):**
- Вариант A — блок `home_page:` прямо в YAML подсистемы, переиспользует
  существующую структуру `HomePage` (раскладка `rows`/`widgets`, `title`,
  `layout`). Всё про раздел — в одном файле, обратно совместимо, ближе к 1С.
- Виджеты остаются в общем пуле `widgets/*.yaml`, ссылаются по имени —
  виджет можно переиспользовать в нескольких разделах.
- Главная без выбранной подсистемы (вход в систему, голый `/ui`) →
  глобальный `config/home_page.yaml` (фолбэк). Раздел без `home_page` —
  тоже фолбэк на глобальный.

**Связанный план:** `Plans/31-home-page-widgets.md` (виджеты и `HomePage`),
`Plans/10-subsystems.md` (подсистемы). Этот план — точечное расширение их обоих.

## Синтаксис / YAML

`subsystems/закупки.yaml` — добавляется опциональный блок `home_page`
(формат идентичен `config/home_page.yaml`):

```yaml
name: Закупки
title: Закупки
icon: shopping-cart
order: 20
contents:
  documents: [ПоступлениеТоваров]
  catalogs:  [Контрагент, Номенклатура, Валюта]
  reports:   [ДвиженияТоваров, Взаиморасчеты]
home_page:                       # ← новое, опционально
  title: Рабочий стол закупщика
  layout: rows                   # grid | rows (как у глобальной)
  rows:
    - widgets: [ЗалежалыеТовары, ТоварыВПути]
    - widgets: [ЗаказыПоставщикам]
```

Виджеты `ЗалежалыеТовары`, `ТоварыВПути`, `ЗаказыПоставщикам` — обычные
`widgets/*.yaml`, как описано в `Plans/31`. Никаких новых типов виджетов.

Поведение:
- `/ui` без `?subsystem=` → глобальный `config/home_page.yaml`.
- `/ui/?subsystem=Закупки`, у раздела есть `home_page` → стол закупщика.
- `/ui/?subsystem=Продажи`, у раздела нет `home_page` → глобальный фолбэк.

## Хранилище / SQL

Изменений схемы БД нет. Конфигурация раздела — файловая (YAML), как и сейчас.

## Изменения в коде

### 1. `internal/metadata/homepage.go` — вынести нормализацию дефолтов
Сейчас дефолты задаются inline внутри `LoadHomePage` (строки 61-70: `Title`
по умолчанию, вывод `Layout` grid/rows по наличию `Widgets`). Вынести в метод,
чтобы переиспользовать для блока подсистемы:

```go
func (h *HomePage) applyDefaults() {
    if h.Title == "" {
        h.Title = "Главная"
    }
    if h.Layout == "" {
        if len(h.Widgets) > 0 {
            h.Layout = "grid"
        } else {
            h.Layout = "rows"
        }
    }
}
```
`LoadHomePage` после `yaml.Unmarshal` вызывает `hp.applyDefaults()`.
`WidgetNames()` и `DisplayTitle()` уже есть — переиспользуются как есть.

### 2. `internal/metadata/subsystem.go` — поле HomePage
- В структуру `Subsystem` (строки 12-19) добавить `HomePage *HomePage`.
- В `rawSubsystem` (строки 44-59) добавить
  `HomePage *HomePage` с yaml-тегом `home_page` (поля `HomePage` уже размечены
  yaml-тегами, размаршалится напрямую вложенным блоком).
- В `LoadSubsystemFile` (строки 76-91) после анмаршала: если
  `raw.HomePage != nil` — вызвать `raw.HomePage.applyDefaults()` и положить в
  возвращаемый `Subsystem.HomePage`.

### 3. `internal/ui/handlers.go` — `index()` учитывает подсистему
Заменить начало `index()` (строка 111):

```go
hp := s.reg.HomePage() // глобальная страница по умолчанию
if sub := r.URL.Query().Get("subsystem"); sub != "" {
    if ss := s.reg.GetSubsystem(sub); ss != nil && ss.HomePage != nil {
        hp = ss.HomePage
    }
}
widgets, defaulted := homePageWidgets(hp, s.reg)
```
Остальная часть `index()` без изменений: `homePageWidgets` резолвит имена через
общий пул (`reg.GetWidget`), отсутствующие виджеты дают «missing»-карточку,
заголовок берётся из `hp.DisplayTitle(lang)`. Используются уже существующие
`GetSubsystem` (`registry.go:565`) и `HomePage` (`registry.go:112`).
Маршруты `/ui` и `/ui/` уже идут в `index` (`server.go:111-112`), а вкладки и
ссылка «Главная» уже передают `?subsystem=`.

### 4. Кеш виджетов
Не трогаем. `widgetCache` кешируется по имени виджета (+ пользователь),
виджеты общие — наборы разделов используют один кеш корректно.

### 5. Пример для демонстрации/проверки
В `examples/trade/subsystems/закупки.yaml` добавить блок `home_page`,
ссылающийся на виджеты из `examples/trade/widgets/`. Для наглядного контраста
с «Продажами» (там своего стола нет → глобальный) можно переиспользовать
существующие виджеты (`ПоследниеДокументы`, `БыстрыеДействия`) и/или добавить
1-2 новых виджета закупок (`widgets/товары_в_пути.yaml` и т.п. на Query Language).

### 6. Конфигуратор (опционально, можно отдельным под-этапом)
`internal/launcher/configurator.go` показывает подсистемы read-only через
`cfgSubsystem` (строки 258-264) — новый блок там не отобразится, но
редактирование подсистемы идёт сырым YAML (шаблон создания — `configurator.go:2061`),
поэтому ключ `home_page` сохраняется при ручном редактировании. Отдельный
визуальный редактор стола раздела — возможный follow-up, в этот этап не входит.
