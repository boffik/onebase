## Контекст и цель

Завершающий этап плана 57 — два эффектных ИИ-помощника в конфигураторе:
1. **Объяснение ошибок `check`** — вывод проверки конфигурации → человеческое
   объяснение + предложенный фикс.
2. **Подсказка запроса** — описание словами → текст запроса OneBase, опираясь на
   реальную конфигурацию.

Оба — **тонкие одиночные LLM-вызовы** (по образцу существующего `cfgAIAssist`,
`internal/launcher/ai_assist.go`): без tool-use, staging, применения. Реальная
проверка — ручная (нужен LLM-ключ + браузер), как в этапе 2b.

## Целевая архитектура

Новый файл `internal/launcher/ai_explain_query.go`. Оба хендлера повторяют структуру
`cfgAIAssist` (получение базы, `getAuthDB`, `GetLLMConfig`, 90с-таймаут, `runner.Run`).

### 3a — Объяснение ошибок (`cfgAIExplain`)
`POST /bases/{id}/configurator/ai-explain` `{ "text": "<вывод check-all>" }`:
```go
var aiExplainSystem = "Ты помогаешь разработчику конфигурации OneBase понять ошибки " +
  "проверки (onebase check). Объясни по-русски, кратко, что означают ошибки и как их " +
  "исправить — по пунктам, с конкретным советом. Не выдумывай: опирайся на текст ошибок."
```
→ `runner.Run("конфигуратор", {System: aiExplainSystem, Messages:[UserText("Вывод проверки:\n"+text)]})`
→ `{ ok, text, model }`.

### 3b — Подсказка запроса (`cfgAIQuery`)
`POST /bases/{id}/configurator/ai-query` `{ "description": "средний чек по менеджерам за месяц" }`:
- системный промпт `queryHintSystem(schema)` = роль + краткий справочник языка запросов
  (ВЫБРАТЬ/ИЗ/ГДЕ/СГРУППИРОВАТЬ ПО; виртуальные таблицы `Регистр.Имя.Остатки(&НаДату)`/
  `.Обороты(&Нач,&Кон)`; параметры `&Имя`; «верни ТОЛЬКО текст запроса») + **срез
  конфигурации** `h.configSchemaText` (этап 1).
- → `runner.Run("конфигуратор", {System: queryHintSystem(schema), Messages:[UserText(description)]})`
  → `{ ok, query: resp.Text, model }`.

### Маршруты
В admin-зоне конфигуратора (рядом с `ai-assist`, server.go):
`r.Post(".../configurator/ai-explain", h.cfgAIExplain)` и
`r.Post(".../configurator/ai-query", h.cfgAIQuery)`.

### UI-хуки (`configurator_tmpl.go`)
JS обоих хуков — в шаблоне `cfg-foot` (там же `runCheckAll` и JS конструктора),
поэтому покрывается `renderCfgFoot`; HTML-кнопки — рядом с целевыми панелями.
- **Объяснение:** кнопка «🤖 Объяснить» в шапке `#check-all-panel` (tab-tree) →
  JS `explainCheckErrors()` в cfg-foot читает `innerText` `#check-all-body`, POST
  `ai-explain`, показывает ответ в `#check-all-explain-out` (через `textContent`).
- **Подсказка запроса:** в `#qb-modal` (cfg-foot) строка: input `#qb-ai-desc` +
  кнопка `#qb-ai-gen` → POST `ai-query`, результат в `#mqb-qry` (готов к «Вставить»).

Содержимое — через `textContent`/`.value` (без XSS). Кнопки безвредны и при
ненастроенном ИИ — эндпоинт вернёт понятную ошибку.

## Тесты

- **Бэкенд:** `queryHintSystem(schema)` — юнит-тест: содержит переданный `schema` и
  ключевые слова языка (`ВЫБРАТЬ`, `Остатки`). Сами хендлеры — тонкие обёртки над
  `llm.Run` (как `cfgAIAssist`), проверяются сборкой; `aiExplainSystem` — константа.
- **UI (структурно, `renderCfgFoot`):** rendered cfg-foot содержит `configurator/ai-explain`,
  `configurator/ai-query`, `explainCheckErrors`, `qb-ai-desc`, `qb-ai-gen`.
- **Ручная приёмка (в браузере, ручной шаг):** «Проверить конфигурацию» с ошибкой →
  «🤖 Объяснить» → понятное объяснение; конструктор запроса → «средний чек по
  менеджерам» → валидный запрос в поле, «Вставить» работает.

## Безопасность

- Эндпоинты — admin-зона конфигуратора (как `ai-assist`).
- Только чтение/генерация текста; ничего не пишут в конфиг и не исполняют.
- UI выводит ответы через `textContent` (без XSS).
- Срез конфигурации (3b) уходит во внешний LLM — та же модель угроз, что у `cfgAIAssist`/чата
  (осознанно, admin-only).

## YAGNI / вне 3
