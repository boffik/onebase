# План 80 - Итоги регистров (предрасчёт остатков)

Дата проектирования: 2026-07-08.
Статус: ⬜ Проект (по итогам ревью масштабируемости; продолжает план 76).

## Контекст

Остатки регистров считаются **на лету** суммированием всех движений:

- регистр накопления — `genBalances` (`internal/query/query.go:1174`):
  `SUM(CASE WHEN вид_движения='Приход' THEN <ресурс> ELSE -<ресурс> END)`
  по всей таблице `рег_*` до момента времени;
- бухрегистр — аналогичный `SUM(CASE WHEN r.счётдт=... )` по `акк_*`.

При десятках-сотнях тысяч движений `Остатки()` / `ОстаткиИОбороты()`
деградируют линейно от объёма истории: каждый отчёт по остаткам и каждая
проверка «хватает ли на складе» в проведении сканируют весь регистр. Свёртка
базы (план 74) сжимает старую историю в входящие остатки, но это **ручная
периодическая** операция, а не инкрементальный механизм.

В 1С ровно эту проблему решают «Итоги регистров»: платформа хранит
предрасчитанные остатки на границах периодов и досчитывает только «хвост» от
ближайшей границы. Это ключевой механизм, делающий учёт с миллионами движений
интерактивным. Сейчас его нет — и это главный оставшийся риск масштаба после
плана 76.

## Цель

Инкрементальные итоги для регистров накопления и бухрегистра, прозрачные для
DSL/запросов: `Остатки()` и `ОстаткиИОбороты()` читают предрасчёт + добираются
хвостом, семантика и синтаксис не меняются. Опционально (флаг), чтобы малые базы
не платили за обслуживание таблиц итогов.

## Хранилище / SQL

Таблица итогов на каждый регистр с включённым предрасчётом:

- `итоги_<рег>` — колонки: границы периода (`period_boundary`), все измерения,
  все ресурсы (накопленный остаток на начало границы), `line_hash`/версия для
  контроля целостности;
- гранулярность границы — месяц по умолчанию (как «периодичность итогов» в 1С),
  настраивается в метаданных регистра: `totals: { enabled: true, period: month }`.

Обслуживание:

- при записи/отмене движений (`entityservice`, `MovementsCollector` flush)
  инкрементально сдвигать итоги затронутых `(измерения, границы ≥ периода
  движения)` в той же транзакции, что и сами движения (консистентно);
- команда `onebase recalc-totals --project ... --register X` — полный
  пересчёт с нуля (восстановление после сбоя/массовой правки);
- свёртка (план 74) переиспользует пересчёт итогов как источник входящих
  остатков.

## Изменения в коде

- `internal/metadata/register.go` — блок `Totals` (enabled, period) + валидация;
- `internal/storage/register.go` — DDL таблицы итогов, инкрементальный upsert
  `applyMovementDelta`, полный `RecalcTotals`;
- `internal/query/query.go` — `genBalances`/`genBalancesAndTurnovers`: если у
  регистра есть итоги, строить `остаток на границе (из итогов) + Σ движений в
  хвосте (граница..МоментВремени)` вместо `Σ по всей таблице`;
- `internal/entityservice/` — хук обновления итогов в транзакции проведения/
  отмены; аккуратно с defer-FK и posting-lock (планы 74/76);
- `internal/cli/` — команда `recalc-totals`;
- `internal/storage/rollup.go` — переиспользовать `RecalcTotals`.

## Тесты

- юнит: остаток через итоги == остаток «Σ по всей таблице» на случайных наборах
  движений (property-style: инвариант равенства двух путей);
- инкремент: запись/отмена/перепроведение документа оставляет итоги
  консистентными (сверка с полным пересчётом);
- граница периода: движения точно на границе, будущей датой, задним числом;
- бенч `BenchmarkBalancesLargeRegister` — предрасчёт vs on-the-fly на 100k+
  движений (ожидаем сублинейность по хвосту), в bench-job CI (план 56).

## Verification

1. Заполнить регистр 200k движений (`loadtest/seed`).
2. `Остатки()` на конце периода: сравнить время и результат с `totals.enabled=false`.
3. Провести/отменить документ задним числом → `recalc-totals` не требуется,
   итоги совпали с полным пересчётом.
