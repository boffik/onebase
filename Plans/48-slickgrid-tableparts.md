# План 48: SlickGrid для редактируемых табличных частей managed-форм

**Дата:** 2026-06-07
**Связанные документы:** [АнализТаблицOPUS.md](../АнализТаблицOPUS.md),
[АнализТаблицGLM.md](../АнализТаблицGLM.md),
[Plans/46-tablepart-commands-and-picker.md](46-tablepart-commands-and-picker.md)

## Цель

Заменить рендер редактируемой табличной части в **managed-формах** (`tp-table`,
`<input>`/`<select>` с именами `tp.{ТЧ}.{idx}.{поле}`) на грид **SlickGrid** ради:
- **Excel-навигации с клавиатуры** (стрелки, Home/End, PageUp/PageDown, Tab/Shift+Tab,
  Enter) — главная причина выбора SlickGrid (см. АнализТаблицOPUS.md §4.2);
- современного вида ячеек/заголовков, выравнивания чисел, строки итогов;
- виртуализации (бонус, не самоцель — ТЧ это 10-500 строк).

**Чего НЕ делаем:** списки/журналы (отдельно, Tabulator — вне этого плана);
табличные документы (`spreadsheet_document.go` — остаются на `toHTML`, не трогаем);
autogen-формы (`templates.go`, `tp-table`) — остаются на улучшенном `<table>`,
миграция autogen вне области.

## Область

Только **managed-формы** (`templates_managed.go`, блок `ТабличнаяЧасть`,
стр. 125-193). Контракт round-trip (`obFire`/`/form-event`) и механизм подбора
(план 46) **сохраняются** — SlickGrid встраивается поверх них, а не вместо.

## Что уже есть и что НЕЛЬЗЯ ломать (выверено по коду)

1. **Round-trip managed-форм** (`internal/ui/handlers_managed_events.go`):
   - `buildObjectFromForm(r, entity)` (стр. 319) → `formToFields` + `parseTablePartRows`;
   - **`parseTablePartRows` читает именованные инпуты `tp.{ТЧ}.{idx}.{поле}`** —
     это текущий контракт сериализации ТЧ;
   - ответ `formEventResponse{Values, TableParts, Messages, PickerData}` (стр. 41-50);
   - `serializeTablePartRowsForEntity` (стр. 263) нормализует регистр имён полей.
2. **Клиентский round-trip** (`templates_managed.go`):
   - `obFire(elementName, eventName, extraParams)` (стр. 520) сериализует форму
     через `FormData(form)` → `URLSearchParams` → POST `/form-event`;
   - `applyTableParts(tps)` (стр. 406) **перерисовывает `tbody#tp-body-<ТЧ>`** из
     JSON-ответа, пересоздавая `<tr>` с именованными инпутами;
   - ветка `if (data.pickerData) { openItemPicker(...) }` (стр. 563) — подбор.
3. **План 46 (подбор/команды ТЧ)** уже частично в коде:
   - `internal/ui/picker.go` — билтин `ПоказатьПодбор`, `parsePickResult`;
   - чекбоксы выделения строк (`._tp-sel`), `data-tp-cmd="1"`, тулбар команд
     (`tpCommandButtons`, стр. 134-146);
   - `selectedTPRows`/`_tp_selected`, `ПодборРезультат`/`_pick_result`.
   - **Важно:** эти контракты завязаны на DOM-строки (`tbody.rows`, `._tp-sel`).
     При переходе на SlickGrid выделение/подбор должны читаться **из модели грида**,
     а не из DOM-строк (см. Фаза 6).
4. **Деньги — на сервере** (decimal). Текущий `recalcTpRow` — клиентский, но
   архитектурно денежные пересчёты должны идти через DSL round-trip. Сохраняем:
   клиент только отображает форматированное значение.
5. **Embed-паттерн вендоров** (`internal/webassets/assets.go`): `//go:embed <dir>`
   + `Handler()` + монтирование `/vendor/<x>/*` в `internal/launcher/server.go:79+`
   и `internal/ui/static.go`. Monaco/ECharts уже так живут — копируем шаблон.

## Ключевые архитектурные решения

### Р1. Мост данных: модель SlickGrid ↔ сериализация формы — через JSON-инпут

SlickGrid держит данные в **своей JS-модели (DataView)**, а не в DOM-инпутах.
Текущий `parseTablePartRows` читает `tp.{ТЧ}.{idx}.{поле}`. Мост:

- На каждую ТЧ-грид рендерим **один скрытый инпут** `<input type="hidden"
  name="tp_json.{ТЧ}">`. Перед сабмитом формы и перед каждым `obFire` JS
  сериализует `dataView.getItems()` в JSON и кладёт в этот инпут.
- На сервере **расширяем `parseTablePartRows`** (или `buildObjectFromForm`)
  веткой: если есть `tp_json.{ТЧ}` — парсим JSON-массив строк; иначе — старый путь
  именованных инпутов (backward-compat с autogen и не-мигрированными ТЧ).
- **Обратный путь** (`applyTableParts`): после round-trip сервер возвращает
  `tableparts` JSON — делаем **SlickGrid-aware** ветку: если для ТЧ есть грид,
  обновляем `dataView.setItems(rows)` + `grid.invalidate()` вместо пересоздания
  `<tr>`. Старая DOM-ветка остаётся для не-мигрированных ТЧ.

Почему JSON-инпут, а не сохранять именованные инпуты из грида: грид не владеет DOM
ячеек постоянно (виртуализация рендерит только видимые), поэтому именованные
инпуты «как раньше» физически не существуют для прокрученных строк. JSON-снимок
модели — единственный надёжный способ. Это совпадает с рекомендацией
АнализТаблицOPUS.md §7.

### Р2. Деньги — серверный round-trip, не клиентский расчёт

Денежные/зависимые колонки (Сумма = Количество × Цена и т.п.) пересчитываются
**на сервере** через существующий обработчик `ПриИзменении`/round-trip:
- редактор числовой ячейки по `onCellChange` грида вызывает `obFire(elementName,
  'ПриИзменении', {_tp:'<ТЧ>', _tp_row: idx})` (или формоуровневый пересчёт);
- сервер пересчитывает в DSL (decimal), возвращает `tableparts` → грид
  обновляется через SlickGrid-aware `applyTableParts`.
- Клиентский `recalcTpRow` для мигрированных ТЧ **не используется**.

Открытый вопрос O3: дёргать round-trip на каждое изменение числовой ячейки может
быть «болтливо». Альтернатива — пересчёт только по уходу с ячейки/строки
(`onActiveCellChanged`) или явной командой. Решить до Фазы 6.

### Р3. Ссылочный редактор — кастомный, с интеграцией серверного picker

Колонка-ссылка (`isRef`) в SlickGrid требует **кастомного редактора** (SlickGrid
из коробки даёт text/число/select/чекбокс, но не «селект с поиском + кнопка
picker»):
- редактор показывает значение как `_label`, по Enter/двойному клику — выпадающий
  список из `TPRefOptions[ТЧ][поле]` (server-preloaded, уже есть в `_tpRefOpts`);
- кнопка «…» внутри/рядом — открывает существующий серверный picker
  `openRefPicker`/`/_ref-open` (НЕ переписываем picker, переиспользуем);
- хранит в модели `{id, _label}`; в JSON-сериализацию (Р1) уходит `id` (UUID),
  как ожидает `parseTablePartRows`/`serializeTablePartRowsForEntity`.

### Р4. Два движка в бинарнике — осознанно

Списки остаются/перейдут на Tabulator (другой план), ТЧ — SlickGrid. Это
зафиксированный компромисс (АнализТаблицOPUS.md §5): цена — два embed и две темы,
выгода — Excel-клавиатура в ТЧ. Тему SlickGrid настраиваем под палитру OneBase
(slate/indigo), чтобы грид не выбивался.

### Р5. Вендоринг SlickGrid (6pac, MIT)

Берём форк **6pac/SlickGrid** (MIT, активный). Нужны: `slick.core`, `slick.grid`,
`slick.dataview`, `slick.editors`, `slick.formatters` + CSS (`slick.grid.css` +
тема). Зависимость **SortableJS** (для drag колонок/строк) — вендорим тоже или
отключаем reorder на старте. Складываем в `internal/webassets/slickgrid/`,
монтируем `/vendor/slickgrid/*`. Single-file сборка предпочтительна; если только
модульная — собираем минимальный bundle офлайн и вендорим результат (UI должен
работать офлайн — решение проекта).

## Фазы (риск ↑ слева направо)

### Фаза 0 — CSS-рефреш (предусловие, можно параллельно)
Не зависит от SlickGrid, снимает саму жалобу «некрасиво». Если после неё жалоба
снята — Фазы 1+ можно отложить. (Детали — АнализТаблицOPUS.md §6, Фаза 0.)

### Фаза 1 — Вендоринг и embed SlickGrid (≈3-5 ч, низкий риск)
1. Положить ассеты в `internal/webassets/slickgrid/` (js + css + тема + SortableJS).
2. `internal/webassets/assets.go`: `//go:embed slickgrid` + `SlickGridHandler()`
   по образцу `MonacoHandler` (Cache-Control immutable).
3. Смонтировать `/vendor/slickgrid/*` в `internal/launcher/server.go` (рядом со
   стр. 83-86) и `internal/ui/static.go` (рядом со стр. 16-19).
4. Тест `internal/webassets/assets_test.go`: GET `/vendor/slickgrid/slick.grid.js`
   → 200, непустое тело (по образцу существующих тестов Monaco/ECharts).

### Фаза 2 — Read-only грид одной ТЧ (≈6-10 ч, низкий риск)
Цель — паритет с текущим **отображением** без редактирования.
1. В `templates_managed.go`, блок `ТабличнаяЧасть`: за флагом (атрибут
   `use_grid: true` у элемента ТЧ или глобальный тумблер) рендерить
   `<div id="sg-{{$tpName}}" class="ob-grid"></div>` вместо `<table>`.
2. Колонки генерируем из `$tpMeta.Fields` (имя/заголовок/тип) в JSON для JS.
3. JS-инициализация: `new Slick.Grid('#sg-<ТЧ>', dataView, columns, options)` с
   формат­терами (число — `text-align:right`, `tabular-nums`, разделители).
4. Данные — из `$tpRows` (server-rendered JSON).
5. `enableCellNavigation:true`, ресайз колонок, тема OneBase.
6. **Проверка:** существующий документ trade открывается, ТЧ отображается гридом,
   значения совпадают, числа выровнены.

### Фаза 3 — Редактирование: текст/число + клавиатура + add/del (≈8-12 ч, средний)
1. Редакторы: `Slick.Editors.Text` / кастомный числовой (decimal-форматирование).
2. `editable:true`, `autoEdit`, проверить **навигацию**: стрелки, Home/End,
   PageUp/PageDown, Tab/Enter — это ключевая фича, тестировать вручную.
3. Кнопка «+ Добавить строку» → `dataView.addItem({})` + фокус на первую ячейку.
4. Удаление строки: колонка-кнопка «×» или Delete на выделенной строке.
5. **Деньги пока локально не считаем** — пустые/ручные значения; серверный
   пересчёт в Фазе 6.

### Фаза 4 — Ссылочный редактор (≈8-12 ч, высокий риск)
Реализация Р3:
1. Кастомный editor: dropdown из `_tpRefOpts[ТЧ][поле]` с фильтром-поиском.
2. Кнопка «…» → `openRefPicker` (серверный, переиспользуем) с возвратом выбранного
   `{id,_label}` в ячейку грида (нужен колбэк из picker в активную ячейку).
3. Формат­тер ссылки показывает `_label`, модель хранит `id`.
4. Лупа «открыть карточку» → `openRefCurrent` по `id` из ячейки.
5. **Проверка:** выбор контрагента/номенклатуры из списка и из picker,
   значение видно как наименование, в модель уходит UUID.

### Фаза 5 — Мост сериализации (≈6-10 ч, высокий риск)
Реализация Р1:
1. Скрытый `<input type="hidden" name="tp_json.{ТЧ}">` на каждую грид-ТЧ.
2. JS: перед сабмитом формы (`submit`-listener) и **в начале `obFire`** —
   сериализовать `dataView.getItems()` (с раскрытием ссылок в UUID) в JSON-инпут.
3. Сервер: расширить `parseTablePartRows` (или `buildObjectFromForm`,
   `handlers_managed_events.go:319` и место сохранения документа) веткой
   `tp_json.{ТЧ}` → `json.Unmarshal` → `[]map[string]any` с нормализацией регистра.
4. SlickGrid-aware `applyTableParts`: если для ТЧ есть грид
   (`window._obGrids[tpName]`) → `dataView.setItems(rows); grid.invalidate()`;
   иначе старая DOM-ветка.
5. **Проверка:** записать документ → перечитать → строки ТЧ сохранены корректно
   (включая ссылки/числа); round-trip-событие обновляет грид, а не ломает его.

### Фаза 6 — Серверный пересчёт денег + интеграция плана 46 (≈8-14 ч, высокий)
Реализация Р2 + перенос контрактов плана 46 на грид:
1. `onCellChange` числовой колонки → `obFire('ПриИзменении', {_tp, _tp_row})` →
   сервер пересчитывает (decimal) → `tableparts` → SlickGrid-aware applyTableParts.
   (Учесть O3 — частота round-trip.)
2. **Выделение строк для команд ТЧ:** вместо чтения `._tp-sel`/`tbody.rows`
   (`obFire` стр. 539-548) — брать выделение из `grid.getSelectedRows()` →
   индексы строк модели → `_tp_selected`. Переписать эту ветку `obFire` для
   грид-ТЧ.
3. **Командная панель ТЧ** (`tpCommandButtons`, тулбар) — оставить как есть (над
   гридом), кнопка по-прежнему зовёт `obFire(name,'Нажатие',{_tp})`.
4. **Подбор (фаза 2 плана 46):** `openItemPicker` возвращает строки через
   `obFire(el,'Выбор',{_pick_result})`; сервер добавляет строки в
   `Объект.<ТЧ>` → ответ `tableparts` → грид обновляется. Проверить, что
   диалог подбора корректно дописывает строки в грид (а не в старый tbody).
5. **Проверка:** ввод накладной с клавиатуры; изменение количества пересчитывает
   сумму на сервере; «Подбор» добавляет отмеченные позиции в грид; команда
   «изменить выделенные» применяется к выделенным строкам грида.

### Фаза 7 — Тесты, тема, референс-конфигурация (≈4-8 ч)
1. Go-юнит: парсинг `tp_json.{ТЧ}` в `buildObjectFromForm`
   (`handlers_managed_events_test.go`) — корректные строки, ссылки в UUID,
   нормализация регистра.
2. Тема SlickGrid под палитру OneBase (CSS-переменные).
3. Референс-форма в `examples/trade` (НЕ PuT — решение проекта) с грид-ТЧ:
   ввод накладной, подбор, пересчёт суммы.
4. Документация: как включить грид у ТЧ (флаг `use_grid`), ограничения.

## Открытые вопросы (решить до кодинга)

- **O1. Включение грида:** глобально для всех managed-ТЧ или флаг `use_grid: true`
  у конкретной ТЧ в YAML формы? Предложение: **флаг на ТЧ** — постепенная
  миграция, безопасный откат, autogen не трогаем.
- **O2. Формат сериализации ссылок в `tp_json`:** хранить в модели `{id,_label}`,
  в JSON отдавать только `id` (UUID-строка) — совпадает с текущим
  `serializeTablePartRowsForEntity`. Подтвердить, что `_label` не нужен серверу.
- **O3. Частота серверного пересчёта денег (Р2):** на каждое изменение ячейки vs
  по уходу со строки vs явная команда. Предложение: **по `onCellChange`
  числовых колонок с debounce**, измерить «болтливость» на реальной накладной.
- **O4. SortableJS:** вендорить (drag колонок/строк) или отключить reorder в
  v1? Предложение: **отключить в v1** (минус зависимость, проще embed), вернуть
  позже при запросе.
- **O5. Совместное существование с планом 46:** часть контрактов плана 46 (DOM
  `._tp-sel`) переписывается на модель грида. Убедиться, что не-грид managed-ТЧ
  продолжают работать по-старому (две ветки в `obFire`/`applyTableParts`).

## Риски

1. **Round-trip ↔ модель грида** — главный риск (Фазы 5-6). Несинхронизация
   модели и `tp_json` даст потерю строк при записи. Митигировать юнит-тестами и
   ручной проверкой записи/перечитки.
2. **Ссылочный редактор + серверный picker** — колбэк из модального picker в
   активную ячейку грида нетривиален (Фаза 4). Переиспользовать существующий
   `openRefPicker`-механизм, не плодить второй.
3. **Два движка** — рост поддержки (Р4). Принято осознанно.
4. **Офлайн-embed** — если SlickGrid только модульный, нужен офлайн-бандл; не
   тянуть с CDN (решение проекта).
5. **Регрессия плана 46** — выделение/подбор завязаны на DOM-строки; грид-ветка
   не должна ломать не-грид ТЧ.
