# Текущий приоритет развития onebase

Дата ревизии: 2026-06-25.

## Рекомендуемый порядок

1. **Смерджить готовые инфраструктурные PR.**
   PR #202 закрывает `onebase lint` и структурные `slog`-логи, PR #203 закрывает
   graceful shutdown scheduler. Оба PR зелёные, mergeable, и их объединение
   проходит `go test ./...` локально без конфликтов.

2. **Стабилизационный спринт: lint-clean shipped examples/templates.**
   Новый `onebase lint` уже выявляет предупреждения в поставляемых
   `examples/` и `templates/`: неизвестные YAML-ключи, объекты без RBAC-ролей,
   мёртвые DSL-процедуры. Пока примеры не приведены к нулю предупреждений,
   нельзя включать lint-gate в CI для всего набора: пользователи быстро
   привыкнут игнорировать вывод инструмента.

3. **План 34 F3 — автодоступ к полям ссылок (`this.X.Y`, `Стр.X.Y`).**
   В examples уже встречается `this.Задача.Проект`, но текущий runtime требует
   обходного `ЗначениеРеквизитаОбъекта(Ссылка, "Реквизит")`. Это небольшой, но
   важный разрыв между DSL и ожидаемой 1С-подобной моделью.

4. **План 74 — `onebase fmt` и общий `aicontract`.**
   Делать после очистки lint-предупреждений: formatter и машинный контракт
   должны фиксировать уже согласованную схему YAML/DSL, а не закреплять текущий
   дрейф примеров.

5. **План 60B — marketplace конфигураций.**
   Версионирование конфигурации в БД и UI истории уже реализованы. Marketplace
   целесообразно брать после lint-clean examples/templates, потому что именно
   они станут первыми устанавливаемыми конфигурациями.

6. **План 26 — REST API v2.**
   Делать при наличии конкретного внешнего потребителя: мобильный клиент, Excel
   или интеграция. Без такого потребителя это крупная работа с меньшей
   немедленной отдачей, потому что HTTP-сервисы `/hs/*` уже закрывают часть
   сценариев.
