## Контекст и цель

Чтобы ИИ генерировал конфигурацию и подсказывал запросы, ему нужен **точный срез
текущей конфигурации** — иначе он выдумывает несуществующие справочники/поля.

Сейчас:
- Пользовательский ИИ-чат (`internal/ui/ai_tools.go` `aiSchemaText`) уже строит
  компактный текст-срез из `runtime.Registry`, но **без ТЧ, флага проведения и форм**,
  и логика заперта в пакете `ui` (метод `*Server`).
- **Конфигуратор-ИИ (`internal/launcher/ai_assist.go` `cfgAIAssist`) не знает
  конфигурацию вообще** — шлёт модели только список builtin-функций. Это главный пробел.

**Цель этапа 1:** вынести компактный сборщик контекста в переиспользуемый пакет,
дополнить его (ТЧ, проведение, формы) и **дать срез конфигуратору**.

## Целевая архитектура

Новый пакет **`internal/aicontext`** зависит только от `internal/metadata` (без
`runtime`/`project` — нет риска циклов). Оба потребителя заполняют общий `Input`:

```go
package aicontext

// NamedTitle — имя + синоним для объектов, у которых в срез идут только они
// (отчёты, обработки). Позволяет не тащить пакеты report/processor в aicontext.
type NamedTitle struct{ Name, Title string }

// Input — срезы метаданных для построения текстового контекста. Заполняется из
// runtime.Registry (пользовательский чат) или project.Project (конфигуратор).
type Input struct {
	Entities         []*metadata.Entity
	Registers        []*metadata.Register
	InfoRegisters    []*metadata.InfoRegister
	AccountRegisters []*metadata.AccountRegister
	ChartsOfAccounts []*metadata.ChartOfAccounts
	Enums            []*metadata.Enum
	Constants        []*metadata.Constant
	Reports          []NamedTitle
	Processors       []NamedTitle
	Journals         []*metadata.Journal
	Subsystems       []*metadata.Subsystem
}

// SchemaText возвращает компактный текстовый срез конфигурации для системного
// промпта модели. Пустой Input → «В конфигурации нет объектов...».
func SchemaText(in Input) string
```

`SchemaText` переносит формат текущего `ui.aiSchemaText` и **дополняет** по сущностям:
- **табличные части** — для каждой ТЧ: `имя: поля`;
- **флаг проведения** — пометка `(проводится)` у документов с `Posting`;
- **формы** — список `entity.Forms` именами с видом (`object`/`list`/...).

Формат — **текст** (не JSON): дешевле по токенам, модель читает нормально.

## Потребители

1. **Пользовательский чат** (`ui/ai_tools.go`): `aiSchemaText` становится тонкой
   обёрткой — собирает `aicontext.Input` из `s.reg` (с адаптерами `[]NamedTitle`
   для `Reports()`/`Processors()`) и зовёт `aicontext.SchemaText`. Попутно чат
   получает ТЧ/проведение/формы. Существующий формат — не ломается (golden/substr).

2. **Конфигуратор** (`launcher/ai_assist.go` `cfgAIAssist`): через
   `materializeProject(ctx, h, b)` + `project.Load(dir)` (как `configuratorCheckAll`,
   `check_handlers.go:60,78`) собирает `aicontext.Input` из `proj` и **добавляет срез
   в системный промпт** перед запросом (builtin-список там уже есть). Срез строится
   best-effort: при ошибке загрузки конфигурация просто не добавляется (помощник
   продолжает работать как раньше, по builtin-списку).

## Чего НЕ делаем (YAGNI)

- Не тащим тела модулей / полный YAML в контекст (раздувает токены).
- «Проводки» = только флаг `Posting`; детальные движения живут в `.posting.os`
  и появятся в контексте генерации (этапы 2–3), не здесь.
- Не кэшируем загрузку проекта в конфигураторе — `cfgAIAssist` интерактивен и
  вызывается редко (как `check`); преждевременная оптимизация не вносится.

## Тесты

- `internal/aicontext/aicontext_test.go`: на собранном `Input` (документ с ТЧ и
  `Posting`, справочник с формой, регистр, перечисление) `SchemaText` содержит имена
  объектов, поля, **строку ТЧ**, пометку **проведения**, **имя формы**; пустой Input →
  сообщение-заглушка.
- `internal/ui/ai_tools_test.go`: существующие проверки `aiSchemaText` зелёные; добавить
  проверку, что в срезе появилась ТЧ (делегирование работает).
- Конфигуратор: вынести сборку среза в тестируемый хелпер (напр.
  `configSchemaText(ctx, b) (string, error)`); юнит-тест самого хендлера с реальным LLM
  не делаем — проверка ручная (см. Verification). `go build` + `onebase check`.

## Verification

1. `go test ./internal/aicontext/ ./internal/ui/ -count=1` — зелёные.
2. Запустить базу с конфигурацией (например `examples/trade`), открыть ИИ-чат →
   «опиши данные» — в ответе видны ТЧ и формы.
3. В конфигураторе открыть ИИ-помощника, спросить «какие документы есть и их ТЧ» —
   ответ опирается на реальную конфигурацию (раньше модель не знала объектов).
