# Текущий приоритет развития onebase

Дата фиксации: 2026-06-25.

## Рекомендуемый порядок

`onebase lint` из плана 56 уже в работе, поэтому не считаем его свободным
следующим пунктом. После него приоритет смещается с новых фич на готовность к
росту пользователей.

1. **План 76 — готовность к многопользовательской нагрузке.**
   Первый срез: RBAC в текущем REST API, лимиты/пагинация REST list, атомарная
   optimistic locking запись, индексы под табличные части и большие списки,
   server-side reference picker. Это самый практичный шаг, если база пойдёт к
   100+ активным пользователям.

2. **План 56 / 43.3 — точечный `slog`-контур.**
   Не большая механическая миграция, а единый logger, request_id и редактирование
   секретов в production-коде. `/metrics` уже есть, scheduler уже на `slog`.
   Делать рядом с планом 76, чтобы оптимизировать по фактическим p95/p99, а не
   на глаз.

3. **План 26 — REST API v2.**
   Делать после guardrails из плана 76: v2 должен сразу наследовать RBAC,
   пагинацию, токены, OpenAPI и лимиты. Без этого новый API закрепит старые
   проблемы текущего REST.

4. **План 60B — marketplace конфигураций.**
   Версионирование, UI истории, diff, rollback, ZIP/OBZ export уже реализованы.
   Следующий экосистемный шаг — реестр конфигураций, checksum, `check` в
   песочнице и установка из лаунчера/CLI.

5. **DSL follow-up — автодоступ к полям ссылок.**
   Общие модули, `ИначеЕсли`, дата/строка builtins, `ТаблицаЗначений`,
   `Документы.X.Создать()` и write-hooks уже есть. Остаток с высокой
   прикладной отдачей — `Контрагент.ИНН` / `Стр.Номенклатура.Единица`
   без ручного `ЗначениеРеквизитаОбъекта(...)`.
