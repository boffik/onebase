# План 81 - Безопасная реструктуризация схемы

Дата проектирования: 2026-07-08.
Статус: ⬜ Проект (по итогам ревью; аналог «реструктуризации» 1С).

## Контекст

Миграция схемы сейчас **аддитивна** (`internal/storage/migrate.go`,
`ddl.go`):

- `AddColumnIfMissing` добавляет новые колонки — существующие не трогает;
- смена типа поля в метаданных **игнорируется** на уровне БД (колонка уже есть →
  `AddColumnIfMissing` ничего не делает, старый тип остаётся);
- удалённое из конфигурации поле оставляет **осиротевшую** колонку (данные не
  чистятся, но и не удаляются);
- переименование поля не отслеживается: у полей нет устойчивого идентификатора,
  поэтому rename выглядит как «удалить старое + добавить новое» → новая колонка
  **пустая**, данные остаются в осиротевшей старой (для пользователя — потеря);
- `renameSnakeCols` покрывает лишь один частный случай смены конвенции имён и
  только на PostgreSQL (SQLite выходит сразу).

Модель безопасна в смысле «никогда не удалит данные молча», но неполна: типовой
для бизнес-платформы сценарий «переименовал реквизит / поменял тип / удалил
лишнее поле в конфигураторе» либо теряет связь с данными, либо тихо не
применяется. 1С решает это «реструктуризацией»: устойчивые ID объектов + план
преобразования + явное предупреждение о необратимых шагах.

## Цель

- устойчивая идентичность полей/объектов, чтобы rename отличался от drop+add и
  **сохранял данные**;
- явный план миграции (add / rename / retype / drop) с dry-run и подтверждением
  необратимых шагов;
- безопасная смена типа с проверяемой конверсией (или отказ до потери данных);
- одинаковая семантика на SQLite (пересоздание таблицы) и PostgreSQL (ALTER).

## Синтаксис / метаданные

Ввести необязательный устойчивый `id` у поля (и объекта). Отсутствие `id` →
текущее аддитивное поведение (обратная совместимость):

```yaml
fields:
  - { id: f_partner, name: Контрагент, type: ref, ref: Контрагенты }
  - { id: f_sum,     name: Сумма,       type: decimal }
```

Конфигуратор проставляет `id` автоматически при создании поля и сохраняет его
при переименовании — так `name` меняется, а `id` держит связь с колонкой.

## Хранилище / SQL

Планировщик миграции сравнивает «желаемую» схему (метаданные) с фактической
(`information_schema` / PRAGMA) по `id`:

- **add** — новое `id` → `ADD COLUMN` (как сейчас);
- **rename** — `id` есть, изменилось имя колонки → `RENAME COLUMN` (PostgreSQL)
  или пересоздание таблицы с `INSERT … SELECT` (SQLite);
- **retype** — `id` есть, изменился тип → `ALTER … TYPE … USING <cast>` с
  предварительной проверкой конвертируемости; неконвертируемо → отказ с отчётом;
- **drop** — колонка есть в БД, `id` нет в метаданных → только по явному
  подтверждению (`--allow-destructive`), иначе оставить осиротевшей и
  предупредить.

SQLite: путь rename/retype/drop через каноничный `CREATE new → INSERT SELECT →
DROP old → RENAME` в транзакции (у SQLite ограниченный ALTER).

## Изменения в коде

- `internal/metadata/` — поле `ID` у `Field`/`Entity`; заполнение по умолчанию;
- `internal/storage/migrate.go` — планировщик `PlanSchemaMigration(desired,
  actual) []Change` + исполнитель по диалекту; заменяет точечный
  `renameSnakeCols`;
- `internal/storage/ddl_sqlite.go` — table-rebuild helper;
- `internal/cli/migrate.go` — флаги `--dry-run` (печать плана), `--allow-destructive`;
- конфигуратор (`internal/launcher/`) — авто-`id`, предупреждение о
  необратимых шагах перед применением;
- `onebase check` — lint `schema.*`: несохранённый `id`, потенциально
  разрушительное изменение.

## Тесты

- rename поля с данными → данные на месте, колонка переименована (оба диалекта);
- retype `text`→`decimal` с валидными/битыми значениями (успех/отказ);
- drop без флага — колонка осиротела + предупреждение; с флагом — удалена;
- dry-run печатает план и не меняет БД;
- обратная совместимость: поля без `id` мигрируют аддитивно, как раньше.

## Verification

1. База с данными; в конфигураторе переименовать реквизит справочника → данные
   видны под новым именем.
2. Сменить тип поля на несовместимый → `migrate` отказывает до применения,
   объясняет причину.
3. `onebase migrate --dry-run` печатает add/rename/retype/drop без изменений.

## Эстимейт

7–10 дней: идентичность полей + конфигуратор (2), планировщик+PostgreSQL (2–3),
SQLite table-rebuild (2), dry-run/подтверждение/lint (1), тесты (2). Основной
риск — SQLite-путь и конверсии типов; закрывается тестами на реальных данных.
