# План 46: Команды табличной части + подбор (мультивыбор)

## Цель

Дать платформе OneBase **обобщённый механизм команд табличной части** (кнопки-команды
над ТЧ с DSL-обработчиками), частным случаем которого является **подбор** —
модальный список с мультивыбором, количеством и поиском (аналог кнопки «Подбор»
в расходных документах 1С).

Ключевой принцип: платформа даёт **примитивы**, а конкретные команды (подбор по
остаткам, заполнить цены, очистить, изменить выделенные и т.п.) пишутся **на уровне
конфигурации** в DSL. «Подбор» не хардкодится в платформе — он собирается из
платформенных кирпичей в `.form.os`.

## Что уже есть (не переписываем)

Managed-формы уже имеют рабочий round-trip — на нём всё и строится:

- `obFire(elementName, eventName)` (`internal/ui/templates_managed.go:495`) —
  сериализует всю форму (поля + строки ТЧ) и POST'ит на `/ui/{kind}/{entity}/form-event`.
- `handleManagedFormEvent` (`internal/ui/handlers_managed_events.go:48`):
  - `buildObjectFromForm(r, entity)` — собирает `runtime.Object` из текущего состояния формы (стр. 125);
  - инъектирует `Объект`/`ЭтотОбъект` как `formObjectThis`, и `Объект.Товары.Добавить()`
    **реально мутирует объект** через `formTpProxy` (стр. 143-145);
  - `Сообщить` инъектируется через `buildDSLVarsWithMessages` (стр. 142) —
    **готовый side-channel-паттерн** для будущего `ПоказатьПодбор`;
  - ответ возвращает `Values` + `TableParts`, клиент перерисовывает через
    `applyTableParts` (`templates_managed.go:393,520`).
- Кнопка `Кнопка` с обработчиком `Нажатие` уже рендерится с `onclick="obFire(name,'Нажатие')"`
  (`templates_managed.go:116`).

**Следствие:** простые команды ТЧ (Заполнить/Очистить/Пересчитать) технически
работают уже сегодня — кнопкой с обработчиком `Нажатие`. Не хватает только:
1. командной панели **над конкретной таблицей** ТЧ (UX 1С);
2. контекста **выделенных строк** ТЧ в обработчике;
3. для команд-**диалогов** (подбор) — возврата данных диалога и второй фазы.

## Ограничение области

Механизм работает **только для managed-форм** (`obFire`/`form-event` живут в
`templates_managed.go`; autogen-формы из `templates.go` его не имеют). Документ,
которому нужны команды ТЧ/подбор, должен иметь управляемую форму. Это фиксируем
явно; распространение на autogen — вне области плана.

## Архитектура: 3 слоя

### Слой A — Команды табличной части (generic, фундамент)

Команда ТЧ = кнопка в командной панели над таблицей, с обработчиком `Нажатие`,
который в DSL мутирует `Объект.<ТЧ>` и/или возвращает сообщения. Покрывает **любые**
команды без диалога: Заполнить, Очистить, Пересчитать итоги, Изменить выделенные,
Загрузить из файла и т.д.

**A1. Командная панель ТЧ.**
`ТабличнаяЧасть` уже контейнер (`metadata.form_module.go:417`, `IsContainer`).
Рендерить дочерние элементы `Кнопка` элемента ТЧ как тулбар **над** таблицей
(перед `<table>` в `templates_managed.go:134`). Кнопка вызывает
`obFire(name,'Нажатие', {_tp:'<ТЧ>'})`. Если у ТЧ нет своих кнопок — поведение
как сейчас (только «+ Добавить строку»).

**A2. Контекст выделенных строк.**
- В рендере строки ТЧ (`templates_managed.go:144`) добавить чекбокс выделения
  (первая колонка) + чекбокс «выделить все» в `<thead>`.
- `obFire` (расширенный, см. C2) при наличии `_tp` собирает индексы отмеченных
  строк и кладёт в параметр `_tp_selected` (CSV индексов).
- На сервере (`handleManagedFormEvent`) распарсить `_tp_selected` и прокинуть в DSL
  как `ВыделенныеСтроки` — массив строк ТЧ (или массив индексов; решается в
  «Открытых вопросах»). Обработчик типа «скидка на выделенные» итерирует их.

**A3. Декларация в YAML формы** (`forms/<entity>/формаобъекта.form.yaml`):
```yaml
- kind: ТабличнаяЧасть
  data_path: Объект.Товары
  children:
    - kind: Кнопка
      name: КнопкаПодбор
      title: { ru: "Подбор" }
      events: { Нажатие: КнопкаПодборНажатие }
    - kind: Кнопка
      name: КнопкаЗаполнитьЦены
      title: { ru: "Заполнить цены" }
      events: { Нажатие: ЗаполнитьЦеныНажатие }
```

### Слой B — Диалоги формы (generic, для подбора и не только)

Двухфазный механизм «обработчик показал диалог → JS открыл → результат вернулся
во вторую фазу». Подбор — частный вид диалога; та же схема годится для
`ПоказатьВопрос`, `ВвестиЗначение` и пр.

**B1. Side-channel-билтин `ПоказатьПодбор(Данные, Колонки, Конфиг)`** —
по образцу `Сообщить`:
- регистрируется через `vars["__picker_sink__"] = &pickerPayload` в
  `handleManagedFormEvent` перед `interp.Run`;
- `Данные` — массив структур (каждая = строка диалога, содержит UUID-идентификатор
  и значения колонок);
- `Колонки` — **массив структур** `{Имя, Заголовок, Тип, Редактируемая, Поумолчанию}`
  (НЕ строка-мини-DSL — структура надёжнее и совпадает с `PickerColumn`);
- `Конфиг` — структура `{Заголовок, ПолеПоиска, ПолеКоличества, ВыбратьВсе}`;
- билтин пишет в sink → после `interp.Run` сервер кладёт `pickerData` в ответ.

**Семантика — терминальная.** Вызов `ПоказатьПодбор` означает «прервать и показать
диалог»: код DSL после него ещё выполнится, но на UI не повлияет (аналог
асинхронного `ОткрытьФорму(..., Оповещение)` в 1С). Это документируется, иначе
конфигуратор напишет логику после `ПоказатьПодбор` и удивится. Рекомендуем делать
`ПоказатьПодбор` последней операцией фазы 1.

**B2. Фаза 2 — возврат результата.**
- JS-диалог по «Перенести в документ» собирает отмеченные строки и вызывает
  `obFire(element, 'Выбор', {_pick_result: JSON.stringify(rows)})`.
- Сервер: если `_pick_result` не пуст → `json.Unmarshal` → DSL-переменная
  `ПодборРезультат` (массив структур с UUID + значениями колонок).
- Обработчик события `Выбор` итерирует `ПодборРезультат`, добавляет строки в ТЧ.

**B3. Новое событие формы** `FormEventOnChoice = "Выбор"`
(`internal/metadata/form_module.go`, рядом с `ОбработкаВыбора`).

### Слой C — Подбор как частный случай (конфигурация + точечный UI)

**C1. JS-диалог `openItemPicker(payload, elementName)`** в `form-shared-js`
(`templates.go:1218`, рядом с `openRefPicker`):
- модальный overlay (`position:fixed; z-index:9999`);
- поиск (фильтр по `Конфиг.ПолеПоиска`);
- таблица: чекбокс | колонки из `payload.columns` | для `Редактируемая`-числовых
  (количество) — `<input type="number">`;
- «выделить все»/«снять все» в шапке;
- счётчик «Выбрано: N»;
- «Перенести в документ» → `obFire(elementName,'Выбор',{_pick_result:...})`;
- «Отмена» → `modal.remove()`.

**C2. Расширение `obFire(elementName, eventName, extraParams)`**
(`templates_managed.go:495`):
- `extraParams` добавляются в тело запроса перед POST;
- в callback **перед** `applyTableParts`:
  ```js
  if (data.pickerData) { openItemPicker(data.pickerData, elementName); return; }
  ```
  (не трогаем ТЧ — это сделает фаза 2);
- при наличии `_tp` в `extraParams` — добавить `_tp_selected` (см. A2).

**C3. Расширение `formEventResponse`** (`handlers_managed_events.go:40`):
```go
PickerData *PickerPayload `json:"pickerData,omitempty"`
```
со структурами `PickerPayload{Columns []PickerColumn; Rows []PickerRow; Config PickerConfig}`.

**C4. Конфигурация-пример** (см. «Референс-пример»): обработчики
`КнопкаПодборНажатие` (фаза 1: собрать остатки → `ПоказатьПодбор`) и
`КнопкаПодборВыбор` (фаза 2: `ПодборРезультат` → `Объект.Товары.Добавить()`).
Issue #27 («подбор по остаткам») закрывается здесь: фаза 1 зовёт DSL-функцию
остатков (`GetBalances` через `internal/ui/dsl_registers.go`).

## Открытые вопросы (решить до кодинга)

1. **`ВыделенныеСтроки` — что передавать в DSL:** массив строк ТЧ (удобнее для
   конфигуратора: `Стр.Количество`) или массив индексов (проще на сервере)?
   Предложение: **массив строк** (`MapThis`), как в `ПодборРезультат`.
2. **Привязка кнопки к ТЧ:** через `children` элемента ТЧ (как в A3) или через
   поле `table_part` у отдельной кнопки? Предложение: **children** (соответствует
   модели контейнера 1С и `FormElement.Children`).
3. **Формат идентификатора в `Данные`/`ПодборРезультат`:** UUID-строкой
   (`НайтиПоИдентификатору`) — да; имена полей колонок — оригинальный регистр.

## Порядок реализации

1. **Слой B+C (подбор) сначала** — даёт измеримый результат и закрывает Issue #27:
   B3 → C3 → B1 → C1 → C2 → B2 → пример в trade → тесты → i18n.
2. **Слой A потом** — обобщение на произвольные команды ТЧ: A1 (тулбар) → A2
   (выделение строк). Подбор после этого переезжает в командную панель ТЧ как
   первая «настоящая» команда.

Такой порядок: сначала рабочий подбор (узкий путь по существующему механизму),
затем обобщение инфраструктуры, чтобы подбор стал одной из многих команд ТЧ.

## Проверка

1. `cd onebase && go build ./...` — компиляция платформы.
2. `go test ./internal/ui/...` — юнит-тесты механизма.
3. `onebase check --project <trade>` — валидация конфигурации.
4. Ручной тест в браузере (новый документ trade):
   - указать склад → «Подбор» → модалка с номенклатурой и остатками;
   - отметить 3 позиции, изменить количество → «Перенести» → строки в ТЧ с
     номенклатурой/количеством/ценой;
   - «Отмена» → модалка закрывается без изменений;
   - команда без диалога («Заполнить цены») → ТЧ пересчитана;
   - выделить строки → команда «на выделенные» → применилась только к ним.
