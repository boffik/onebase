# План 84 - Enterprise-аутентификация (2FA / SSO / LDAP)

Дата проектирования: 2026-07-08.
Статус: ⬜ Проект (опционально, для многопользовательских/корпоративных установок).

## Контекст

Аутентификация сейчас: логин+пароль, серверная сессия (токен вне URL/логов,
план 53), rate-limit логина, CSRF и security-заголовки; для REST — Bearer
API-токены (план 26). Этого достаточно для фрилансера/малого бизнеса, но по мере
движения платформы к серьёзному многопользовательскому применению (RLS план 79,
масштаб план 76, изолированные сессии план 78) не хватает корпоративного
минимума:

- **второй фактор** для привилегированных учёток (админ конфигуратора,
  бухгалтерия);
- **единый вход (SSO)**, чтобы не заводить отдельные пароли, а пускать через
  корпоративный провайдер;
- опционально каталог пользователей (LDAP/AD).

Всё — строго **opt-in**: базы без включённой фичи работают как сейчас.

## Цель

- TOTP-2FA (RFC 6238) для учёток, где включён; резервные коды;
- вход через внешнего OIDC-провайдера (Google/Microsoft/Keycloak и совместимые),
  маппинг на локальную учётку и роли;
- (опц.) LDAP/AD bind как источник учёток;
- админ-политики: «требовать 2FA для ролей X», «разрешить только SSO».

## Хранилище / метаданные

- `_users`: `totp_secret` (enc:, план 83), `totp_enabled`, `backup_codes` (хэши);
- `_auth_providers` в `_settings`: тип (`oidc`/`ldap`), issuer/client_id/секрет
  (через резолвер секретов), правила маппинга `claim → роль`;
- политика в `app.yaml`/админке: `auth: { require_2fa_roles: [...], sso_only: bool }`.

## Дизайн

**2FA (TOTP).** Профиль пользователя: «Включить 2FA» → QR (otpauth://) →
подтверждение кодом. При входе после пароля — шаг ввода кода; резервные коды
одноразовые. Секрет TOTP шифруется (план 83).

**SSO (OIDC).** Стандартный Authorization Code + PKCE. Кнопка «Войти через …» →
редирект к issuer → callback: валидация id_token, поиск/создание локальной
учётки по `email`/`sub`, назначение ролей по правилам маппинга. Сессия — та же
серверная, что сейчас.

**LDAP (опц.).** Проверка пароля bind'ом к каталогу; локальная учётка —
проекция. Ниже приоритетом, чем OIDC (многие корпоративные каталоги уже за OIDC).

## Изменения в коде

- `internal/auth/` — TOTP (генерация/проверка/резервные коды), провайдеры
  (`oidc.go`, `ldap.go`), маппинг ролей, политики;
- `internal/ui/admin.go` — управление провайдерами и политиками;
- `internal/ui/` — шаг 2FA при входе, кнопки SSO, профиль (вкл. 2FA);
- `internal/api/` — учесть, что API-токены (план 26) остаются альтернативой для
  сервис-доступа независимо от 2FA/SSO;
- зависимость OIDC — минимальная (стандартная библиотека OAuth2/OIDC), внести в
  `go.mod` под vuln-скан (govulncheck-job).

## Тесты

- TOTP: валидный/просроченный/переигранный код, резервные коды одноразовы;
- политика `require_2fa`: учётка без 2FA в защищённой роли не входит до настройки;
- OIDC (мок issuer): callback создаёт/находит учётку, применяет маппинг ролей;
  битый/просроченный id_token → отказ;
- `sso_only`: локальный пароль отклоняется.

## Verification

1. Включить 2FA у админа → повторный вход требует код; резервным кодом тоже.
2. Настроить мок-OIDC → «Войти через …» заводит учётку и роли по claim.
3. База без включённых провайдеров/2FA — вход как раньше.

## Эстимейт

7–10 дней: TOTP+резервные коды+UI (2–3), OIDC (3–4), политики/маппинг/админка
(1–2), тесты (2). LDAP — отдельные +2–3 дня, при спросе. Держать всё opt-in и не
регрессировать текущий вход — главный критерий приёмки.
