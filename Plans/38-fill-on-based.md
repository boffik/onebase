# Этап 38 — Ввод на основании (ОбработкаЗаполнения)

## Контекст

В 1С механизм «Ввод на основании» — стандартный способ создавать новый документ/справочник с автозаполнением реквизитов по данным существующего объекта. Типовые сценарии: «Реализация → Возврат от покупателя», «Заказ покупателя → Счёт на оплату», «Поступление → Возврат поставщику».

Механизм состоит из трёх частей:

1. **Метаданные конфигурации**: у документа-приёмника указан список типов источников («Вводится на основании»).
2. **Модуль объекта-приёмника**: процедура `ОбработкаЗаполнения(ДанныеЗаполнения)`, которая копирует поля из источника в `this`.
3. **UX**: команда «Ввести на основании ▾» в форме/списке источника. Также возможен программный вызов: `Документы.X.СоздатьДокумент().Заполнить(Источник)`.

В `Plans/35-1c-feature-inventory.md` пункт `ОбработкаЗаполнения (ввод на основании)` помечен ❌ — этот план закрывает пробел.

---

## YAML

В YAML документа/справочника-приёмника новый ключ `based_on`:

```yaml
# documents/возврат_от_покупателя.yaml
name: ВозвратОтПокупателя
posting: true
based_on:
  - РеализацияТоваров        # документ
  - Контрагент               # можно вводить и на основании справочника
fields:
  - name: Номер
    type: string
  - name: Дата
    type: date
  - name: Покупатель
    type: reference:Контрагент
  - name: Склад
    type: reference:Склад
  - name: Сумма
    type: number
tableparts:
  - name: Товары
    fields:
      - name: Номенклатура
        type: reference:Номенклатура
      - name: Количество
        type: number
      - name: Цена
        type: number
      - name: Сумма
        type: number
```

Валидация: каждое имя в `based_on` должно существовать в реестре (документ или справочник). Иначе — ошибка при загрузке проекта.

## DSL

Хук — в `src/<имя приёмника>.os` (там же, где `ПриЗаписи`):

```
Процедура ОбработкаЗаполнения(ДанныеЗаполнения)
  // ДанныеЗаполнения — Object источника (как this, но read-only):
  // .Поля шапки + .ТабличныеЧасти

  Если ТипЗнч(ДанныеЗаполнения) = "РеализацияТоваров" Тогда
    this.Покупатель = ДанныеЗаполнения.Покупатель;
    this.Склад      = ДанныеЗаполнения.Склад;
    Для Каждого Стр Из ДанныеЗаполнения.Товары Цикл
      Нов = this.Товары.Добавить();
      Нов.Номенклатура = Стр.Номенклатура;
      Нов.Количество   = Стр.Количество;
      Нов.Цена         = Стр.Цена;
      Нов.Сумма        = Стр.Сумма;
    КонецЦикла;
  КонецЕсли;

  Если ТипЗнч(ДанныеЗаполнения) = "Контрагент" Тогда
    this.Покупатель = ДанныеЗаполнения;
  КонецЕсли;
КонецПроцедуры
```

Алиас в `runtime.eventAliases`:

```go
var eventAliases = map[string]string{
    "onwrite": "призаписи",
    "onpost":  "обработкапроведения",
    "onfill":  "обработказаполнения",   // NEW
}
```

### Программный API в DSL

Симметрично существующим `Записать()`/`Провести()`:

```
Док = Документы.ВозвратОтПокупателя.СоздатьДокумент();
Док.Заполнить(Основание);
Док.Записать();
```

Метод `.Заполнить(Источник)` зовёт ту же функцию `entityservice.Fill`, что и UI-handler — единая точка вызова `OnFill`.

### Сигнатура: упрощения относительно 1С

В 1С:
```
Процедура ОбработкаЗаполнения(ДанныеЗаполнения, ТекстЗаполнения = "", СтандартнаяОбработка = Истина) Экспорт
```

В onebase MVP — только `ДанныеЗаполнения`:
- `СтандартнаяОбработка` не нужна: автозаполнения по совпадению имён в onebase нет, копирование явное в коде хука.
- `ТекстЗаполнения` (ввод по строке поиска) — редкий кейс, откладывается.

## Хранилище / SQL

Изменений в БД не требуется. Источник загружается через существующий `storage.GetByID` + `storage.GetTablePartRows`. Поле «ДокументОснование» на приёмнике — обычное `reference:` поле в YAML, объявляется пользователем при необходимости (отдельной автоматики не делаем).

## Изменения в коде

### `internal/metadata/types.go`
- В `Entity` добавить `BasedOn []string`.

### `internal/metadata/yaml.go`
- В `rawEntity` добавить `BasedOn []string \`yaml:"based_on"\``.
- В `LoadFile` копировать в `e.BasedOn`.

### `internal/metadata/validate.go`
- Валидация: для каждого имени в `BasedOn` искать в `entities` (catalog или document). Если нет — ошибка.

### `internal/runtime/registry.go`
- Поле `basedOnIndex map[string][]string` (lowercase sourceName → []receiverName).
- В `Load(opts)`: после построения `entities` пройти и заполнить индекс.
- Метод `ReceiversOf(sourceName string) []*metadata.Entity` — для UI.
- В `eventAliases` добавить `"onfill": "обработказаполнения"`.

### `internal/dsl/interpreter/interpreter.go`
- `Run(proc, obj, vars)` уже принимает обработчик без параметров. Добавить вариант с одним аргументом: связать первый `proc.Params[0]` со значением из `vars["ДанныеЗаполнения"]` (или передавать как явный `arg any`).
- Альтернатива: новая функция `RunWithArg(proc, obj, arg, vars)`. Решить по ходу — выбираем минимальную инвазивность.

### `internal/entityservice/service.go`
- Новая функция `Fill(ctx, receiver *Entity, srcType, srcID) (FillResult, error)`:
  1. Загрузить источник: `Store.GetByID` + `Store.GetTablePartRows` для всех ТЧ.
  2. Опционально применить `PrepareHook` / `EnrichTPRows` для src (чтобы Ref-поля распаковались как в обычном хуке).
  3. Создать пустой `recvObj *Object` (поля + пустые TablePartRows).
  4. Найти `OnFill` через `Reg.GetProcedure(receiver.Name, "OnFill")`. Если нет — вернуть пустой результат без ошибки.
  5. Запустить `Interp.RunWithArg(proc, recvObj, srcObj, vars)`.
  6. Вернуть `recvObj.Fields` и `recvObj.TablePartRows`.

### `internal/ui/handlers.go`
- В `form()`: если в query есть `based_on` + `based_on_id`:
  - Проверить, что приёмник разрешает такой источник (`entity.BasedOn` содержит src).
  - Проверить право `read` на источник.
  - Позвать `entityService.Fill(...)`.
  - Сконвертировать `Fields` → `values map[string]string` (как делает `formEdit`).
  - Положить `TablePartRows` в шаблон.

### `internal/ui/templates.go`
- В toolbar формы документа-источника (после кнопки «История»): если `len(.Receivers) > 0` — dropdown «Ввести на основании ▾» со ссылками `/ui/{kind}/{recv}/new?based_on={src}&based_on_id={id}`.
- В `s.list` для каждой строки — кнопка «Ввести на основании ▾» в колонке действий (опционально для MVP — можно отложить, оставить только из формы).

### `internal/ui/server.go`
- В `form()` собрать `Receivers` через `s.reg.ReceiversOf(entity.Name)` и положить в шаблон формы (для случая «открыли существующий документ → ввести на основании»).

### DSL builtin `.Заполнить()`
- В месте, где сейчас зарегистрированы методы Object/Document (`internal/ui/dsl_documents.go` / `internal/dsl/interpreter/builtins*.go` — уточнить по факту), добавить метод `Заполнить(arg)`:
  - Проверка: arg — это Object/Ref.
  - Если arg — Ref → загрузить объект.
  - Вызвать `entityService.Fill(...)`.

## Открытые вопросы

1. **Множественный ввод** (несколько строк в списке → один документ-приёмник). В 1С `ДанныеЗаполнения` может быть массив ссылок. В MVP — только одиночный источник. Расширение — отдельным планом.
2. **«ПриКопировании»**. Близкий механизм (создание копии без основания). Не входит в этот план, можно сделать отдельным мини-этапом.
3. **`СтандартнаяОбработка = Ложь`** — отказ от автозаполнения. Не нужен, т.к. у нас нет неявного автозаполнения.
