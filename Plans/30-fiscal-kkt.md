# Этап 30 — Фискальный регистратор (ФЗ-54)

## Контекст

Подсистема торгового оборудования уже реализована (коммит «feat: подсистема торгового оборудования»). Сейчас чеки **нефискальные** — драйвер `escpos` просто печатает текст на термопринтер. Для легальной розницы в РФ нужен фискальный учёт по 54-ФЗ: фискальный накопитель (ФН), передача в ОФД, теги ФФД 1.2.

Где что лежит (для ориентации):
- `internal/equipment/` — драйверы и интерфейсы (`Device`, `ReceiptPrinter`, `Scale`, `PaymentTerminal`, `EventSource`), реестр `Register/Open/Drivers`, транспорт.
- `internal/equipment/device.go` — доменные модели (`Receipt`, `ReceiptItem`, `PaymentResult`).
- `internal/dsl/interpreter/equipment_builtins.go` — DSL-объект `ПодключитьОборудование`.
- `internal/deviceagent/agent.go` — HTTP-агент кассы.
- `examples/trade/` — справочник `ПодключаемоеОборудование`, РМК-обработки.

## Ключевое решение: НЕ фискализировать самим

Самостоятельная реализация ФН/ОФД/ФФД — это **сертификация и месяцы работы**, не функция приложения. Правильный путь — опереться на **драйвер ККТ производителя** (АТОЛ, Штрих-М), который сам общается с ФН и ОФД. onebase лишь формирует структуру чека и отдаёт её драйверу.

Рекомендуемая цель интеграции — **АТОЛ «Драйвер ККТ v10»**: принимает задания в JSON (регистрация чека с позициями, налогами, оплатами) по HTTP/именованному каналу. Это идеально ложится на наш `device-agent` (он и так на машине кассира).

## Часть А — интерфейс и модель

`internal/equipment/device.go`:

```go
// FiscalRegistrar — фискальный регистратор (ККТ): пробивает фискальный чек.
type FiscalRegistrar interface {
	Device
	RegisterReceipt(r FiscalReceipt) (FiscalResult, error)
}

type FiscalReceipt struct {
	Type      string          // "приход" | "возвратПрихода" | ...
	Items     []FiscalItem
	Payments  []FiscalPayment // наличные/безнал/аванс
	Taxation  string          // СНО: "osn" | "usnIncome" | ...
	Email     string          // для электронного чека (необязательно)
	Phone     string
}

type FiscalItem struct {
	Name        string
	Qty         float64
	Price       float64
	Sum         float64
	VAT         string // "vat20" | "vat10" | "none" | ...
	PaymentType string // признак способа расчёта (ФФД тег 1214)
	ItemType    string // признак предмета расчёта (ФФД тег 1212)
}

type FiscalPayment struct {
	Type string  // "cash" | "electronically"
	Sum  float64
}

type FiscalResult struct {
	FN  string // номер ФН
	FD  string // фискальный документ (тег 1040)
	FP  string // фискальный признак (тег 1077)
	QR  string // данные для QR кода чека
}
```

## Часть Б — драйвер `atol_kkt`

`internal/equipment/atol.go` — реализует `FiscalRegistrar`. Транспорт — HTTP к локальному сервису АТОЛ (по умолчанию `http://127.0.0.1:16732`). `RegisterReceipt` собирает JSON-задание формата АТОЛ v10 и POST-ит его, опрашивает статус, возвращает `FiscalResult` из ответа.

Маппинг `FiscalReceipt` → JSON АТОЛ: `{"type":"sell","items":[...],"payments":[...],"taxationType":...}`.

Регистрация: `Register("atol_kkt", ...)`. Параметр «Порт» = адрес сервиса АТОЛ.

## Часть В — метаданные (теги ФФД)

Расширить чек реквизитами, которых нет в нефискальной модели:
- Справочник `Номенклатура`: поля `СтавкаНДС` (enum), `ПризнакПредмета` (enum).
- Константа/настройка: `СистемаНалогообложения` (enum СНО).
- enum: `СтавкаНДС` (НДС20/НДС10/БезНДС/...), `СистемаНалогообложения`.

РМК-обработка `ПечатьЧека` дополняется сбором налогов/оплат и вызовом фискального драйвера вместо (или вместе с) `escpos`.

## Часть Г — DSL и агент

- DSL: метод `dslDevice.CallMethod` → `зарегистрироватьчек`/`registerreceipt`, маппинг DSL-структуры в `FiscalReceipt`, возврат `MapThis{фн, фд, фп, qr}`. Helper `asFiscal`.
- Агент: эндпоинт `POST /fiscal` (DTO чека → `RegisterReceipt` → JSON результата).

## Юридическое (вне кода — для пользователя)

- Регистрация ККТ в ФНС, фискализация ФН.
- Договор с ОФД.
- Корректные реквизиты организации (ИНН, СНО) в настройках.

## Тесты (без железа)

- Эмулятор HTTP-сервиса АТОЛ (httptest): принимает JSON-задание, возвращает фискальный результат. Проверить маппинг `FiscalReceipt`→JSON и разбор результата.
- DSL-тест: `ПодключитьОборудование("atol_kkt", …).ЗарегистрироватьЧек(чек)` → `Результат.ФД`.
- Агент-тест: `POST /fiscal` → `{fn, fd, fp}`.

## Этапность

1. Интерфейс + модель + драйвер `atol_kkt` + эмулятор-тест (1–2 дня).
2. DSL + агент `/fiscal` (полдня).
3. Метаданные (налоги/СНО/признаки) + доработка РМК (1 день).
4. Реальная интеграция с физической ККТ АТОЛ + регистрация в ФНС/ОФД (вне автотестов).

## Риски

- Формат JSON АТОЛ v10 нужно сверить с актуальной документацией драйвера.
- ФФД 1.2 добавляет обязательные теги (маркировка) — для товаров с маркировкой отдельная ветка.
- Штрих-М имеет иной протокол — второй драйвер `shtrih_kkt` по тому же интерфейсу.
