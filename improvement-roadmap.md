## 🔴 Tier 1 — критичные пробелы (без них пет-проект упрётся в стену)

Это вещи, которые прямо ограничивают аудиторию из обзорной статьи (фрилансеры, мелкий бизнес, пет-проекты). Без них «вечерний таск-трекер» собрать можно, а вот «мини-CRM с email-уведомлениями» — уже нет.

### 1.1. JOIN в языке запросов
**Где:** `internal/query/query.go` — реализованы `UNION`/`ОБЪЕДИНИТЬ`, но ни один JOIN. **Блокер для:** любых нетривиальных отчётов (продажи по контрагентам с группировкой по менеджерам = справочник + регистр + справочник).

```
ВЫБРАТЬ Прод.Клиент, Клиент.Менеджер, Прод.Сумма
ИЗ Документ.Реализация КАК Прод
    ЛЕВОЕ СОЕДИНЕНИЕ Справочник.Клиент КАК Клиент ПО Прод.Клиент = Клиент.Ссылка
```

**План:** `ВНУТРЕННЕЕ`/`ЛЕВОЕ`/`ПРАВОЕ`/`ПОЛНОЕ СОЕДИНЕНИЕ ... ПО`. Эстимейт: 4–6 дней.

### 1.2. HTTP-клиент в DSL
**Где:** в `internal/dsl/interpreter/builtins.go` нет ни одной функции для HTTP. **Блокер для:** интеграций с внешними API (курсы валют, маркетплейсы, банк-клиент, Telegram-бот).

```
Соединение = Новый HTTPСоединение("api.cbr.ru");
Запрос = Новый HTTPЗапрос("/v1/rates");
Ответ = Соединение.Получить(Запрос);
данные = ПрочитатьJSON(Ответ.ПолучитьТелоКакСтроку());
```

**План:** объект `HTTPСоединение`, `HTTPЗапрос`, `HTTPОтвет` с GET/POST/PUT/DELETE, заголовками, телом, таймаутами. Эстимейт: 3–4 дня.

### 1.3. Хранение файлов / вложения к документам
**Где:** нигде. **Блокер для:** «прикрепить скан договора», «загрузить фото товара», «вложение к заявке» — типичные сценарии любой CRM/учёта.

**План:**
- системный справочник `_attachments` (id, owner_kind, owner_id, filename, mime, size, blob/path);
- файлы в файловой системе (`~/.onebase/files/<base>/`) или в S3-совместимом хранилище;
- виджет загрузки на форме документа (drag-and-drop);
- DSL-API: `Файлы.Добавить(...)`, `Файлы.Получить(...)`.

Эстимейт: 5–7 дней.

### 1.4. Email-уведомления (SMTP)
**Где:** нигде. **Блокер для:** «уведомить клиента о заказе», «прислать отчёт раз в неделю», «забыли пароль».

**План:**
- настройки SMTP в `_onebase_config` или в админке;
- DSL-функция `ОтправитьПисьмо(кому, тема, текст, [вложения])`;
- HTML-шаблоны писем как печатные формы (`mailforms/`).

Эстимейт: 3 дня.

### 1.5. Печать в PDF / экспорт в Excel
**Где:** есть HTML-печатная форма, но нет «скачать PDF» и «выгрузить в xlsx». **Блокер для:** обычной офисной работы — отправить накладную клиенту, выгрузить отчёт в Excel.

**План:**
- PDF: либо через `chromedp`/headless browser, либо через `gofpdf`. Для YAML-печатных форм — генерация в один клик.
- Excel: `xuri/excelize` для отчётов и списков (кнопка «Экспорт в Excel» на любой таблице).

Эстимейт: 4–5 дней.

### 1.6. Поиск и фильтры в списках
**Где:** в журналах есть фильтры (`internal/storage/journal.go`), но в обычных списках справочников/документов — нет универсального поиска. **Блокер для:** баз с >100 записей.

**План:**
- поле «Поиск» с подсветкой по строковым полям;
- быстрые фильтры по ссылкам и перечислениям;
- сохранение фильтров пользователем.

Эстимейт: 4 дня.

## 🟠 Tier 2 — важные расширения (нужны на горизонте 1–3 месяцев)

### 2.1. Расширение языка запросов
- ❌ Подзапросы (`ИЗ (ВЫБРАТЬ ...)`).
- ❌ Временные / пакетные запросы (`ПОМЕСТИТЬ ВТ_X`, `;`).
- ❌ `ИМЕЮЩИЕ` (HAVING) — фильтр после агрегата.
- ❌ `ПЕРВЫЕ N` (LIMIT) и `ОТСТУП M` (OFFSET).
- ❌ `МЕЖДУ` (BETWEEN), `ПОДОБНО` (LIKE), `В` (IN с массивом).
- ❌ `ВЫРАЗИТЬ КАК` (CAST) — особенно для приведения ссылок.
- ❌ Поддержка вложенных полей через точку: `Документ.Контрагент.Менеджер.Наименование`.

Эстимейт суммарно: 7–10 дней.

### 2.2. Конструктор запросов в браузере
Аналог из 1С: визуальный построитель с галочками таблиц, полей, условий. У вас уже есть `ui/query_builder.go` и `ui/tpl_query_builder.go` — значит, заложено. Доделать до состояния, когда отчёт можно собрать без знания текста запроса.

Эстимейт: 5–7 дней.

### 2.3. SCD-подобная компоновка отчётов
- группировки (по нескольким уровням);
- условное оформление (раскраска ячеек по условию);
- подвалы/итоги по группам;
- расшифровка ячейки (drill-down).

Это «лайт-СКД» — не полная компоновка из 1С, но 70% задач закроет.

Эстимейт: 10–14 дней.

### 2.4. Графики и дашборды
Текущие отчёты — таблицы. Современный пользователь ждёт диаграмм:
- линейные/столбчатые/круговые графики (Chart.js / ECharts);
- декларативное описание в YAML рядом с отчётом;
- стартовая страница базы как дашборд (виджеты с ключевыми KPI).

Эстимейт: 6–8 дней.

### 2.5. Регулярные выражения и работа с файлами в DSL
```
// Регексы
рег = Новый Регекс("\\d+");
найдено = рег.НайтиВсе(текст);

// Файлы
ф = Новый ЧтениеТекста("data.csv");
Пока ф.Прочитать(стр) Цикл
    // ...
КонецЦикла;
ф.Закрыть();
```

Эстимейт: 4 дня.

### 2.6. CSV / XML / Архивы (Zip) в DSL
Парсинг и запись CSV (`ПрочитатьCSV` / `ЗаписатьCSV`), XML (`ПрочитатьXML`), работа с zip — для импорта/экспорта.

Эстимейт: 3–4 дня.

### 2.7. Расширение REST API
Сейчас CRUD для catalogs/documents без фильтров и пагинации:
- `?page=&limit=` (или cursor-based);
- `?filter[поле]=значение`, диапазоны, операторы (`gt`, `lt`, `in`);
- `?sort=поле,-другое`;
- `?expand=Контрагент,Менеджер` — резолв ссылок;
- `?fields=Имя,Сумма` — projection;
- API для регистров (включая виртуальные таблицы);
- API для запуска отчётов и обработок;
- автогенерация OpenAPI/Swagger описания;
- API-ключи (помимо session cookies).

Эстимейт: 8–10 дней.

### 2.8. Веб-хуки (исходящие)
Декларативно объявить «при создании РеализацияТоваров → POST на URL» — для интеграций с n8n / Zapier / Make.

```yaml
name: ЗаказыВCRM
trigger:
  document: РеализацияТоваров
  events: [created, posted]
url: https://crm.example.com/webhook
secret: env:CRM_SECRET
```

Эстимейт: 3 дня.

## 🟡 Tier 3 — расширения метаданных (приближение к 1С)

### 3.1. Подписки на события (event subscriptions)
Декларативная привязка процедуры к событию объекта без правки самого объекта — модульность для расширений.

```yaml
name: УведомитьМенеджера
event: Document.РеализацияТоваров.AfterPost
handler: src/notify_manager.proc.os
```

Эстимейт: 3 дня.

### 3.2. Дополнительные реквизиты
Возможность добавить произвольное поле к существующему объекту без правки YAML конфигурации (как «Дополнительные реквизиты и сведения» в 1С). Хранение в EAV-таблице.

Эстимейт: 5–6 дней.

### 3.3. Расширения конфигурации
Аналог расширений 1С: пакет с дополнительными объектами / переопределёнными процедурами, который ставится поверх основной конфигурации. Это уровень maturity, который позволяет иметь маркетплейс.

Эстимейт: 10–14 дней.

### 3.4. Бизнес-процессы и задачи (workflow)
Простая state-machine для документов: «Согласование заявки» — список этапов, исполнители, переходы. Без BPMN, минимально — линейный/ветвящийся workflow.

Эстимейт: 8–10 дней.

### 3.5. Регистры расчёта (для зарплаты)
Если хочется реально заходить в нишу учёта: периодика, графики, сторно. Это значимо больше, чем регистры накопления.

Эстимейт: 14–20 дней.

### 3.6. Полнотекстовый поиск
PostgreSQL `tsvector` + индекс по конфигурируемым строковым полям. Глобальный поиск по базе («найти всё, где упоминается ИНН Х»).

Эстимейт: 5 дней.

## 🟢 Tier 4 — DX и инструменты разработки

### 4.1. VS Code расширение для `.os`-файлов
- подсветка синтаксиса (TextMate grammar);
- сниппеты для типовых конструкций;
- запуск тестов на DSL;
- по возможности — простой LSP с автодополнением встроенных функций.

Эстимейт: 5–7 дней (без LSP — 2 дня).

### 4.2. Линтер для конфигурации
```bash
onebase lint ./my-project
# warning: catalogs/Контрагент.yaml: поле «ИНН» не имеет валидации
# error: src/Документ.os:42: переменная «х» используется до объявления
```

Эстимейт: 4–5 дней.

### 4.3. Юнит-тесты на DSL
```
// src/Утилиты.test.os
Тест("РассчитатьНДС считает 20%")
    результат = РассчитатьНДС(1000);
    УтверждатьРавно(результат, 200);
КонецТеста
```

Запуск: `onebase test ./my-project`.

Эстимейт: 4 дня.

### 4.4. Отладчик DSL
Минимум — `pause` / `step` / просмотр переменных. Через WebSocket к веб-интерфейсу базы (как Chrome DevTools).

Эстимейт: 8–10 дней.

### 4.5. Хранилище конфигурации
Сейчас конфигурация в БД — без истории. Добавить:
- версионирование изменений (кто, когда, что менял);
- diff между версиями;
- возможность отката;
- интеграция с git для коммитов изменений из конфигуратора.

Эстимейт: 7 дней.

### 4.6. Сравнение и объединение конфигураций
«Сравнить, объединить» — ключевая фича 1С при мерже разработческих веток.

Эстимейт: 7–10 дней.

### 4.7. Шаблоны проектов
`onebase init --template crm` / `--template warehouse` / `--template tasks`. Готовые мини-конфигурации, которые ставятся одной командой и сразу работают. Это резко снижает порог входа.

Эстимейт: 3 дня (на 3-4 шаблона).

## 🔵 Tier 5 — эксплуатация и продакшн

### 5.1. Бэкап и восстановление в один клик
Кнопка «Бэкап» в лаунчере → `pg_dump` → файл. Кнопка «Восстановить» → выбор файла → `pg_restore`. Ежедневные авто-бэкапы по расписанию.

Эстимейт: 3 дня.

### 5.2. Журнал регистрации
То, что в 1С называется «Журнал регистрации»: единая лента всех действий пользователей (вход, изменения, проведение, удаление, ошибки). Уже есть `audit_log` — расширить до полноценного журнала с фильтрами.

Эстимейт: 4 дня.

### 5.3. Метрики и мониторинг
- `/metrics` в формате Prometheus (запросы, длительности, активные сессии, длина очереди заданий);
- структурированные логи (JSON);
- готовый Grafana dashboard в `examples/ops/`.

Эстимейт: 3–4 дня.

### 5.4. Многобазовая инсталляция и удалённые базы
Сейчас лаунчер — локальный. Добавить:
- удалённые базы (URL вместо локального процесса);
- единый SSO между базами;
- управление пользователями централизованно.

Эстимейт: 7–10 дней.

### 5.5. Поддержка SQLite для маленьких сценариев
PostgreSQL — overkill для домашнего таск-трекера. SQLite-режим (один файл, без сервера) сделал бы установку буквально one-click для пет-проектов.

Эстимейт: 10–14 дней (адаптация SQL-генерации, миграций, drivers).

### 5.6. Кластеризация и HA
Несколько `onebase.exe` за load balancer + общий PostgreSQL + распределённый scheduler (только один экземпляр выполняет регламент). Для серьёзных нагрузок.

Эстимейт: 10–14 дней.

## 🟣 Tier 6 — экосистема и сообщество

### 6.1. Готовые типовые конфигурации
Стартовый список:
- **Tasks** — мини-Jira (проекты, задачи, статусы, исполнители);
- **CRM-lite** — клиенты, сделки, активности, воронка;
- **Warehouse-lite** — склад с поступлением, перемещением, остатками;
- **Personal Finance** — домашние финансы (счета, переводы, бюджет);
- **Freelance** — учёт фриланс-проектов, актов, оплат.

Каждая конфигурация: YAML + примеры данных + видео-разбор. Это — главный driver роста.

Эстимейт: 3–5 дней на каждую.

### 6.2. Marketplace конфигураций
Реестр (даже простой github-репозиторий с YAML-каталогом) + кнопка «Установить из marketplace» в лаунчере.

Эстимейт: 7 дней (минимальная версия).

### 6.3. Демо-сервер
`demo.onebase.dev` — развёрнутая база с тестовыми данными, песочница «потыкать без установки». Запись действий не сохраняется дольше суток. Отдельный лендинг.

Эстимейт: 3 дня (один раз поднять).

### 6.4. Документация-сайт
mkdocs / docusaurus вместо MD-файлов в репо: поиск, навигация, тёмная тема, версионирование.

Эстимейт: 3–5 дней.

### 6.5. Сообщество
- Telegram-канал для анонсов;
- Telegram-чат для вопросов;
- GitHub Discussions для архитектурных обсуждений;
- регулярный changelog;
- issue/PR templates.

Эстимейт: 1–2 дня.

### 6.6. Видео-туториалы
План в `youtube-video-plan.md` — серия из 6 роликов. Это **главный канал привлечения** разработчиков 1С.

## 🛡 Tier 7 — безопасность и качество

### 7.1. Безопасные модули (sandbox)
Сейчас DSL имеет доступ ко всему. После добавления HTTP/файлов нужен режим «безопасного исполнения» для конфигураций из marketplace: запрет файловой системы, ограничение исходящего трафика, лимиты CPU/памяти.

Эстимейт: 5–7 дней.

### 7.2. Rate limiting и защита от DDoS
Базовый rate limit на login и API.

Эстимейт: 2 дня.

### 7.3. Audit security
- хеширование паролей (проверить — `bcrypt` / `argon2`?);
- защита от CSRF;
- защита от SQL-инъекций (по идее — параметры везде, проверить query builder);
- журналирование подозрительной активности.

Эстимейт: 3 дня (проверка) + правки.

### 7.4. Подписи / шифрование
Базовые криптографические builtins в DSL: `Хэш(текст, "SHA256")`, `Подписать(...)`, `Зашифровать(...)`. Для интеграций с госорганами и банками.

Эстимейт: 3–4 дня.

## 🌍 Tier 8 — интернационализация

Сейчас всё на русском (билингвальность только в синтаксисе). Чтобы выйти за пределы ru-аудитории:
- i18n веб-интерфейса (en, как минимум);
- локализованные форматы дат и чисел;
- английская версия примеров и документации;
- международный лендинг (`onebase.dev`) с альтернативным позиционированием — «1C-style platform for small business».

Это меняет масштаб проекта, но открывает мировой open-source рынок.

Эстимейт: 7–10 дней (без переписывания примеров).

- **Неделя 1:** JOIN в запросах + HTTP-клиент.

## 🧭 Стратегическая развилка

В долгую у проекта три возможных вектора развития. Они не взаимоисключающие, но требуют расстановки приоритетов:

**A. Платформа для пет-проектов и малого бизнеса** (текущая ниша).
Фокус: Tier 1, 4, 6. SQLite-режим, шаблоны проектов, готовые конфигурации, marketplace.

**B. «Открытая 1С» для среднего бизнеса** (амбициозно).
Фокус: Tier 2, 3, 5. Регистры расчёта, бизнес-процессы, кластеризация, расширения конфигурации, СКД.

**C. Headless-платформа для разработчиков** (API-first).
Фокус: Tier 2 (REST API, OpenAPI), Tier 4 (LSP, тесты), Tier 6 (документация). UI вторичен — вместо него фронтенд клиента на React/Vue.

**Рекомендация:** ориентируйтесь на **A** на горизонте 6 месяцев. Это совпадает с позиционированием статьи и видео, не требует огромных вложений в архитектуру и даёт измеримый рост через готовые конфигурации и marketplace. Векторы B и C — после того, как наберётся комьюнити и будут понятны реальные запросы.
