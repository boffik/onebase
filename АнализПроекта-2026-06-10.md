## 0. TL;DR

Проект здоров: **393 Go-файла / 93 213 строк** (код 71 242 / тесты ~22 000, ≈24 %),
`go vet` чист, `go test ./...` зелёный, архитектура слоистая, круговых импортов нет.
За прошедшие дни добавлен полноценный **ИИ-помощник** (провайдеры/модели/профили,

Болевые точки остались **качественные, а не функциональные**. По приоритету:

1. 🔴 **Гонка данных в разделяемом интерпретаторе** — *подтверждено, не исправлено*.
   Теперь видно, что бьёт по **горячему пути проведения документов**, не только по ИИ.
2. 🟠 **Сессионный токен утекает в логи** (`?_tk=` + `middleware.Logger`) — новое.
3. 🟠 **ИИ-доступ к данным = чтение всего в обход RBAC + выгрузка во внешний LLM** —
   новый класс рисков, требует осознанной выдачи флага и оговорки для оператора.
4. 🟠 **Нет CSRF-токенов, rate-limiting (логин и ИИ-чат), security-заголовков** — частично из прошлого.
5. 🟡 **God-файлы** (`handlers.go` 3.9 K строк/119 функций, `configurator_tmpl.go` 4.9 K) — *подтверждено*.

Снято по сравнению с 09.06: пункт «нет `middleware.Recoverer`» — **неверен**, Recoverer
стоит на родительском роутере (`api/server.go:35`) и покрывает все UI-маршруты.

## 1. Что хорошо (не сломать при доработках)

- ✅ **Слоистая архитектура** `metadata → runtime → storage → dsl → query → ui/api`.
  Новый ИИ-слой добавлен правильно: `internal/llm` намеренно без зависимостей от
  storage/dsl (комментарий в `config.go:1`), связка с базой — через узкий интерфейс
  `aiassist.ConfigSource` (один метод `GetLLMConfig`).
- ✅ **Транзакции безопасны к панике** (`storage/tx.go:22,39`): `recover → Rollback →
  re-panic` — критично для единственного SQLite-коннекта.
- ✅ **`middleware.Recoverer` активен** на корневом роутере (`api/server.go:35`),
  применяется и к группе с UI — паника в любом слое не роняет процесс. *(Снимает
  вчерашнюю 2.4.)*
- ✅ **ИИ-чат не уязвим к XSS**: ответ модели вставляется через `textContent`
  (`templates.go:538,547`), панель сообщений — через `escapeHtml`.
- ✅ **ИИ-запросы реально read-only by construction**: язык запросов (`query.go:299`
  `kwMap`) **не содержит** `INSERT/UPDATE/DELETE/DROP` — любая попытка DML превратится
  в идентификатор и даст ошибку компиляции, а не запись. Значения параметризуются,
  строковые литералы экранируются (`query.go:1782`).
- ✅ **Секреты не утекают через ИИ**: API-ключи маскируются в админ-форме
  (`llm.Redacted`), мерджатся обратно по маске на сохранении (`UnmaskKeys`), не
  попадают в `describe`/экспорт; `llm.SafeErr` вырезает `key=`/`token=` из текста
  ошибок (`types.go:99-109`); тело ответа провайдера ограничено 8 МиБ
  (`httpjson.go:12`) — защита от OOM.
- ✅ **Фолбэк-движок ИИ корректен**: ретрай только на 429/5xx/сетевых
  (`types.go:88`), уважает отмену контекста (`run.go:55,102`), лимит раундов
  tool-use = 12 (`tools.go:35`).
- ✅ **Реестр метаданных потокобезопасен**: `Load*` собирают новые map и атомарно
  меняют под `mu.Lock()` (`registry.go:191`), читатели берут `RLock` — hot-reload
  `--watch` не гонщит с HTTP-обработчиками.
- ✅ **Демо-режим закаляется**: `SetFileSandbox(proj.Dir)` ограничивает файловые
  builtins, ИИ-ключи не входят в `.obz` (деплоятся через `app.yaml`/env при старте).
- ✅ **Все тесты зелёные**, `go vet` молчит, TODO/FIXME всего 8.

## 2. Баги и проблемы (по приоритету)

### 🔴 2.1. Гонка данных в разделяемом `*Interpreter` — ПОДТВЕРЖДЕНО, НЕ ИСПРАВЛЕНО

**Где:** `internal/dsl/interpreter/interpreter.go:72-74` (поля) + `:170-171` (запись) +
точки использования общего `interp` во всём `ui`/`entityservice`.

Сервер создаёт **один** интерпретатор (`cli/run.go:214 interp := interpreter.New()`),
кладёт в `Server.interp` и в `entityservice.Service.Interp`. При этом интерпретатор
хранит изменяемое состояние, которое пишется на **каждом операторе**:

```go
type Interpreter struct {
    ...
    DebugHook DebugHook  // одно поле на все сессии
    curFile   string     // пишется в execBlock на каждом стейтменте
    curLine   int
}

func (i *Interpreter) execBlock(stmts []ast.Stmt, e *env) {
    for _, s := range stmts {
        if loc := getLocation(s); loc != nil {
            i.curFile = loc.File   // ← гонка при конкурентных запросах
            i.curLine = loc.Line
        }
        ...
```

**Новое за сегодня — путь срабатывания шире, чем казалось.** Гонит не только ИИ, а
**горячий путь проведения**: `entityservice.Service.Save` запускает хук документа на
общем интерпретаторе (`service.go:175 s.Interp.Run(proc, obj, vars)`), и то же —
`Service.Fill` (`:390`), события управляемых форм (`handlers_managed_events.go:194,
619,674`), отчёты/обработки/консоль кода (`handlers.go:1472,1640,2147,2200,3887`).
Любые два одновременных «Провести» гонят по `curFile/curLine`.

**Последствия:**
1. Запись `string` (ptr+len) не атомарна → конкурентная запись может склеить `ptr`
   одной и `len` другой → чтение за границей памяти → **краш рантайма**, а не просто
   неверный текст в `DSLError`.
2. `DebugHook` глобален (`debug_handlers.go:55 s.interp.DebugHook = sess`): один
   пользователь включил отладку — брейкпоинты и пошаговое исполнение срабатывают во
   **всех** чужих сессиях.

**Почему не ловится:** desktop single-user (основной сегодня) исполняет запросы
последовательно; тесты тоже последовательны; на Windows `-race` требует CGo (которого
в сборке нет) — race-детектор **не запускался ни разу**. Выстрелит ровно при включении
заявленного многопользовательского режима.

**Как чинить (по возрастанию усилий):**
- Вынести `curFile/curLine` из `Interpreter` в `*env` (или в per-call execution
  context). `newEnv(this)` уже создаётся на каждый `Run` → состояние станет
  естественно изолированным. Это дешёвая и правильная починка.
- `DebugHook` сделать per-session (передавать в `Run`/в env), а не полем общего
  интерпретатора.
- Добавить **конкурентный race-тест**: N горутин гоняют `Run` по одному интерпретатору
  под `-race`; гонять в **CI на Linux** (там `-race` работает без CGo-проблем).

### 🟠 2.2. Сессионный токен утекает в логи через `?_tk=` — НОВОЕ

**Где:** `auth/middleware.go:33` (чтение `_tk` из query) + `api/server.go:34`
(`r.Use(middleware.Logger)`) + `launcher/configurator_tmpl.go:2690` (формирование URL).

Конфигуратор открывает пользовательский режим в iframe, передавая сессионный токен
**в query-параметре URL**: `/ui?_tk=<token>`. Middleware затем ставит его cookie. Но
`middleware.Logger` стоит на корневом роутере **выше** auth-группы и логирует полный
RequestURI — то есть **токен пишется в stdout-лог** на каждом таком заходе.

Токены в URL — известный анти-паттерн: утекают также через `Referer` на внешние
ресурсы и через историю браузера. Токен живёт 24 часа (`users.go:245`) и даёт полный
доступ к сессии.

**Как чинить:**
- Передавать токен не в URL, а через `POST`-bootstrap (уже есть `/auth/bootstrap` —
  но он тоже берёт `token` из query; перевести на короткоживущий one-time код).
- Минимум — редактировать `_tk`/`token` в формате логгера (кастомный `LogFormatter`,
  вырезающий чувствительные query-параметры), и сократить TTL такого токена.

### 🟠 2.3. ИИ-доступ к данным = чтение всего в обход RBAC + выгрузка во внешний LLM — НОВОЕ

**Где:** `ui/ai_tools.go:25-31` (`aiDataAllowed`) + `:188-236` (`aiRunQuery`).

Флаг пользователя `AIDataAccess` (и роль админа) открывает инструмент
`выполнить_запрос`, который компилирует и исполняет **произвольный** read-запрос
**без проверки объектного RBAC** (`s.can(...)` не вызывается — в отличие от обычных
list/form-обработчиков). Это честно задокументировано в комментарии
(`ai_tools.go:22-24`: «в обход объектного RBAC»), но по факту флаг означает
**«читать всё в базе»**, эквивалент админ-чтения.

Вторая грань: результаты запроса уходят во **внешний LLM-провайдер** (Anthropic/z.ai/
Gemini). Выдача `AIDataAccess` пользователю = разрешение выгружать любые данные базы
третьей стороне через диалог. Для публичной демо/мультибазы это особенно чувствительно.

**Рекомендации (не баг, а явная модель угроз — задокументировать и подстраховать):**
- В карточке пользователя рядом с флагом показать предупреждение «доступ к чтению всех
  данных + передача во внешний ИИ».
- Рассмотреть опциональное уважение RBAC в `aiRunQuery` (фильтровать источники по
  `s.can(read)`), либо хотя бы белый список объектов, доступных ИИ.
- Логировать ИИ-запросы в журнал регистрации (аудит): какой пользователь что
  запросил у ИИ.

### 🟠 2.4. Нет rate-limiting: логин и ИИ-чат — ПОДТВЕРЖДЕНО (логин) + НОВОЕ (ИИ)

- **Логин** (`auth/handlers.go:67 LoginSubmit`, `:114 LoginJSON`): брутфорс пароля
  ничем не ограничен — нет счётчика попыток/задержки на IP/логин.
- **ИИ-чат** (`ui/ai_chat.go:30`): нет лимита частоты. Каждый авторизованный
  пользователь может слать запросы без ограничения, а один запрос — это до 12 раундов
  tool-use × вызовы LLM. Это **вектор cost-DoS**: неконтролируемый расход денег на
  API провайдера (особенно болезненно для публичной демо). Минимум — лимит запросов
  ИИ на пользователя/минуту и суточный потолок токенов.

---

### 🟠 2.5. Нет CSRF-токенов и security-заголовков — ПОДТВЕРЖДЕНО

- **CSRF:** защита держится только на `SameSite=Lax` (cookie в `auth/handlers.go:104`,
  `middleware.go:56`). Lax прикрывает большинство cross-site POST, но для платформы,
  где POST = провести/удалить/изменить документ, стоит добавить полноценный CSRF-токен
  (double-submit cookie или синхронизатор формы). Поиск по коду: `csrf` — **0 совпадений**.
- **Security-заголовки:** нет `X-Frame-Options`/`Content-Security-Policy`. База
  встраивается в iframe конфигуратора — clickjacking на пользовательский режим
  потенциально открыт. Поиск: `x-frame-options|content-security-policy` — 0 совпадений.

---

### 🟡 2.6. Нет объектного RBAC на вложениях (IDOR, смягчён UUID) — НОВОЕ (низкий)

`attachmentDownload`/`attachmentDelete` (`handlers.go:4192,4212`) ищут вложение по
UUID и **не проверяют** право чтения/изменения родительской сущности (`s.can`). Любой
авторизованный пользователь, зная `aid`, скачает/удалит вложение объекта, к которому у
него нет доступа. Severity низкий: UUID неперечислим. Стоит добавить проверку прав на
родительскую запись.

---

### 🟡 2.7. Footgun: «нет пользователей ⇒ нет авторизации» + bind на все интерфейсы — НОВОЕ

`isAdmin`/middleware трактуют отсутствие пользователей как «auth не настроен» и
пропускают всё (`admin.go:668-670`, `middleware.go:20`) — включая **консоль кода**
(произвольный DSL → файловые builtins). Сервер слушает `:port` (`api/server.go:86`),
то есть на всех интерфейсах. Для desktop-localhost это норма, но если кто-то поднимет
`onebase run` без создания пользователей и пробросит порт наружу — вся база и консоль
кода открыты без аутентификации. Стоит: по умолчанию bind на `127.0.0.1` (флаг для
`0.0.0.0`), и/или предупреждение при старте без пользователей на не-loopback адресе.

## 3. Технический долг и DX

1. **CI с `-race` и покрытием.** Прогон `go test -race ./...` на Linux ловит 2.1;
   публиковать coverage по пакетам (особо для больших `ui`/`launcher`).
2. **`golangci-lint`** (errcheck, ineffassign, gosec) — машинно поймает проглоченные
   ошибки и часть security-замечаний.
3. **Фронт из строк → в файлы** (`go:embed`) — см. 2.8: подсветка, prettier/eslint,
   диффабельность.
4. **Структурные логи + `/metrics`.** Для «продакшна малого бизнеса» — JSON-логи
   (с редактированием секретов, см. 2.2) и Prometheus-эндпоинт (активные сессии,
   длительности, очередь заданий, **токены/стоимость ИИ**).
5. **Линтер конфигураций** (`onebase lint` поверх `check`) — предупреждения о
   неиспользуемых переменных DSL, невалидируемых полях и т.п.
6. **Аудит ИИ-вызовов** — отдельная таблица: кто/когда/какой запрос/какая модель/
   сколько токенов (нужно и для безопасности 2.3, и для контроля стоимости 2.4).

## 4. Идеи развития и киллер-фичи

База закрыта — дифференциация не «догнать 1С по галочкам», а сделать то, чего у 1С
нет или что у неё мучительно.

### 4.1. 🌟 Достроить ИИ-помощника до «ИИ-конфигуратора» (killer-feature №1)
Инфраструктура уже есть (`llm` + tool-use + `ai-guide`/`AGENTS.md`). Следующий шаг —
**инструменты записи под подтверждение пользователя**: «опиши задачу на русском →
получи каркас справочника/документа/проводки», «объясни ошибку `onebase check`
человеческим языком», «построй запрос по описанию отчёта». Для аудитории «бывшие
1С-ники + малый бизнес» это снимает главный барьер входа. Предпосылка — **песочница
для сгенерированного DSL** и явное подтверждение перед записью.

### 4.2. 🌟 «Живая» интеграционная платформа (killer-feature №2)
HTTP-клиент в DSL уже есть → достроить **исходящие веб-хуки** (декларативно «при
проведении X → POST на URL») и **входящие** (REST-эндпоинт, вызывающий DSL-обработчик).
Превращает OneBase в хаб для n8n/Make/Telegram-ботов — ниша, где 1С слаба, а малому
бизнесу нужно ежедневно.

### 4.3. Шаблоны проектов одной командой
`onebase init --template crm|tasks|warehouse`. Примеры уже в `examples/` — обернуть в
установщик. Дёшево (~3 дня), резко снижает порог входа.

### 4.4. Лёгкая компоновка отчётов (drill-down, группировки, графики)
Самый частый запрос после базового учёта. Не полная СКД, но группировки + условное
оформление + расшифровка ячейки + графики (Chart.js) закроют ~70 % задач. Дашборды/
виджеты уже есть — естественное продолжение.

### 4.5. Marketplace конфигураций
Когда есть шаблоны и `onebase check` — github-реестр YAML-конфигураций + кнопка
«Установить» в лаунчере. Предпосылка — **sandbox для DSL** (ограничение FS/сети/CPU),
иначе ставить чужое из интернета опасно (та же песочница нужна и для 4.1).

### 4.6. Версионирование конфигурации в БД (git-подобное)
